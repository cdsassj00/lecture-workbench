API는 '타인이 만든 기능을 호출하는 창구'
API(Application Programming Interface, 타인이 만든 기능을 내 프로그램에서 호출해 쓰는 창구)는 작동 원리가 한 줄로 요약됩니다 — 창구에 질의를 송신하면 응답이 돌아옵니다. 거점리더가 알아야 할 상식은 셋뿐입니다. 첫째, 키는 비밀번호와 같은 민감도이므로 노출·공유·캡처 금지입니다. 둘째, 유료인 경우가 많아 쿼터(일정 시간당 호출 횟수 한도)·요금을 먼저 확인해야 합니다. 셋째, 외부 창구는 간헐적으로 실패하므로 재시도 정책이 필요합니다. REST(API를 HTTP 주소와 메서드로 일관되게 구성하는 대표 방식)·헤더·JSON(구조화된 데이터를 텍스트로 주고받는 표준 포맷) 같은 기술 세부의 코드 작성은 에이전트가 처리함. 거점리더는 어떤 호출이 일어나고 어떤 응답을 받는지 한 줄로 설명할 수 있어야 함.
창구 비유커리큘럼 공식 용어
API = 타인이 구축한 기능을 호출해 사용하는 창구

간헐적 실패 — 재시도 지시는 말로 끝낸다
외부 창구 호출은 서버 부하·네트워크 지연·일시 점검 등으로 항상 실패 가능성을 안고 있으므로, "한 번에 성공"이라는 가정을 버리고 재시도를 기본값으로 설계합니다. 거점리더가 결정하는 것은 정책뿐입니다. 첫째, 어떤 실패를 재시도할지 — 일시 지연은 재시도, 쿼터 초과는 대기 후 재시도입니다. 둘째, 어떤 실패를 즉시 중단할지 — 인증 오류는 중단·점검, 창구 점검은 일정 시간 차단입니다. 셋째, 로그·알림을 어디에 남길지 — 모든 실패와 재시도 기록을 logs/에 보관하고 일 1회 요약을 받습니다. 지수 백오프·서킷브레이커 같은 용어는 몰라도 되며, 에이전트가 표준 패턴을 씁니다.
왜 재시도실패는 일상
창구 너머의 상황은 우리가 통제하지 못한다

서버 부하·네트워크 지연·일시 점검 등으로 호출이 실패하는 일은 항상 있다. 한 번에 성공이라는 가정을 버리고, 재시도를 기본값으로 설계한다.

MCP = 에이전트가 외부 도구를 연결하는 표준 콘센트
MCP(Model Context Protocol, Anthropic이 2024년 11월 공개한 개방 표준 — 에이전트와 외부 도구·데이터 간 연결 규격을 통일한 프로토콜)는 커리큘럼 공식 비유로 콘센트에 해당합니다. 규격만 맞으면 어떤 도구든 꽂혀 동일한 방식으로 호출되기 때문입니다. 창구(API)와는 층위가 다릅니다. API는 개별 건물의 출입구 하나이고, MCP는 그 건물들을 꽂아 쓰는 공용 콘센트 규격입니다. 거점리더가 해야 할 동작도 단순합니다. 첫째, 이미 공개된 서버 카탈로그에서 재료 후보를 발굴합니다. 둘째, 부서 업무에 닿는 서버 후보를 메모합니다. 셋째, 설정 파일 한 번 수정으로 에이전트에 꽂습니다. 설치·코드·SDK 세부는 모두 에이전트 몫입니다.
콘센트 비유커리큘럼 공식 용어
MCP는 '도구 연결 표준 콘센트'

우리 일상의 멀티탭처럼 규격이 맞으면 어떤 도구든 꽂힌다. MCP가 깔리면 에이전트는 파일·메일·DB·외부 서비스를 같은 방식으로 호출한다. 창구(API)는 개별 건물의 출입구, MCP는 그 건물들을 꽂아 쓰는 공용 콘센트 규격이다.

API와의 차이
창구 ≠ 콘센트

API는 외부 기능을 부르는 창구. MCP는 에이전트가 여러 도구를 꽂는 콘센트. 층위가 다르다.

탐색 원칙이미 있는 콘센트부터
기존 MCP 서버 카탈로그에서 재료 후보를 발굴

이미 수많은 MCP 서버가 공개돼 있다. 마트 재료 진열대를 훑듯 카탈로그를 순회해 우리 부서에 쓸 만한 서버를 찾는 것이 첫 단계. 없는 것만 직접 제작.

① 에이전트에 MCP 연결 = 설정 1회 수준

카탈로그에서 고른 서버를 에이전트에 연결하는 작업은 설정 파일 한 번 수정 정도. 클릭·복사·붙여넣기로 끝난다.

② MCP 서버 직접 제작도 에이전트에 위임

MCP 연결은 설정 1회 수준 — 강사 시연
카탈로그에서 고른 MCP 서버를 에이전트에 꽂는 작업은 몇 분 안에 끝나는 3단계 시연입니다. 첫째, 서버 고르기 — 카탈로그에서 이름·설명·쿼터를 확인하고 설치·연결 안내를 복사합니다. 둘째, 설정 파일 수정 — Claude Desktop이나 Cursor의 "mcpServers" 블록에 카탈로그가 준 설정을 그대로 붙여넣습니다. 셋째, 에이전트 재시작과 첫 호출 — "방금 연결한 도구로 X 좀 해줘"로 동작을 확인합니다. 설정 파일 문법·디렉터리 경로·SDK 호환성은 에이전트에게 위임하며, 실패 시 "로그 보고 고쳐줘" 한 문장이면 됩니다. 한 번 꽂은 콘센트는 계속 쓰입니다.

# 강사 시연 흐름 · 비개발자 친화 설명 1) 카탈로그에서 서버 고르기 - 이름·설명·쿼터 확인 - 설치·연결 안내 복사 2) 에이전트 설정 파일 열기 - Claude Desktop·Cursor 등 - "mcpServers" 블록에 카탈로그가 준 설정 그대로 붙여넣기 3) 에이전트 재시작 + 첫 호출 - "방금 연결한 도구로 X 좀 해줘" - 응답이 오면 연결 성공 - 응답이 없으면 설정·로그를 에이전트에게 "고쳐줘" 라고 지시

칸 01 · 트리거 / 칸 02 · 재료
캔버스의 출발점 두 칸입니다. 첫째, 트리거는 "이 업무는 어떤 사건이 불러오는가"에 답하는 칸으로, 사람이 매번 누르는 버튼이 아니라 자동으로 시작되는 순간을 적습니다. 유형은 다섯입니다 — 시간 스케줄(매일 09시)·파일 도착(지정 폴더 변경)·메일·민원 수신·결재 승인·사람 요청. 둘째, 재료는 에이전트가 판단에 쓰는 모든 입력을 모으는 칸이며 네 층으로 구성됩니다. 공공 API 창구(Part B 재료 카드), MCP 콘센트(Part C 카탈로그 또는 내부 제작 서버), 부서 문서·과거 사례(매뉴얼·판례·응대 이력), 사람 입력(담당자 요청·폼·선택지)입니다. 재료의 정확도가 산출물의 상한선이며, "재료 1종이 빠지면 어떻게 되나"를 자문해 대체재를 확보합니다.
칸 01 · 트리거언제 작동하는가
"이 업무는 어떤 사건이 불러오는가"

사람이 매번 누르는 버튼이 아니라, 자동으로 시작되는 순간. 업무 성격에 맞는 한 가지를 고른다.

유형예시부서 업무
시간 스케줄매일 09시일일 브리핑
파일 도착지정 폴더 변경제출 문서 접수
메일·민원 수신메일함·민원창구민원 응대 착수
결재 승인전자결재 알림후속 보고 생성
사람 요청메신저 멘션즉석 질의 처리
HINT
"이 일, 언제 시작하세요?"를 부서원에게 묻는다

답이 트리거 칸의 정답. 추상어 대신 실제 사건으로 기록.

PITFALL
▸ 너무 촘촘한 스케줄
5분 간격으로 돌면 쿼터·요금·로그 폭증. 업무 주기에 여유 포함.
칸 02 · 재료무엇을 받는가
재료 = 에이전트가 판단에 쓰는 모든 입력

공공 API 6종 재료 카드 + 부서 DB·문서 + MCP 콘센트 + 과거 사례. 재료의 정확도가 산출물의 상한선.

재료 1 · 공공 API 창구
Part B 재료 카드 중 선택

기상·법령·통계·행정 표준 등 부서 업무에 닿는 창구.

재료 2 · MCP 콘센트
Part C 카탈로그 또는 내부 제작 서버

메일·파일·DB·노션 등 업무 도구를 콘센트로.

재료 3 · 부서 문서·과거 사례
매뉴얼·판례·응대 이력

검색용 인덱스로 등록. 에이전트가 근거로 인용.

재료 4 · 사람 입력
담당자 요청·폼·선택지

사람이 직접 줄 것 vs 시스템이 가져올 것을 구분.

설계 질문
"재료 1종이 빠지면 어떻게 되나"

답이 나오면 의존도·대체재가 명확한 설계. 답이 없으면 재료 계획 보강.

칸 03 · 처리 / 칸 04 · 시스템 프롬프트
재료가 어떻게 흘러가고 어떤 인격으로 처리되는지 결정하는 두 칸입니다. 첫째, 처리는 "재료를 어떻게 엮어 산출물로 보내는가"에 답합니다. 멀티 에이전트 조합 패턴 카드 7종(단일·순차·병렬·오케스트레이터·검증 쌍·라우팅·보고 분리) 중 업무 분업 구조와 닮은 1종을 골라 적습니다. 기술 판단이 아니라 "지금 부서원이 어떻게 나눠 하는가"가 정답의 근거입니다. 둘째, 시스템 프롬프트는 에이전트에게 주는 상근 계약서로 매 요청마다 반복하지 않도록 한 번 고정합니다. 다섯 항목을 한 단락에 담습니다 — 역할, 재료 사용 규칙, 톤, 금칙·마스킹, 응답 형식. 신입 주무관에게 주는 업무 가이드 수준으로 쓰면 에이전트에도 충분합니다.
칸 03 · 처리조합 패턴 카드 1종
처리 칸 = "재료를 어떻게 엮어 산출물로 보내는가"

Part F-1의 조합 패턴 카드 7종 중 하나를 골라 기입. 단일 에이전트가 될 수도 있고, 순차·병렬·검증 쌍·라우팅 등일 수도 있다. 첫 캔버스는 카드 1개로만.

카드언제 고르나
단일 에이전트단순 1단계 업무
순차형단계가 분명
병렬형재료 여러 종 동시 수집
오케스트레이터접수·분배·통합 통합
검증 쌍민감 산출물
라우팅형유형별 담당이 다름
보고 분리결과 전달 톤 별도
HINT
"지금 부서원이 어떻게 나눠 하는가"

실제 업무 분업 구조와 카드가 닮은 쪽이 정답. 기술 판단이 아님.

칸 04 · 시스템 프롬프트역할·톤·제약
시스템 프롬프트 = 에이전트에게 주는 상근 계약서

매 요청마다 반복하지 않도록 한 번 고정. 역할·톤·금칙·인용 규칙·응답 형식을 문단으로.

# 캔버스 칸 04 · 시스템 프롬프트 양식 [역할] 당신은 _____ 부서 _____ 담당 주무관이다. [재료 사용 규칙] - 재료 칸의 창구·콘센트·문서만 사용 - 추측 답변 금지 - 근거 출처를 항상 제시 [톤] - 공문서 어투 · 존중 · 간결 [금칙·마스킹] - 개인정보 8필드 마스킹 후 처리 - 법령·조례 없는 약속 금지 [응답 형식] - 요약 / 근거 / 다음 행동 3단 [검증·보고] - 검증자·보고자 역할은 별도 에이전트
KEY
"신입 주무관에게 주는 업무 가이드"

사람에게 설명할 수 있는 수준이면 에이전트에도 충분하다.

PITFALL
▸ 예시 없이 규칙만
좋은 응답 1 + 나쁜 응답 1을 함께 제시.

칸 05 · 산출물 / 칸 06 · 배포
"일회성 데모"와 "부서 자산"을 가르는 마지막 두 칸입니다. 첫째, 산출물은 다음 단계의 재료가 되도록 남기는 칸으로 네 층을 기록합니다 — 문서(HWP·PDF·XLSX 보고서·초안), 메시지·알림(메일·메신저 담당자 통보), 로그·기록(폴더·DB 감사 근거), 부서 자산(공용 폴더·위키 재사용 템플릿). 규칙은 "산출물 1개 + 로그 1개 = 최소 세트"입니다. 둘째, 배포는 어디서 돌고 누가 운영하는가를 적는 칸으로 네 단계로 올라갑니다 — Tier 1 개인 PC(첫 시연·파일럿), Tier 2 부서 공용 PC(동료 1~2명 공동 사용), Tier 3 기관 내부망(보안·감사 절차 병행), Tier 4 클라우드(상급 기관 승인 필요). 첫 캔버스는 반드시 Tier 1에서 시작합니다.
칸 05 · 산출물무엇이 누구에게
산출물 = 다음 단계의 재료가 되도록 남긴다

대화창에서만 보이면 자동화가 아니다. 문서·메시지·로그·부서 자산의 4층으로 남긴다.

형태매체부서 예시
문서HWP·PDF·XLSX보고서·초안
메시지·알림메일·메신저담당자 통보
로그·기록폴더·DB감사 근거
부서 자산공용 폴더·위키재사용 템플릿
RULE
"산출물 1개 + 로그 1개 = 최소 세트"

사람이 보는 결과 + 감사 로그는 항상 짝. 칸 05에 두 줄로 적는다.

설계 질문
"이 산출물이 사라지면 누가 곤란한가"

답이 분명해야 산출물의 가치와 수취인이 확인된다.

칸 06 · 배포어디서 돌아가는가
배포 = 어디서 돌고 누가 운영하는가

개인 PC → 부서 공용 → 기관 내부망 순서로 단계를 올린다. 최종 발표에서는 첫 배포 위치와 운영 담당까지 명시.

TIER 1 · 개인 PC
거점리더 본인 노트북

첫 시연·파일럿. 가치 검증에 가장 빠른 경로.

TIER 2 · 부서 공용 PC
동료가 함께 쓰는 수준

파트너 1~2명과 공동 사용. 공용 폴더·공용 계정.

TIER 3 · 기관 내부망
기관 서버·VM

보안·감사 절차 병행. 내부망 LLM 결합 시 완전 폐쇄형 가능.

TIER 4 · 클라우드
상급 기관 승인 필요

공공 데이터는 비식별·암호화가 기본 전제. 일반 업무는 Tier 1~3 먼저.

PITFALL
▸ 첫 캔버스가 Tier 4
첫 배포는 Tier 1. 가치 증명 이후 승격.

8주의 마지막 산출물 —
캔버스 1장과 코칭 계획 4항.

8회차 산출물은 두 가지입니다. 첫째, AI 아키텍처 캔버스 6칸(트리거·재료·처리·시스템 프롬프트·산출물·배포)을 한 장에 작성한다. 둘째, 코칭·전파 계획 4항(대상 동료 2~3명·일정·자료·후속 점검 시점)을 함께 제출한다. 두 가지가 모두 충족되면 과정을 수료합니다.

"1회차 거점 미션은 8회차 캔버스 6칸으로 완성되었다. 복귀 후 30일, 동료 2~3명에게 시연하고 공용 자산으로 올리는 것까지가 거점리더의 책임이다."
— 8회차 공식 종결 메시지

복귀 30·60·90 · 동료 2~3명 코칭·전파 계획 · 10MIN
복귀 후 30·60·90일 플랜
과정의 공식 과제는 90일 안에 캔버스를 부서 공용 폴더에 등록하고 동료 2~3명이 재사용하게 만드는 일입니다. 코칭·전파 가이드는 넷입니다. 첫째, 대상은 코칭 동료 2~3명(직무·관심 유형 명시)으로 정합니다. 둘째, 일정은 복귀 후 4주 안 시범 코칭·8주 안 부서 단위 시연으로 잡습니다. 셋째, 자료는 공용 저장소 산출물과 빠른 파악 루틴 체크리스트로 묶습니다. 넷째, 후속 점검은 3개월 시점 회고 1장으로 마감합니다. 이를 30·60·90 단계에 풀어 쓰면 다음과 같습니다. Day 1~30은 공유 단계로 개인 재정비(8주 산출물 폴더 정리)·동료 1명 시연(10분)·팀 회의 5분 공유·공용 자산 1개 업로드를 수행합니다. Day 31~60은 공통화 단계로 부서 AGENTS.md·Skill 1개를 배포하고 월 1회 "부서 AI 공유회" 30분 자리를 만들어 성공·실패를 공개합니다. Day 61~90은 확장 단계로 타 부서 거점리더 1명과 MCP·Skill을 교차 사용하고 기관 표준화 제안 문서 1장 초안을 작성합니다. 90일 이후 지속 원칙은 셋 — 주 1회 빠른 파악 루틴, 월 1회 공유회, 분기 1회 네트워크 미팅.
