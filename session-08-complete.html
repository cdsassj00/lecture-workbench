API의 개념과 구조
엔드포인트·메서드·페이로드가 두렵게 쓰여 있어도 결국 "주소·동사·편지". 기술 용어를 일상어로 번역하면 진입 장벽이 사라진다.
THREE WORDS기술 용어의 인문학적 번역
API는 편지 왕래다 — 이 한 문장이 전부

API는 내가 직접 만들지 않은 함수를 네트워크 너머로 호출하는 창문이다. 창문 너머 누가 있는지 몰라도, 약속된 문법으로 말을 걸면 약속된 형식으로 답이 온다. 공공 업무의 온나라·법령·KOSIS·국민신문고를 이 창문을 통해 드나들 수 있다.

API 실패 처리 패턴
에이전트가 외부 API·MCP를 호출한다는 것은 "실패 가능성 경로"가 늘어난다는 뜻. 실패를 가정한 설계가 곧 신뢰다.
RETRY지수 백오프 · 재시도 규칙
5회 · 1·2·4·8·16초 대기 기본

대부분의 장애는 몇 초 안에 해소. 즉시 5회 재시도보다 지수 간격이 서버 부하를 줄이고 성공률을 높인다.

# tenacity 라이브러리 활용 from tenacity import ( retry, wait_exponential, stop_after_attempt, retry_if_exception_type ) @retry( wait=wait_exponential(min=1, max=30), stop=stop_after_attempt(5), retry=retry_if_exception_type( (ConnectionError, Timeout) ), ) def call_api(): return requests.get(URL, timeout=10)
RETRY 대상
재시도할 것 vs 하지 말 것

5xx·429·Timeout → 재시도. 401·403·400 → 중단(재시도해도 같은 실패).

MCP의 개념과 역할
API는 각자 다른 소켓, MCP는 모든 에이전트가 알아듣는 한 규격. 이 차이가 생태계를 바꾼다.
WHY MCPAPI와의 차이
"우리 에이전트만의 도구"가 아니라 "모두의 도구"

기존: Gmail·Slack·Notion 각각을 쓰려면 에이전트마다 개별 코드. MCP(Model Context Protocol): 도구 쪽에서 MCP 서버를 한 번 만들면 모든 MCP 지원 에이전트(Claude·Cursor·Antigravity·Copilot·Continue)가 동일하게 사용.

HOST에이전트 앱
Claude Desktop · Cursor · Antigravity

MCP를 소비하는 쪽. 사용자는 여기에 설정 한 줄만 추가.

CLIENT연결부
Host 안에 내장된 MCP 프로토콜 처리기

사용자 눈에 보이지 않음. Host 업데이트에 따라 기능이 늘어난다.

SERVER도구 제공자
tools · resources · prompts 세 가지를 노출

Gmail·Slack·내부 DB·파일시스템을 에이전트에 열어준다. 거점리더가 만들거나 시킬 수 있는 것은 바로 이 Server.

KEY TAKEAWAY
도구 개발 1번, 에이전트 생태계 전체가 사용

부서 내부 MCP 하나가 수십 명의 에이전트에 동시에 연결된다.

# server.py · Python MCP from mcp.server.fastmcp import FastMCP import sqlite3 app = FastMCP("dept-민원") DB = "./data/민원.db" @app.tool() def search(keyword: str, n: int = 5) -> list: """민원 키워드 검색""" con = sqlite3.connect(DB) rows = con.execute( "SELECT id,title,summary FROM c " "WHERE title LIKE ? " "ORDER BY created DESC LIMIT ?", (f"%{keyword}%", n) ).fetchall() con.close() return [dict(id=r[0], title=r[1], summary=r[2]) for r in rows] @app.tool() def recent(days: int = 7) -> list: """최근 N일 민원""" return query_recent(days) if __name__ == "__main__": app.run()

ROLES4역 상세
PLANNER · 기획자Opus·Sonnet
목표 → 단계 · 순서 · 검증 기준

사용자 요청을 받아 작업을 분해한다. "무엇을 만들고 어떻게 검증하는가"를 정한다. 추론 능력이 높은 모델이 유리.

EXECUTOR · 실행자Sonnet·Haiku
도구 호출 · 실제 작업 수행

MCP 도구·API·코드를 실제로 실행. 속도·안정성이 우선. Haiku급 경량 모델도 충분한 경우가 많다.

VERIFIER · 검증자별도 모델
결과 검증 · 규정 적합 · 개인정보

Executor 결과를 다른 모델이 검증한다. 같은 모델로 자기 검증하면 놓치기 쉬운 실수를 잡아낸다.

REPORTER · 보고자Haiku·Flash
사람에게 요약·보고 · 로그

최종 결과를 사람 눈높이로 요약. 감사 로그·실행 시간·사용 토큰을 정리. 경량·저렴 모델이 적합.
