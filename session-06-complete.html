조수 모드에서 위임 모드로
4·5회차 동안 여러분은 ChatGPT·Claude 채팅창에 한 문장씩 질의하고 답을 복사해 코드를 실행했습니다. 이 방식이 어시스트(조수 모드)입니다. 6회차부터는 입력 단위가 바뀝니다. 차이는 셋입니다. 첫째, 어시스트는 1턴에 한 답이지만, 에이전트(위임 모드)는 한 번의 지시로 파일 읽기·편집·터미널 실행·테스트·수정을 8~25턴 자율 반복합니다. 둘째, 입력 단위가 한 문장 프롬프트에서 데이터·처리·결과물·검수 4섹션 기획서 한 장으로 늘어납니다. 셋째, 사람이 매 단계 복사·실행하던 노동이 사라지는 대신 기획서 작성 30분 + 검수 12항목 점검이 추가됩니다.

위임 모드가 가동되는 세 장면
위임 모드가 실제로 돌아갈 때 무슨 일이 일어나는지 세 장면으로 정리합니다. 첫째, 자율 루프입니다. 에이전트가 스스로 실행해 보고, 결과를 확인하고, 고쳐서 다시 실행하는 과정을 반복해 품질을 올립니다. 둘째, 도구 호출입니다. 파일 읽기·쓰기·검색·터미널 실행·웹 탐색 같은 기능을 에이전트가 판단해 집어듭니다. 셋째, 막힘 우회입니다. 에러 트레이스백을 읽고 원인을 가설화해 다른 접근으로 돌아옵니다. 사람은 기획서와 검수 기준만 주면 됩니다.
LOOP자율 반복
SCENE 01 · 루프
조수는 한 번 답, 팀원은 될 때까지 반복

어시스트는 답을 한 번 내고 멈춘다. 에이전트는 실행해 보고, 결과 보고, 고쳐서 다시 실행한다. 같은 과제도 루프가 돌면서 품질이 올라간다.

SCENE 02 · 도구 호출
필요한 도구를 스스로 집어든다

파일 읽기·검색·편집·터미널 실행·웹 검색 — 에이전트가 판단해 호출한다. 사람은 기획서와 검수 기준만 주면 된다.

SCENE 03 · 막히면 우회
에러를 읽고 원인 가설로 재시도

의존성 문제·경로 오류·권한 차이 — 트레이스백을 읽고 다른 접근으로 돌아온다. 사람이 에러를 복사해 다시 질의할 필요가 없다.

위임 모드 전환 시점 판단
모든 업무를 에이전트로 옮기는 것이 목적이 아닙니다. 기획서 작성 시간이 구현 시간을 넘어서면 오히려 손해입니다. 판단 기준은 다섯 신호입니다. 첫째, 같은 지시를 세 번 이상 반복하고 있을 때입니다. 둘째, 파일이 다섯 개 이상 얽혀 있을 때입니다. 셋째, 독립 실행물로 배포해야 할 때입니다. 넷째, 다음 사람이 재현해야 할 때입니다. 다섯째, 결과물 검수가 필요할 때입니다. 두 개 이상이면 위임 모드, 한 개 이하면 ChatGPT·Claude 채팅으로 가는 어시스트가 적합합니다.
위임 신호에이전트로 넘어갈 때
SIGNAL 01
같은 지시를 3번 이상 반복하고 있다

"한국어로", "주석은 쉽게", "파일명 규칙" — 반복이 있으면 규칙 파일에 박아두고 위임 모드로 넘어간다.

SIGNAL 02
파일이 5개 이상 엮여 있다

4·5회차 산출물은 한 파일 한 함수. 파일 간 의존성이 있으면 에이전트가 트리 전체를 읽어야 한다.

SIGNAL 03
독립 실행물로 배포해야 한다

PyInstaller .exe(유형 3)·Electron 앱(유형 4)·HTML 단일 파일(유형 5) — 배포 단계가 붙으면 에이전트 영역.

SIGNAL 04·05
④ 다음 사람이 재현해야 한다 / ⑤ 결과물 검수가 필요하다

재현성·검수로그가 필요한 공공 업무는 에이전트 + 규칙 파일 + 로그의 조합이 가장 깔끔하다.

요구사항을 아키텍처로 그려 전달
위임 모드에서는 코드를 쓰는 대신 아키텍처로 그려 전달합니다. 세 섹션으로 나눠 진술합니다. 첫째, 데이터 섹션에는 어느 파일·어느 시트·어느 컬럼·샘플 3행까지 에이전트가 파일을 열지 않고도 구조를 알 수 있을 만큼 구체적으로 씁니다. 둘째, 처리 섹션에는 변환 규칙·분류 라벨·예외 처리·반복 구조를 에이전트가 루프를 돌 수 있는 단위로 쪼개 진술합니다. 셋째, 결과물 섹션에는 파일명·컬럼·인코딩만이 아니라 유형 3·4·5 중 어느 배포 형태까지 낼지를 명시합니다. 정체 구간에서는 어느 섹션에서 멈췄는지 단계 관점으로 되돌아가 해당 단계만 다시 진술합니다.
STEP 1 · 데이터무엇이 들어오는가
입력 데이터를 손으로 그린다

데이터 섹션은 "어느 파일, 어느 시트, 어느 컬럼, 샘플 3행"까지 손으로 그려 씁니다. 에이전트가 파일을 열지 않고도 구조를 알 수 있을 만큼 구체적이어야 합니다. 개인정보 포함 여부·공개 가능 여부 같은 취급 규칙도 이 섹션에 명시해 두면 이후 처리·결과물 단계에서 예외 판단이 흔들리지 않습니다.

# ① 데이터 - 파일: ./민원_2025Q4.xlsx - 시트: 접수 (총 1,083행) - A=접수번호 B=내용 C=접수일 - 샘플 3행: 24-0001 | 도로 포장 파손 | 2025-10-01 24-0002 | 가로등 고장 | 2025-10-01 24-0003 | 소음 민원 | 2025-10-02 - 주의: 개인정보 없음 · 공개 가능
POINT
데이터 섹션에는 처리 방법을 쓰지 않는다

"분류해서…"는 다음 단계의 말이다. 여기서는 재료의 상태만.

STEP 2 · 처리어떻게 가공하는가
처리 흐름을 단계로 쪼갠다

처리 섹션은 변환 규칙·분류 라벨·예외 처리·반복 구조를 에이전트가 루프를 돌 수 있는 단위로 번호를 붙여 쪼갭니다. 각 단계는 입력·조건·출력의 한 줄로 읽히도록 쓰고, 빈 셀이나 짧은 텍스트 같은 실패 상황의 대응도 함께 명시합니다. 대량 처리는 샘플 5~10건 시험 실행 후 사람 승인을 대기하는 조항을 넣어 1,000건 직행 사고를 막습니다.

# ② 처리 1) B열 '내용' 텍스트 추출 2) 6개 라벨로 분류 [환경/교통/복지/세무/안전/기타] 3) 빈 셀 · 짧은 텍스트(10자↓) → '미분류' 4) 신뢰도 0.6↓ → '재검토' 5) 대량(100건↑)은 샘플 5건 먼저 시험 실행 후 승인 대기
정체 구간 점검
막히면 '어느 단계'에서 막혔는지 되돌린다

1~5단계 중 어디서 에이전트가 멈췄는지 단계 관점으로 질문한다. 전체를 다시 설명하지 않는다.

STEP 3 · 결과물무엇이 나오는가
배포 형태까지 포함해 그린다

결과물 섹션은 파일명·컬럼·인코딩만 적지 않고 유형 3(.exe)·유형 4(Electron)·유형 5(HTML 단일) 중 어느 배포 단위로 낼지까지 명시합니다. 로그 경로·GUI 요소·출력 파일명 규칙을 함께 적어두면 에이전트가 부수적인 보조 파일까지 한 번에 생성합니다. 6회차부터 결과물은 스크립트가 아니라 배포물이라는 사실을 이 섹션에서 확정합니다.

# ③ 결과물 - 형태: 유형 3 (PyInstaller .exe) - GUI: tkinter · 파일 선택 버튼 1개 - 출력: 원본 동일 폴더에 민원_2025Q4_분류.xlsx - 신규 시트 '분류결과' - 컬럼: 접수번호, 원문, 분류, 신뢰도 - 로그: logs/{YYYYMMDD}.log - 배포: dist/민원분류.exe 단일 파일
정리
데이터 → 처리 → 결과물, 셋을 분리해서 진술

세 섹션이 섞이면 에이전트가 루프 중단 기준을 잃는다.

신호 02

스크립트만 나오고 .exe가 안 나오면 결과물 단계 명세 부족. PyInstaller 빌드·dist 경로를 구체화.

유형 3·4·5 — 배포 형태 3종 순회
오후 후반부 실습 블록은 배포 형태 3종을 차례로 체험합니다. 첫째, 유형 3은 PyInstaller(파이썬 스크립트를 .exe 실행 파일로 패키징하는 도구)로 tkinter GUI를 단일 .exe로 빌드하는 독립 실행 프로그램입니다. 둘째, 유형 4는 Electron(웹 기술로 데스크톱 앱을 만들게 해 주는 프레임워크)으로 HTML·CSS·JS를 데스크톱 앱으로 감싸는 크로스 플랫폼 구조입니다. 셋째, 유형 5는 한 개의 .html 파일에 CSS·JS·CDN을 모두 담아 더블클릭만으로 브라우저에서 실행되는 단일 파일 앱입니다. 넷째, 공통 원칙은 4·5회차 어시스트 산출물을 재료로 같은 과제를 두 번 수행한다는 점입니다.
유형 315:20–16:20 · 60MIN
파이썬 독립 프로그램 (.exe)

유형 3은 tkinter 또는 PyQt로 GUI를 만든 뒤 PyInstaller(파이썬 스크립트를 .exe 실행 파일로 패키징하는 도구)로 단일 .exe 파일을 빌드하는 배포 형태입니다. 파이썬 런타임과 의존성이 통째로 패키징되기 때문에 파이썬이 설치되지 않은 동료 PC에서도 더블클릭만으로 실행됩니다. 5회차에 어시스트로 만든 .py 스크립트는 같은 폴더에 main.py·requirements.txt·민원분류.spec를 추가한 뒤 `pyinstaller --onefile --windowed --name 민원분류 main.py` 한 줄로 dist/민원분류.exe(약 10~50MB) 한 개를 만들어 동료에게 USB·메일로 전달합니다. 공공 현장에서 수요가 가장 높은 배포 경로입니다.

# 유형 3 · 핵심 명령 pip install pyinstaller tkinter pyinstaller --onefile \ --windowed \ --name "민원분류" \ main.py # 결과: dist/민원분류.exe

유형 416:30–17:00 · 30MIN

유형 4는 Electron(웹 기술로 데스크톱 앱을 만들게 해 주는 프레임워크)이 HTML·CSS·JS로 만든 화면을 데스크톱 앱으로 감싸는 형태입니다. 같은 코드 한 벌로 Windows·macOS·Linux에서 동일한 UI가 나오며 4회차에 만든 웹 자산이 그대로 재활용됩니다. 30분 세션은 맛보기 수준으로 구조 감각과 빌드 한 번 성공을 목표로 합니다. 여러 파일을 한꺼번에 다뤄야 하므로 Cursor Composer나 Antigravity Cascade가 유리합니다.

# 유형 4 · 핵심 명령 npm init -y npm install --save-dev electron # main.js · index.html 구조 npm start # 빌드 (electron-builder) npm install -D electron-builder npm run dist

유형 517:00–17:30 · 30MIN
HTML+CSS+JS 단일 파일 앱

유형 5는 한 개의 .html 파일 안에 HTML 마크업·CSS·JS 로직·외부 라이브러리 참조를 모두 담는 형태입니다. 더블클릭만으로 브라우저에서 즉시 실행되며 설치·권한 승인·빌드가 모두 필요 없어 공공 환경에서 가장 간편한 배포 경로입니다. 내부망 이관 시에는 CDN을 ./vendor/ 폴더의 로컬 파일로 교체해 인터넷 차단 환경에서도 동일하게 돌아가도록 만듭니다. 메일 첨부·공유 드라이브·USB 어느 경로로도 공유 가능해 감사 로그가 가장 단순합니다.
