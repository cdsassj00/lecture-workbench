자기소개 + Fuse.js 챗봇 리버스엔지니어링

강사가 만든 사이트 https://cdsassj00.github.io/2026-proposals-rss/ 를 함께 연다. 이 사이트의 챗봇은 진짜 LLM이 아니다. 검색 라이브러리 Fuse.js가 미리 들어있는 문서 더미에서 키워드 점수가 높은 문장을 골라 보여주는 '챗봇 흉내내기'다. 거점리더가 외부망 LLM 없이도 부서 매뉴얼 답변기를 만들 수 있다는 가장 가벼운 증거다.

오늘 같이 분해할 것

· 사이트가 어떤 데이터(질문·답변 페어, 제안서 본문)를 들고 있는가
· 사용자 입력이 어떤 단계로 점수가 매겨지는가 (Fuse.js 옵션)
· 왜 답이 매끄럽지 않은가 — LLM이 아니기 때문에 생기는 한계
· 같은 구조를 우리 부서 매뉴얼에 어떻게 옮길 수 있는가

왜 첫 시간에 이걸 보는가

거점리더 후보가 많은 경우 "AI 챗봇 = LLM"이라는 등식을 갖고 들어온다. 하지만 공공 현장에서는 외부망 LLM을 못 쓰는 자리가 더 많다. Fuse.js 같은 클라이언트 검색 라이브러리만으로도 매뉴얼 답변 챗봇은 만들어진다. 첫 시간에 이 차이를 손으로 만져보고 들어가야, 이후 ChatGPT·Claude·Gemini 사용법을 배울 때 '어디에 진짜 LLM이 필요한지' 자기 기준이 선다.

오늘 자기소개에서 이름·소속만 말하지 않습니다. "내 부서에서 매뉴얼 답변기가 필요한 자리"를 한 줄씩 같이 적습니다.

1주차 과제 리뷰 — Hugging Face SLM

1주차 과제는 Hugging Face에서 SLM(Small Language Model) 한 개를 골라 스펙을 정리하는 것이었다. 각자 조사한 결과를 합쳐 48개 SLM 카드 사이트로 정리할 예정이다. 사이트가 완성되면 본 강의 동안 "내 환경에서 어떤 sLLM을 돌릴 수 있을까"를 매번 들춰보는 매뉴얼로 쓴다.

A.각 카드에 들어가는 정보

· 모델명 / 베이스 모델 / 파라미터 수
· 컨텍스트 길이 · 라이선스 · 한국어 지원 여부
· 권장 VRAM (양자화별 4bit · 8bit · 16bit)
· 강점 1줄 / 약점 1줄 / 추천 사용처
· 다운로드 링크 + 같이 보면 좋은 모델

조사가 부족한 칸은 비워둡니다. 채워지는 대로 사이트가 자동 갱신되도록 합니다.

B.사이트는 매뉴얼로 쓴다

5주차에 유형 02 (VBA + 로컬 sLLM), 유형 07 (Ollama 로컬 RAG)를 다룰 때 거점리더 각자가 자기 PC에 맞는 sLLM을 고를 수 있어야 한다. 그때마다 이 48개 카드 사이트를 펼쳐서 "내 노트북 RAM이 16GB이면 어디까지 돌릴 수 있는가"를 보고 결정한다.

이 사이트는 거점리더 기수가 바뀌어도 차기 기수의 시드 라이브러리로 그대로 인계된다. 1주차 과제가 단발성 보고서로 끝나지 않는 이유다.

오늘 30분 안에는 발표 5명 × 4분 + 다 같이 사이트 골격 한 번 보기. 미제출자는 다음 주차 직전까지 보강.

ChatGPT 팁 + 약간의 바이브코딩

ChatGPT의 가장 큰 강점은 '결과물 형태가 다양하다'는 점이다. 텍스트 답변뿐 아니라 SVG · HTML · Mermaid 같은 화면에서 바로 검증 가능한 산출물을 즉석에서 만들 수 있다. 거점리더가 동료에게 '말로 설명' 대신 '한 장 도식'을 빨리 들이밀 수 있게 되는 자리다.

A.오늘 즉석에서 만들 3가지

① SVG 도식 한 장
"우리 부서 민원 처리 흐름을 4단계로, 박스와 화살표로 SVG로 그려줘. 폭 1200, 색은 차분하게."

② HTML 한 페이지
"방금 SVG를 가운데 두고, 위에 제목, 아래 4단계 설명을 카드 4개로 배치한 단일 HTML 파일로 만들어줘. 외부 라이브러리 쓰지 마."

③ Mermaid 다이어그램
"위 흐름을 Mermaid flowchart 코드로도 줘. 결정 분기 1개 포함."

B.몇 가지 작은 팁

· 캔버스(Canvas) 모드로 켜면 결과물을 옆 패널에서 계속 다듬을 수 있다. 도식 1개 만들고 끝나지 않고 6~7번 다듬는 것이 정상.

· 모델 선택은 일을 시키기 전에 결정. 도식·코드는 추론 모델, 빠른 글짓기는 일반 모델.

· 첨부 파일은 PDF·엑셀·이미지까지. 첨부 후 "이 파일에서 ~만 뽑아줘"가 가장 쓸모 있다.

· "한 번 더 검토해서"는 거의 항상 결과를 한 단계 끌어올린다. 바로 만족하지 말 것.

실습 산출물(SVG · HTML · Mermaid 3종)은 구글드라이브 공용 폴더 02-chatgpt/내이름/에 업로드. 다음 슬라이드 토의에서 다시 꺼낸다.

Claude의 강점은 긴 문서를 안정적으로 다루는 능력과 도구 연결(MCP)이다. 거점리더가 부서 문서·조례·회의록을 묶어 일하는 자리에 잘 맞는다. 6주차 Claude Code 본격 학습 전에, 오늘은 외부망 웹 클라이언트 단에서 쓸 수 있는 부분을 정리한다.

A.Projects · 작업 단위로 묶는다

한 미션에 쓰이는 문서·지침·지난 대화를 하나의 Project로 묶는다. 매번 "내 부서 상황은 ~"을 다시 설명하지 않아도 된다. 거점 미션 1건 = Project 1개가 거점리더의 기본 운영 단위가 된다.

B.Artifacts · 결과물을 옆 창에서

HTML · React 컴포넌트 · 마크다운 문서 · SVG 등을 본문 답변과 분리된 옆 패널에 띄워 계속 다듬는다. ChatGPT의 Canvas와 가장 비슷한 자리. 한 산출물을 6~10번 다듬는 흐름에 잘 맞는다.

C.MCP · 도구를 붙인다

MCP(Model Context Protocol)는 "Claude가 쓸 수 있는 도구 모음의 표준 인터페이스"다. 외부망 웹 클라이언트에서도 일부 MCP 커넥터(GitHub · 구글드라이브 등)를 직접 켤 수 있다. 6주차에 본격적으로 다룰 주제지만, 오늘은 "내 클로드가 GitHub 레포를 직접 읽는다"를 1번 시연한다.

D.ChatGPT · Gemini와의 차이

· 긴 컨텍스트 · 안정적 코딩 — Claude
· 멀티모달 · 영상 · 구글 생태계 — Gemini
· 도구 다양성 · 캔버스 워크플로우 — ChatGPT
셋 다 깔아두고 일에 따라 골라 쓰는 것이 거점리더 기본기.

"만약에 이 기능이 있다면"
우리 기관·내 업무 어디에 붙이고 싶은가

오전부터 본 ChatGPT·Gemini·Claude·NotebookLM·노코드 도구를 머릿속에 펼쳐둔다. 그리고 보안망·외부망을 잠시 잊는다. 우리 부서에서 가장 시간 잡아먹는 작업, 가장 자주 반복되는 답변, 가장 자주 빠뜨리는 점검을 떠올린다. 그 자리에 어떤 기능을 붙이고 싶은가.

발언 양식 — 한 사람 60초

· 누가 겪는 일인가 (시민 / 민원인 / 동료 공무원 / 본인)
· 언제 일어나는가 (트리거)
· 지금 막혀 있는 한 지점
· 만약 어떤 기능이 있다면 그 자리에 들어가게 될지

규칙 하나. "실현 가능한가"는 지금 묻지 않는다. 보안 · 예산 · 권한은 잠시 옆에 두고 결핍을 그대로 말한다. 거점리더의 첫 자산은 "우리 기관에서 진짜 막혀 있는 자리들의 목록"이다. 8주 동안 그 목록 위를 다닌다.

강사 추천 사이트 + 실습결과 공유

강사가 매일 만져보고 살아남은 사이트들만 추렸다. 사이트만 소개하고 끝나지 않는다. 한 사이트당 즉석 실습 1개 → 결과물을 구글드라이브 공용 폴더에 올린다. 다른 거점리더가 같은 사이트를 더 잘 활용할 수 있게 만드는 것이 핵심.

A.오늘 함께 둘러볼 곳

워크벤치 강의안에 미리 박아둔 사이트가 있다. 카테고리는 대략 5갈래.

· 대화형 LLM · ChatGPT · Claude · Gemini · Grok
· 문서 / 데이터 · NotebookLM · Perplexity · Felo
· 미디어 생성 · Flow · Runway · ElevenLabs
· 노코드 / 바이브코딩 · Lovable · Bolt · v0 · Replit
· 전문 플랫폼 · Hugging Face · Ollama · LangChain

목록은 매 기수마다 갱신됩니다. references/02-ai-lecture-sites.html 참고.

B.실습결과 공유 약속

· 구글드라이브 폴더 02-curated/<사이트명>/<내이름>
· 결과물 + 사용한 프롬프트 .md 같이 올림
· 1줄 후기 — "이 사이트는 ~할 때 좋고, ~할 때는 막힌다"

거점리더는 산출물을 자기 책상에 두지 않는다. 다른 거점리더가 같은 입력으로 같은 결과를 다시 만들 수 있어야 산출물이 완성된다.

사이트 소개를 듣기만 하지 않습니다. 40분 안에 사이트 3개 × 실습 1개씩이 기본 목표. 막히면 옆자리와 같이 풉니다.

GitHub + 노코드 바이브코딩 도구

1주차 과제(SLM 보고서 GitHub 등록)로 모두 GitHub 가입을 마쳤다. 오늘은 그 위에 노코드 바이브코딩 도구 4종을 얹어, 각자 부서 미션의 '가벼운 프로토타입'을 30분 안에 만들 수 있는 손길을 익힌다. 셋업은 끝나 있다 — 이제 손을 움직인다.

①
Lovable — 자연어로 풀스택 웹앱을 만든다. Supabase 연결이 자연스럽다는 점이 강점. 부서 내부 도구의 1차 프로토타입에 잘 맞는다.실습 — "민원 분류 입력 폼 + 결과 표시 페이지"를 자연어로 한 번 생성, GitHub 푸쉬까지.
②
Bolt — 브라우저 안에서 풀스택 환경이 통째로 돌아간다. 속도가 빠르고 기술 스택이 자유롭다. 단발성 데모 만들 때 가장 가벼움.실습 — "내 부서 안내 한 페이지" 정적 사이트 1개를 5분 안에.
③
v0 — Vercel이 만든 도구. UI 컴포넌트를 자연어로 디자인하는 데 특화. 화면 스케치를 즉석에서 React 컴포넌트로 옮길 때 좋다.실습 — "공공기관 민원 대시보드" 한 화면을 컴포넌트로 생성, 미세 조정.
④
Replit — 클라우드 IDE + 에이전트 결합. 코드까지 직접 보고 만지고 싶을 때 좋다. 6주차 이후 코딩에이전트로 옮겨가기 전 중간 자리.실습 — Lovable에서 만든 결과물을 Replit으로 가져와 한 부분만 손으로 수정해본다.
네 도구 모두 GitHub 연동이 핵심. 오늘 산출물은 모두 자기 GitHub 레포로 푸쉬됩니다. 다음 주차부터는 이 레포가 거점 미션의 본거지가 됩니다.
