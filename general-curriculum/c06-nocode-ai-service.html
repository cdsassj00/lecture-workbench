노코드 AI 서비스 구현

Section 01. 교재 안내와 학습 설계

이 교재는 2026년 AI 데이터기반지능행정 역량강화 사업 Track 2 Module 3 「노코드 AI 서비스 구현」 과정을 책형 원고로 재구성한 문서다. 원본 PPTX는 단순히 최신 도구 이름을 빠르게 나열하는 자료가 아니라, 비개발 실무자가 실제로 작동하는 서비스를 손에 쥐도록 만드는 구현 중심 과정을 설계한 자료다. 따라서 본 교재는 슬라이드 문장을 옮기는 수준을 넘어서, 왜 이 순서로 실습해야 하는지, 어떤 부분에서 보안과 운영 판단이 필요한지, 어떤 산출물까지 만들어야 학습이 끝났다고 볼 수 있는지를 문장으로 풀어 설명한다.

이 과정의 핵심은 두 문장으로 요약할 수 있다. 첫째, 코드를 외워서 직접 타이핑하는 능력보다 무엇을 만들지 정확히 말하고 받은 결과를 검수하는 능력이 더 중요해졌다. 둘째, 공공부문에서는 화려한 클라우드 서비스보다도 내부망, 부서 PC, 기존 엑셀과 한글 양식을 어떻게 활용 가능한 자동화 자산으로 바꾸는지가 더 실질적이다. C06은 바로 그 두 지점을 잇는다. Day 1에는 Python, 단일 HTML, 브라우저 저장소, VBA처럼 가볍고 즉시 실행 가능한 도구를 다루고, Day 2에는 Ollama, Lovable, Stitch, Supabase, GitHub 배포까지 확장해 부서 단위 서비스를 완성한다.

또한 이 과목은 C05 「AI 행정 융합 기획」과 C08 「바이브 코딩 LLM 서비스 개발」 사이의 가교 역할을 한다. C05가 문제를 정의하고 기획 문장으로 바꾸는 능력을 다루었다면, C06은 그 기획을 실제 작동하는 도구와 서비스로 옮기는 구현 감각을 다룬다. 반대로 C08은 이 구현 감각을 바탕으로 LLM 서비스 자체를 설계하고 인프라 수준까지 다루는 과정이다. 즉 C06은 기획의 언어를 구현의 언어로 바꾸는 단계다.

학습 대상

  • 외부 개발자에게 모든 구현을 맡기기 전에 직접 프로토타입을 만들고 싶은 공공 실무자
  • 반복 문서, 엑셀 통합, 간단한 웹 도구, 사내 지식 정리 같은 문제를 스스로 줄이고 싶은 행정 담당자
  • 정보화 담당자와 협업할 때 최소한의 구조와 위험을 설명할 수 있어야 하는 기획 담당자
  • 폐쇄망 또는 반폐쇄 환경에서 AI를 어떻게 활용할지 현실적인 대안을 찾는 조직 구성원

학습 이후 기대 성과

  • 채팅형 LLM과 코딩 에이전트의 차이를 설명하고 과업에 맞는 작업 방식을 고를 수 있다.
  • Python, 단일 HTML, localStorage, VBA, Ollama, Lovable 가운데 어떤 도구가 현재 문제에 가장 적합한지 판단할 수 있다.
  • 공공부문 데이터와 문서를 외부에 넘기지 않으면서도 외부 LLM의 지식과 생산성을 활용하는 비대칭 작업 구조를 설계할 수 있다.
  • 부서 단위 자동화 도구 1종과 공개 가능한 웹 서비스 1종을 실제로 완성하고 설명할 수 있다.
  • 구현 결과를 README, 운영 체크리스트, 권한 설계, 배포 기록까지 포함한 조직 자산으로 정리할 수 있다.

Section 02. 바이브 코딩과 구현 중심 학습의 의미

원본 슬라이드는 처음부터 기술 기능보다 역할 변화와 태도 변화를 먼저 강조한다. 이것은 이 과목의 방향을 정확히 보여 준다. 지금까지의 실무 환경에서 개발자는 구현을 맡고, 기획자는 요구사항을 쓰고, 행정 실무자는 결과물을 받는 위치에 있었다. 그러나 생성형 AI와 코딩 에이전트가 등장하면서 이 경계가 크게 흐려졌다. 중요한 것은 모든 사람이 전문 개발자가 되어야 한다는 뜻이 아니라, 문제를 가장 정확히 아는 사람이 초기 구현을 직접 만들고 검수할 수 있는 시대가 되었다는 점이다.

바이브 코딩은 코드를 직접 타이핑하는 행위를 중심에 두지 않는다. 대신 무엇을 만들고 싶은지, 누가 쓰는지, 어떤 데이터를 다루는지, 어떤 제약이 있는지를 자연어로 설명하고 그 결과를 검수하며 반복해서 보정하는 흐름을 중심에 둔다. 다시 말해 입력은 문법이 아니라 의도이고, 핵심 역량은 암기가 아니라 요구사항 설계와 검수다. 공공부문 실무에서는 이 전환이 특히 중요하다. 현장에서 가장 큰 병목은 대개 복잡한 알고리즘이 아니라 반복적인 보고서, 문서 양식, 민원 분류, 일정 관리, 내부 규정 검색, 작은 계산 도구 부재 같은 문제이기 때문이다.

또한 이 과목은 채팅형 LLM과 코딩 에이전트를 분리해 이해할 것을 요구한다. 채팅형 LLM은 코드 블록을 받아보는 데는 유용하지만, 프로젝트 전체를 읽고 여러 파일을 동시에 수정하며 실행 오류를 기준으로 반복 수정을 이어 가는 데는 한계가 있다. 반면 코딩 에이전트는 IDE 안에서 프로젝트 문맥을 함께 다루고, 파일 구조를 이해한 상태로 구현을 진행할 수 있다. 따라서 단일 HTML 파일 하나나 짧은 VBA 매크로 수준에서는 채팅형 LLM만으로도 충분할 수 있지만, Python 자동화, 다중 파일 웹앱, 배포 구조처럼 조금만 범위가 넓어져도 코딩 에이전트의 효율이 크게 올라간다.

내부망 AI 보조 Ollama, 로컬 SLM 간단한 요약, 분류, 문장 다듬기, 규칙형 질의응답 작은 모델일수록 프롬프트와 검수의 비중이 커짐
외부 공개 서비스 Lovable, Supabase, GitHub, Netlify, Vercel 게시판, 신청서, 예약, 승인 흐름, 공지 서비스 계정, 권한, 환경변수, 개인정보 처리 기준을 함께 설계해야 한다

바이브 코딩 작업 루프

이 과목이 반복해서 훈련시키는 작업 루프는 단순하다. 첫째, 무엇을 만들지 자연어로 정리한다. 둘째, 코드나 화면 초안을 받는다. 셋째, 실행해 보고 차이를 발견한 뒤 다시 요청한다. 이 세 단계는 작고 짧게 반복될수록 좋다. 한 번에 완성된 결과를 기대하기보다, 10분 안에 확인 가능한 단위로 나누어 받아보는 것이 효율적이다.

이 단순한 루프를 교재형으로 더 풀어 쓰면 사실상 여섯 단계다. 먼저 문제를 문장으로 고정한다. 다음으로 입력과 출력의 경계를 적는다. 그다음으로 AI에게 초안을 요청한다. তারপর 실제 실행 환경에서 돌린다. 이어서 결과와 오류를 비교해 수정 요청을 다시 던진다. 마지막으로 README와 사용 절차를 남긴다. 많은 초보자가 세 번째 단계까지만 하고 끝내기 때문에, 그 결과물이 데모에 머문다. 반대로 다섯 번째와 여섯 번째 단계까지 가는 팀은 같은 도구를 써도 훨씬 교재다운 산출물을 만든다.

여기서 중요한 것은 AI가 대신 생각해 준다는 기대보다, 사람이 생각을 잘게 쪼개 전달해야 한다는 사실이다. 예를 들어 “부서 보고 자동화 도구를 만들어 달라”는 요청은 너무 크다. 반면 “폴더 안 CSV를 읽고 누락 부서를 찾아 엑셀 요약 시트를 만드는 Python 스크립트”라고 적으면 AI도 더 안정적으로 응답한다. 즉 바이브 코딩의 본질은 감으로 말하는 것이 아니라, 감을 구조로 바꾸는 일이다. 이 점을 이해하지 못하면 결과는 늘 그럴듯하지만 쓸 수 없는 초안으로 남기 쉽다.

좋은 프롬프트보다 먼저 필요한 문제 분해

실무자들은 종종 “프롬프트를 잘 써야 한다”는 말부터 듣는다. 그러나 실제로 더 중요한 것은 문제 분해다. 같은 수준의 한국어 문장을 써도, 어떤 사람은 금방 쓸 만한 결과를 받고 어떤 사람은 계속 엉뚱한 답을 받는다. 차이는 문장의 미사여구가 아니라, 입력 구조와 금지 조건을 얼마나 분명히 적었는지에서 벌어진다. 따라서 C06에서는 프롬프트 자체를 문학적으로 다듬는 훈련보다, 문제를 다음 네 요소로 쪼개는 훈련이 먼저여야 한다. 누구를 위한 도구인가, 무엇을 넣는가, 무엇이 나와야 하는가, 무엇은 하면 안 되는가.

이 네 요소를 먼저 적고 나면 대부분의 구현은 생각보다 빨리 구체화된다. 예를 들어 단일 HTML 계산기든 Lovable 웹앱이든, 결국 입력 필드, 계산 규칙, 저장 여부, 권한 여부를 설명해야 한다. 즉 좋은 프롬프트는 화려한 문장이 아니라 설계서의 압축본이다. 이 관점이 잡히면 사용자는 AI가 내놓은 코드를 신비한 결과물로 보지 않고, 내가 요청한 설계가 구현된 초안으로 보게 된다.

실무자가 자주 실패하는 요청 패턴

  • 문제 설명보다 도구 이름이 먼저 나오는 경우
  • 입력 데이터 구조를 적지 않고 “엑셀 자동화”처럼 뭉뚱그리는 경우
  • 금지 조건을 적지 않아 민감정보 전송이나 과도한 기능이 섞이는 경우
  • 실행 환경을 적지 않아 Windows 경로, 브라우저, 설치 조건이 어긋나는 경우
  • 초안만 받고 실제 실행과 검수를 건너뛰는 경우

이 다섯 패턴은 단순 실수 같지만, 결과적으로 교재를 얇게 만드는 원인이기도 하다. 설명이 얕으면 요청도 얕아지고, 요청이 얕으면 결과도 슬라이드 재진술 수준에 머무르기 때문이다.

실무 적용 질문

  1. 지금 내 업무에서 가장 자주 반복되지만 아직 도구로 정리되지 않은 문제는 무엇인가.
  2. 그 문제는 단일 HTML, Python, VBA, 클라우드 웹앱 가운데 어떤 수준의 구현으로 시작하는 것이 가장 안전한가.
  3. 결과를 검수할 사람과 실제로 쓸 사람과 운영 책임을 질 사람이 각각 누구인가.

Section 03. 폐쇄망 우선 전략과 도구 선택 원칙

공공조직이 AI 도구를 다룰 때 가장 먼저 부딪히는 질문은 기능이 아니라 경계다. 외부에 무엇을 보내도 되는가, 내부망에서만 처리해야 하는가, 테스트와 운영을 어디에서 구분할 것인가, 비공개 정보와 공개 가능한 정보를 어떤 기준으로 나눌 것인가 같은 질문이 먼저 나온다. 따라서 C06은 처음부터 클라우드 웹앱을 만들지 않는다. Python, 단일 HTML, 브라우저 저장소, VBA처럼 내부 PC에서 바로 실행 가능한 구조를 먼저 체득한 뒤, 그다음에 Ollama와 클라우드 노코드 풀스택으로 확장한다. 이 순서는 학습 난이도 때문이기도 하지만, 무엇보다 조직의 보안 감수성을 존중하는 순서이기 때문이다.

폐쇄망 우선 전략은 외부 도구를 전혀 쓰지 말자는 뜻이 아니다. 핵심은 외부 지식과 내부 데이터를 분리하는 것이다. 예를 들어 외부 LLM에는 실제 민원 데이터나 인사 정보를 보내지 않고, 파일 구조와 컬럼명, 가짜 샘플 몇 줄만 설명한 뒤 코드를 받아온다. 그 코드는 내부망 PC나 부서 PC에서 실제 데이터를 대상으로 실행한다. 이 구조를 익히면 “외부 AI를 쓰면 곧바로 데이터 유출”이라는 이분법에서 벗어나, 무엇을 보내고 무엇을 보내지 않는지 설계하는 수준으로 사고가 올라간다.

또한 도구 선택은 기술 유행이 아니라 문제 규모와 운영 범위로 결정해야 한다. 혼자 쓰는 계산기라면 단일 HTML이 가장 좋다. 같은 부서 동료 여러 명이 기록을 공유해야 한다면 브라우저 저장소만으로는 한계가 있으므로 데이터베이스가 필요하다. 설치가 어려운 조직이라면 Python보다 VBA가 더 현실적일 수 있다. 외부 공개와 다중 사용자 서비스가 목적이라면 Lovable과 Supabase 같은 클라우드 조합이 맞다. 즉 좋은 도구란 가장 강력한 도구가 아니라, 현재 조직에서 가장 적은 저항으로 문제를 해결하는 도구다.

도구 선택의 네 가지 기준

  • 사용자 수: 혼자 쓰는가, 부서가 같이 쓰는가, 외부 사용자가 접근하는가
  • 데이터 민감도: 민감 정보인가, 공개 가능한 정보인가, 샘플로 대체 가능한가
  • 운영 기간: 하루짜리 행사 페이지인가, 장기 운영 서비스인가
  • 유지 인력: 만든 사람이 떠난 뒤에도 남이 읽고 고칠 수 있는가

실행 우선순위 원칙

  1. 개인용 자동화는 로컬에서 먼저 완성한다.
  2. 조직 공유가 필요하면 저장 구조와 권한 구조를 먼저 설계한다.
  3. 외부 공개가 필요하면 GitHub, 배포, 환경변수, 로그 기준을 함께 정리한다.
  4. 최종 운영 단계에서는 README와 관리자 인수인계 문서를 반드시 남긴다.

Section 04. M1 프로젝트 목표 정의 워크숍

원본 슬라이드는 첫 모듈을 인트로로 두면서도 마지막 산출물을 분명히 제시한다. 그것은 2일 프로젝트 목표 1페이지다. 이 한 장이 중요한 이유는 구현 과정에서 방향을 잃지 않게 해 주기 때문이다. 실제 교육 현장에서는 도구를 만지기 시작하는 순간 흥미가 커지고, 그만큼 본래 풀고자 했던 문제는 흐려지기 쉽다. 따라서 M1의 목적은 도구 설명이 아니라, 이후 모든 실습이 하나의 문제를 향해 수렴되도록 질문을 고정하는 데 있다.

좋은 프로젝트 목표서는 복잡하지 않다. 누구의 어떤 불편을 줄이려는지, 어떤 입력이 들어오고 어떤 결과가 나와야 하는지, 처음 구현은 어느 수준까지 할 것인지를 적으면 된다. 여기서 중요한 것은 거창함보다 명료함이다. 예를 들어 “부서 혁신 포털 구축”보다는 “주간 보고서를 매주 손으로 취합하는 시간을 2시간 줄이는 자동 통합 도구”가 훨씬 좋은 시작점이다. 문제의 범위가 명확해야 사용할 도구와 데이터 범위, 검수 방식, 배포 수준도 자연스럽게 정해진다.

Section 05. M2 Python 업무자동화

Python은 공공부문 자동화에서 여전히 가장 범용적인 도구다. 이유는 단순하다. 파일을 읽고, 표를 정리하고, 텍스트를 추출하고, 폴더를 훑고, 문서를 생성하는 작업에 필요한 생태계가 이미 충분히 성숙했기 때문이다. 동시에 Python은 코드 길이가 짧고 설명이 쉬워 LLM과의 협업에도 잘 맞는다. 즉 Python은 복잡한 시스템 개발 언어라기보다, 반복 업무를 스크립트로 바꾸는 접착제에 가깝다.

다만 공공부문에서 Python 학습은 “언어를 정석대로 배운다”는 방식보다 “문제를 해결하기 위한 실행 환경을 만들고, 필요한 패키지를 설치하고, 받은 코드를 돌려보며 고친다”는 방식이 더 현실적이다. 그래서 M2는 아나콘다와 python.org 설치를 모두 보여 준다. 한 조직에서는 아나콘다가 패키지 관리 측면에서 편할 수 있고, 다른 조직에서는 최소 설치만 허용되어 python.org가 더 현실적일 수 있다. 중요한 것은 어느 설치 방식을 택하더라도, 이후의 실행과 검수가 가능해야 한다는 점이다.

공공부문 Python 실습 운영 원칙

첫째, 실제 데이터를 외부에 보내지 않는다. 대신 시트 구조, 파일 경로 예시, 컬럼명, 가짜 샘플 5행 정도만 설명하고 코드를 받는다. 둘째, 테스트용 작은 파일로 먼저 검증한다. 셋째, 라이브러리는 한꺼번에 많이 깔기보다 현재 과업에 필요한 것만 설치한다. 넷째, 실행 결과와 오류를 남겨 다음 사람이 같은 환경을 복원할 수 있도록 기록한다. 이 네 가지 원칙만 지켜도 Python 자동화는 충분히 안전하고 재현 가능하다.

이 원칙을 실제 운영 흐름으로 바꾸면 다음과 같다. 먼저 샘플 데이터를 만든다. 다음으로 작업용 폴더와 출력용 폴더를 분리한다. 그다음 가상환경을 만들고 필요한 패키지만 설치한다. 이후 스크립트를 작은 파일로 시험 실행하고, 마지막에 실제 업무 파일에 확대 적용한다. 중요한 것은 “코드를 받았다”에서 끝나지 않고 “재현 가능한 실행 경로를 남겼다”까지 가는 것이다. Python 자동화가 개인 재주에서 조직 자산으로 넘어가려면 결국 누가 실행해도 같은 결과가 나와야 한다.

venv를 왜 반드시 이해해야 하는가

Python 공식 문서는 venv를 가벼운 독립 실행 환경으로 설명한다. 각 프로젝트는 자신만의 인터프리터와 패키지 집합을 가질 수 있고, 기본 환경과 격리된 상태로 동작한다. 실무적으로 이 설명은 매우 중요하다. 같은 PC 안에 여러 Python 프로젝트가 있을 때, 패키지 버전이 섞이면 문제 원인을 찾기 어려워지기 때문이다. 공공부문에서는 특히 “한 번은 됐는데 다음 PC에서는 안 된다”는 식의 혼란이 흔하다. 가상환경을 쓰면 적어도 어떤 패키지가 어디에 설치되었는지를 프로젝트 단위로 분리할 수 있다.

또한 공식 문서는 가상환경을 이동 가능한 자산이 아니라 “지우고 다시 만들 수 있는 환경”으로 보는 편이 맞다고 설명한다. 즉 .venv 폴더 자체를 오래 들고 다니기보다, 어떤 Python 버전과 어떤 패키지 목록으로 다시 만들 수 있는지가 더 중요하다. 이 점은 교재에도 반드시 들어가야 한다. 왜냐하면 많은 초보자가 환경 폴더를 통째로 복사하는 방식으로 문제를 해결하려 하지만, 경로가 바뀌면 다시 깨질 수 있기 때문이다. 따라서 실무에서는 .venv는 버리고 다시 만들 수 있어야 하고, 대신 requirements.txt나 설치 메모가 남아 있어야 한다.

Windows 환경에서는 활성화 스크립트와 실행 정책 문제도 자주 나온다. 공식 문서는 PowerShell에서 Activate.ps1을 쓸 때 실행 정책 때문에 막힐 수 있으며, 필요시 사용자 범위에서 정책 조정이 필요할 수 있다고 안내한다. 하지만 교재 수준에서는 더 실용적인 태도가 필요하다. 활성화가 막히더라도 .venv\Scripts\python.exe를 직접 지정하면 대부분의 스크립트는 실행된다. 즉 가상환경은 “반드시 활성화해야만 쓰는 것”이 아니라 “올바른 인터프리터 경로를 지정해 주면 되는 것”이라고 이해하는 편이 훨씬 덜 헷갈린다.

Python 자동화의 1일 워크플로우

다음 순서대로 진행하면 실무 도입 실패가 크게 줄어든다.

  1. 업무를 파일 단위로 적는다.
    예: 월별 CSV 12개를 통합하고 누락 부서를 표시한다.
  2. 샘플 입력과 기대 출력 1개씩을 만든다.
    예: 입력 샘플 폴더 3개, 기대 결과 엑셀 1개.
  3. 가상환경을 만들고 패키지를 최소 설치한다.
    예: pandas, openpyxl만 먼저 설치.
  4. 스크립트를 함수 단위로 나눈다.
    예: 읽기, 검증, 통합, 저장을 분리.
  5. 테스트 로그를 남긴다.
    예: 몇 개 파일을 읽었는지, 누락 부서가 무엇인지 출력.
  6. 실제 파일 확대 적용 후 눈검수를 한다.
    예: 무작위 3건 확인, 합계 검토.

이 흐름을 몸에 익히면 Python은 더 이상 어려운 개발 언어가 아니라, 반복 문서를 처리하는 작업 절차가 된다.

공공 실무 예시: 월간 민원 CSV 통합

민원 분류 예시를 들어 보자. 각 부서가 월말에 CSV를 보내고, 총괄부서는 이를 통합해 현황표를 만든다고 하자. 이때 AI에게 바로 “민원 시스템 만들어 달라”고 요청할 필요는 없다. 먼저 Python으로 폴더 안 CSV를 읽고, 필수 컬럼이 빠진 파일을 걸러내고, 날짜 형식을 맞추고, 부서별 건수를 집계하는 스크립트부터 만든다. 이 단계만 되어도 기존 수작업 보고 시간은 크게 줄어든다. 이후 필요하면 이 결과를 HTML 대시보드나 Lovable 조회 화면으로 넘기면 된다.

이 예시는 C06이 왜 Day 1과 Day 2를 나누는지 설명해 준다. 많은 사용자가 곧바로 웹앱부터 만들고 싶어 하지만, 실제로는 데이터가 먼저 정리되어야 화면이 의미를 갖는다. 즉 Python은 화려하지 않지만, 서비스의 바닥을 만드는 기술이다.

환경 선택 판단 기준

아나콘다가 편한 조직은 데이터 분석 패키지 설치와 가상환경 관리가 빠르다는 장점이 있다. 반면 설치 권한이 제한적이거나 최소 환경만 허용되는 조직은 python.org 설치가 더 현실적일 수 있다. 이때 중요한 것은 어느 환경이 더 고급인가가 아니라, 교육 당일과 이후 운영 시점에 같은 실행 환경을 다시 재현할 수 있는가다. 따라서 실습 시작 전에 파이썬 버전, 설치 경로, 가상환경 이름, 주요 패키지 버전을 반드시 메모해 두어야 한다.

또한 부서 PC에서는 설치보다 승인 절차가 더 오래 걸릴 수 있다. 정보보호 담당자에게 필요한 설명은 보통 세 가지다. 무엇을 설치하는지, 네트워크 연결이 필요한지, 생성된 결과물이 어디에 저장되는지다. 학습자는 이 설명을 기술 용어가 아니라 행정 언어로 정리해야 한다. 예를 들어 "외부 사이트에 민감정보를 전송하지 않고 내부 PC에서만 파일을 처리한다"는 문장은 설치 허용을 얻는 데 매우 중요하다.

실습 1. PDF에서 텍스트와 표를 추출해 엑셀로 옮기기

이 실습은 PDF를 보는 문서에서 처리 가능한 데이터로 인식 전환시키는 실습이다. 공공부문에는 회의록, 통계보고서, 민원 접수 문서, 사업 안내서처럼 PDF로 쌓이는 자료가 많다. 문제는 이 자료가 검색과 집계에 바로 쓰이기 어렵다는 점이다. Python 자동화는 이 PDF들에서 날짜, 제목, 담당부서, 주요 항목, 표 데이터를 뽑아 엑셀로 옮기는 일을 맡는다. 한 번 코드가 완성되면 다음 달에도 거의 같은 흐름으로 재사용할 수 있다는 점이 중요하다.

이 실습이 실무에서 중요한 이유는 보고 체계의 출발점을 바꾸기 때문이다. 많은 조직은 PDF를 최종 산출물로만 보고, 그 안의 표와 문장을 다시 사람이 옮겨 적는다. 그러나 PDF 추출 자동화가 자리 잡으면 보고를 위해 새로 입력하는 시간이 줄고, 원문과 요약본의 연결이 더 명확해진다. 나중에 오류가 나더라도 어느 문서의 어느 페이지에서 어떤 값이 왔는지 다시 추적하기 쉬워진다. 즉 이 실습은 단순한 편의 기능이 아니라 보고 품질과 근거 추적성을 함께 높이는 출발점이다.

현장에서는 PDF가 모두 같은 품질이 아니라는 점도 알아야 한다. 텍스트 레이어가 살아 있는 PDF는 추출이 비교적 쉽지만, 스캔본은 OCR이 필요하거나 구조가 불안정할 수 있다. 따라서 실습에서는 항상 같은 양식의 문서 몇 개를 먼저 모아 패턴을 확인하는 것이 좋다. 문서 품질이 제각각이면 범용 코드를 만들기보다 양식별로 짧은 스크립트를 여러 개 두는 편이 오히려 유지보수에 유리할 수 있다.

실습 2. 폴더 안의 CSV를 통합 보고서로 만들기

많은 부서에서 정기 보고는 사실상 파일 통합 노동이다. 월별 CSV 12개를 합치고, 부서별 파일을 묶고, 누락 데이터를 확인하고, 요약 시트와 차트를 만든다. 이것은 고급 AI 문제가 아니라 단순 반복의 문제이므로 자동화 효과가 가장 빨리 체감된다. 이 실습에서 학습자는 파일명 규칙, 컬럼 구조, 요약 규칙만 명확히 설명하면 LLM이 상당 수준의 통합 코드를 제안한다는 사실을 경험하게 된다.

이 실습의 핵심은 데이터 품질 검사를 자동화 흐름 안에 함께 넣는 것이다. 파일 통합만으로는 충분하지 않다. 빈 파일이 들어왔는지, 필수 컬럼이 빠졌는지, 날짜 형식이 깨졌는지, 중복 행이 있는지, 특정 부서 파일이 누락되었는지까지 확인해야 실제 보고 업무가 줄어든다. 따라서 좋은 통합 스크립트는 데이터를 모으는 기능과 오류를 알려 주는 기능을 같이 가져야 한다. 교육 현장에서는 이 부분을 자주 놓치지만, 실무에서는 오히려 오류 알림이 더 큰 가치를 가질 때가 많다.

또한 CSV 통합은 이후 웹앱이나 대시보드의 데이터 공급원으로도 바로 연결된다. Day 1에서 정리한 데이터가 Day 2의 조회 화면이나 승인 화면으로 이어지면, 학습자는 자동화와 서비스가 끊어진 별도 주제가 아니라 하나의 흐름이라는 점을 체감하게 된다. 따라서 이 실습은 독립 과제가 아니라 이후 종합 워크숍을 위한 전처리 단계라고 보는 편이 맞다.

실습 3. 워드와 HWPX 양식 생성

공공부문 문서 자동화의 진짜 가치는 보고서가 아니라 양식 생성에서 크게 드러난다. 참석자별 안내문, 부서별 통지문, 인증서, 회의록, 결과보고서처럼 내용은 비슷하되 수신자와 일부 항목만 달라지는 문서는 사람이 손으로 처리할 이유가 거의 없다. 워드는 python-docx 계열 도구로, HWPX는 전용 라이브러리나 템플릿 기반 패키지로 자동화할 수 있다. 실습의 핵심은 템플릿 안에 자리표시자를 분명히 두고, 입력 표의 컬럼명과 정확히 맞추는 것이다.

양식 자동화는 단순히 시간을 줄이는 것을 넘어 문서 품질을 균일하게 만든다는 장점도 있다. 부서원이 많을수록 사람마다 제목 규칙, 날짜 표기, 문단 간격, 서명 위치가 달라지기 쉽다. 템플릿과 자동화를 결합하면 이 편차를 줄일 수 있고, 이후 감사나 보고 시에도 문서 품질이 안정된다. 따라서 문서 자동화는 개인 편의 기능이 아니라 조직 표준화 수단으로 이해하는 편이 맞다.

특히 HWPX 자동화는 공공조직의 실제 문서 문화와 직접 맞닿아 있다는 점에서 의미가 크다. 많은 기관은 여전히 한글 양식을 기준으로 보고 체계와 결재 문화를 운영한다. 이 환경에서 워드만 자동화해도 일부 효과는 있지만, HWPX까지 연결되면 현장 적용성이 훨씬 올라간다. 따라서 C06에서 HWPX를 별도 도구가 아닌 Python 자동화의 확장 사례로 다루는 것은 적절하다. 핵심은 “새 도구를 배우는 것”이 아니라 “조직이 실제 쓰는 양식 체계까지 자동화 범위를 넓히는 것”이다.

현장 적용을 생각한다면 템플릿 관리 정책도 같이 세워야 한다. 어떤 템플릿이 공식 최신본인지, 자리표시자는 어떤 규칙으로 쓸지, 결과 파일명은 어떤 체계로 저장할지, 출력본 검수는 누가 할지 정해 두어야 한다. 자동화는 생성 속도를 높여 주지만, 템플릿 관리가 느슨하면 오히려 서로 다른 양식이 더 빨리 증식하는 부작용이 생길 수 있다.

오류 대응과 복구 흐름

Python 자동화는 잘만 만들면 강력하지만, 파일 경로 하나만 잘못되어도 실패할 수 있다. 그래서 실습에서는 반드시 복구 가능한 절차를 같이 가져가야 한다. 첫째, 원본 파일을 건드리지 않고 별도 출력 폴더를 사용한다. 둘째, 테스트 결과를 로그 파일이나 화면 출력으로 남긴다. 셋째, 실패한 행이나 파일만 다시 돌릴 수 있게 함수 단위로 나눈다. 넷째, 최종 성공 여부를 사람이 눈으로 검수하는 마지막 단계를 남긴다. 이 네 단계가 있으면 자동화는 위험한 블랙박스가 아니라 관리 가능한 도구가 된다.

조직 차원에서 보면 이 네 단계는 곧 감사 대응 구조이기도 하다. 누가 언제 어떤 입력 파일을 사용했고, 어떤 버전의 스크립트를 돌렸으며, 어떤 출력이 나왔는지가 남아 있어야 문제 발생 시 추적이 가능하다. 따라서 교육이 끝난 뒤 실제 적용을 검토할 때는 단순히 스크립트 파일만 남길 것이 아니라 입력 예시, 실행 로그, 출력 예시를 묶어 보관하는 방식을 추천한다.

설치와 운영에서 자주 놓치는 지점

  • Windows 경로에 공백이 많으면 실행 명령을 따옴표로 감싸야 한다.
  • 보안정책 때문에 패키지 설치가 막힐 수 있으므로 사전 허용 절차를 확인해야 한다.
  • 대량 파일 처리 전에는 반드시 샘플 폴더로 먼저 검증해야 한다.
  • 생성된 결과물은 무작위 3건 이상 열어 눈으로 확인해야 한다.

Section 06. M3 단일 HTML 웹앱

단일 HTML 파일은 이 과정에서 가장 작고도 강력한 구현 단위다. HTML이 화면 구조를, CSS가 디자인을, JavaScript가 동작을 담당하므로 한 파일 안에서 모든 것이 완결된다. 서버가 없고 데이터베이스가 없으며 설치가 필요 없다. 더블클릭만 하면 브라우저에서 열리기 때문에 비개발 실무자가 가장 빠르게 성취감을 느끼는 방식이기도 하다. 동시에 단일 HTML은 조직 내부에서 “파일 하나만 전달하면 똑같이 쓸 수 있다”는 장점이 있어 교육용과 배포용 모두에 적합하다.

단일 HTML은 특히 보여주는 페이지와 계산하는 도구 두 영역에서 효율이 높다. 부서 안내 페이지, 행사 소개 페이지, 자료 목록 페이지처럼 콘텐츠를 구조화해 보여주는 용도에 적합하며, 출장비 계산기, 점검 체크리스트, 신청 자격 자가진단처럼 상호작용 중심의 작은 업무 도구에도 잘 맞는다. 사용자가 입력한 값을 즉시 계산하고, 검증 규칙에 따라 경고를 띄우며, 최종 결과를 인쇄 가능한 형태로 보여 주는 정도는 브라우저 안에서 충분히 처리할 수 있다.

다만 완전 폐쇄망에서는 CDN에 의존하는 방식이 항상 통하지는 않는다. 슬라이드가 지적하듯 인터넷이 있어야 Tailwind나 Chart.js 같은 라이브러리를 CDN으로 불러올 수 있기 때문이다. 이런 경우에는 라이브러리 파일을 미리 받아 같은 폴더에 두거나, 아주 작은 도구라면 인라인 CSS와 순수 JavaScript만으로 구성하는 편이 안정적이다. 따라서 단일 HTML 설계의 핵심은 “얼마나 화려한가”가 아니라 “배포받는 사람이 별도 설치 없이 확실히 열 수 있는가”다.

단일 HTML이 적합한 문제

  • 내부 안내 페이지, 행사 페이지, 정책 안내 페이지
  • 계산기, 체크리스트, 입력 검증 폼
  • CSV 읽기, 차트 표시, 인쇄용 요약 화면
  • 교육용 샘플, 데모, 아이디어 검증 프로토타입

단일 HTML 설계 프롬프트의 기본 요소

좋은 프롬프트는 화면 구조와 동작 규칙을 함께 적는다. 예를 들어 “출장비 계산기”라고만 쓰는 대신, 입력 항목, 계산 규칙, 출력 형식, 경고 조건, 인쇄 여부를 같이 적어야 한다. 여기에 디자인 톤과 접근성 조건을 덧붙이면 재작업이 크게 줄어든다. 단일 HTML은 작기 때문에 오히려 요구사항의 정밀도가 결과 품질에 더 직접적으로 반영된다.

단일 HTML을 실무에 적용할 때는 한 파일의 구조를 이해하고 설명할 수 있어야 한다. head에는 메타 정보와 스타일, body에는 사용자 화면, script에는 동작이 담긴다는 기본 틀만 알아도 디버깅이 훨씬 쉬워진다. 예를 들어 화면은 보이는데 버튼이 안 눌리는 경우, 보통 JavaScript 로직이나 이벤트 연결을 먼저 의심하면 된다. 반대로 디자인이 깨지는 경우는 CSS 우선순위나 외부 라이브러리 불러오기 문제일 가능성이 높다. 즉 단일 HTML은 코드량은 작아도, 구조가 단순하기 때문에 오히려 학습용으로 매우 좋은 구현 단위다.

또 하나 중요한 점은 단일 HTML이 “완성된 서비스”가 아니라 “빠른 문제 검증 도구”로서 매우 강하다는 사실이다. 부서에서 실제로 필요한 계산 항목이 무엇인지, 사용자 입력 흐름이 자연스러운지, 출력 양식이 보고에 적합한지 같은 질문은 단일 HTML로 빠르게 검증할 수 있다. 이후 여러 사용자가 함께 써야 하거나 데이터 공유가 필요해지는 시점에만 클라우드 구조로 넘어가면 된다. 이 순서가 비용과 위험을 모두 줄인다.

단일 HTML 품질 체크리스트

  • 인터넷이 끊겨도 핵심 기능이 동작하는가
  • 입력 오류가 있을 때 사용자에게 이해 가능한 경고를 주는가
  • 인쇄 또는 PDF 저장 시 레이아웃이 무너지지 않는가
  • 다음 담당자가 파일을 열어 구조를 읽을 수 있는가

이 네 가지를 통과하면 단일 HTML은 교육용 예제를 넘어 부서 실무 도구로서도 충분한 가치를 가진다.

단일 HTML 개발 워크플로우를 더 구체적으로 이해하기

단일 HTML이 쉬워 보이는 이유는 파일이 하나이기 때문이다. 하지만 실제로는 설계 단계가 매우 중요하다. 첫 번째 단계는 사용자 시나리오를 한 줄로 적는 것이다. 예를 들어 “담당자가 출장비 조건을 입력하면 규정에 맞는 합계를 즉시 확인한다”처럼 적는다. 두 번째 단계는 입력 항목을 테이블처럼 적는 것이다. 날짜, 직급, 이동수단, 숙박 여부, 증빙 여부처럼 필드를 먼저 정리한다. 세 번째 단계는 계산 규칙과 오류 메시지를 적는다. 네 번째 단계는 화면 초안을 그린다. 다섯 번째 단계에서야 HTML과 JavaScript를 만든다.

이 순서를 밟으면 HTML 코드는 구현 결과일 뿐이며, 서비스 품질은 그 이전의 규칙 정리에 달려 있다는 점이 분명해진다. 반대로 규칙 정리 없이 곧바로 코드를 생성하면, 계산기는 만들어지지만 실제 규정과 맞지 않거나 오류 메시지가 부정확한 도구가 나오기 쉽다. 따라서 단일 HTML의 강점은 단순함이 아니라, 작은 문제를 구조화하는 훈련에 있다.

단일 HTML이 적합한 대표 워크플로우

첫 번째는 “입력 즉시 계산형”이다. 출장비 계산기, 교육 이수 기준 계산기, 점검 체크리스트가 여기에 속한다. 두 번째는 “정리 후 출력형”이다. 회의 개요 작성기, 보고서 초안 조립기, 공문용 문장 생성기가 여기에 속한다. 세 번째는 “읽고 찾기형”이다. 부서 FAQ, 자주 쓰는 링크 모음, 행사 안내 페이지가 여기에 속한다. 이 셋은 모두 서버가 없어도 충분히 가치가 있다.

특히 공공부문에서는 “공식 시스템”과 “보조 도구”를 혼동하지 않는 것이 중요하다. 공식 시스템을 만들기 전이라도, 보조 도구로 문제를 먼저 검증하면 나중에 더 큰 시스템을 요구할 때도 훨씬 구체적인 근거를 제시할 수 있다. 단일 HTML은 바로 그 사전 검증 단계에 가장 잘 맞는다.

실무 팁

  1. 첫 버전은 기능 1개만 넣는다.
  2. 저장 기능이 필요하다면 M4의 브라우저 저장소를 붙인다.
  3. 외부 공유가 필요하면 M11에서 GitHub와 정적 배포로 이어 간다.

공공부문에서 단일 HTML이 유난히 강한 이유

공공조직은 설치와 권한이 복잡한 경우가 많다. 이 환경에서는 실행 파일 배포나 별도 프로그램 설치보다 브라우저로 여는 파일 하나가 훨씬 큰 힘을 가진다. 그래서 단일 HTML은 기술적으로 단순해 보여도 조직 관점에서는 도입 저항이 매우 낮은 도구다. 특히 행사 운영, 현장 점검, 내부 계산, 임시 접수 양식처럼 짧은 수명의 문제에는 오히려 가장 적합한 해법일 수 있다.

Section 07. M4 브라우저 영속성

단일 HTML 도구의 가장 큰 한계는 새로고침하면 입력값이 사라진다는 점이다. M4는 바로 이 문제를 해결한다. 브라우저는 자체 저장 공간을 갖고 있으며, 이를 이용하면 서버 없이도 사용자의 데이터를 일정 기간 기억할 수 있다. localStorage는 키와 값 형태의 간단한 저장에 적합하고, IndexedDB는 조금 더 구조화된 데이터와 검색, 필터링, 다건 저장에 적합하다. 이 두 가지를 이해하면 단일 HTML은 단순한 데모를 넘어 실제 개인 업무 도구로 격상된다.

다만 브라우저 저장소는 공식 기록 시스템이 아니다. 브라우저별로 저장 위치가 다르고, 사용자가 캐시나 사용 기록을 지우면 사라질 수 있으며, 같은 파일이라도 file 경로가 달라지면 저장소 동작이 달라질 수 있다. 따라서 M4의 교훈은 “브라우저가 영구 보관을 해 준다”가 아니라 “개인용 업무 도구 수준에서는 충분히 쓸 만한 임시 저장 구조를 확보할 수 있다”는 데 있다. 이 점을 정확히 이해해야 과신과 오용을 막을 수 있다.

표 5. localStorage와 IndexedDB 비교

항목 localStorage IndexedDB 실무 판단
구조 키-값 형태 구조화된 객체 저장 단순 메모, 설정값은 localStorage가 유리
난이도 매우 낮다 상대적으로 높다 검색, 다건 저장, 대용량이면 IndexedDB 고려
용량 작음 더 큼 큰 CSV나 누적 기록은 IndexedDB가 적합
대표 시나리오 할일 목록, 폼 임시저장, 설정 저장 북마크 라이브러리, 이슈 추적, 대용량 오프라인 데이터 사용자 수와 데이터 구조를 기준으로 선택

실습 1. 개인 기록 도구와 내보내기·불러오기

브라우저 저장소를 제대로 쓰려면 반드시 내보내기와 불러오기 기능을 붙여야 한다. 이것이 있어야 브라우저 캐시 삭제나 PC 교체 상황에 대비할 수 있다. JSON 내보내기는 데이터 구조를 그대로 유지하는 데 좋고, CSV 내보내기는 보고와 공유에 좋다. 즉 M4의 핵심은 단순 저장이 아니라 사용자 스스로 데이터를 통제하게 만드는 것이다.

이 실습은 단순히 파일을 저장하는 기능 이상의 의미가 있다. 개인이 만든 도구가 조직 자산이 되려면 데이터 이동성이 확보되어야 한다. 브라우저 안에만 데이터가 갇혀 있으면 본인 PC를 떠나는 순간 가치가 크게 줄어든다. 반대로 JSON과 CSV 내보내기·불러오기가 있으면, 같은 도구를 다른 PC로 옮기거나 연도별 아카이브를 남기거나 후임자에게 전달하는 일이 쉬워진다. 즉 이 기능은 브라우저 기반 개인 도구를 반영구적 자산으로 바꾸는 최소한의 장치다.

실습 2. 즐겨찾기와 부서 맞춤 자료 라이브러리

이 실습은 개인용 업무 지도 만들기에 가깝다. 자주 쓰는 사이트, 양식, 파일경로, 설명, 태그를 한곳에 저장하고 분류와 검색을 지원하면 작은 부서 인프라가 생긴다. 부서 신입에게 가장 먼저 건네줄 수 있는 도구이기도 하다. 규모는 작지만 업무 적응 속도를 크게 줄여 준다.

여기서 중요한 것은 단순 링크 모음이 아니라 맥락을 함께 저장하는 것이다. 사이트명과 URL만 있으면 브라우저 즐겨찾기와 큰 차이가 없다. 그러나 언제 쓰는지, 어떤 상황에서 열어야 하는지, 관련 양식이나 주의사항이 무엇인지까지 메모로 남기면 비로소 지식 자산이 된다. 공공부문에서는 이 맥락 메모가 특히 중요하다. 같은 시스템도 부서마다 쓰는 이유와 절차가 다르기 때문이다.

실습 3. 폼 입력 임시저장

자동 저장은 더 이상 사치가 아니라 기본 기능이다. 사용자가 긴 신청서를 쓰다가 중단해도 다시 열면 이어서 쓸 수 있어야 한다. 특히 공공부문 양식은 한 번에 작성하기 어려운 경우가 많기 때문에, 임시저장과 최종 제출을 분리하는 패턴은 실제 서비스 설계에서도 그대로 이어진다.

이 패턴은 나중에 클라우드 서비스로 넘어가도 그대로 유효하다. 사용자가 긴 신청서를 쓰는 동안 중간 저장이 필요하다는 사실은 localStorage 단계에서 먼저 검증할 수 있다. 이후 데이터베이스 기반 서비스로 전환하면 저장 위치만 바뀔 뿐 사용성 요구 자체는 동일하다. 따라서 M4는 단순한 브라우저 기능 수업이 아니라, 실제 업무 서비스가 가져야 할 사용성 기준을 배우는 단계라고 볼 수 있다.

localStorage를 쓸 때의 운영 수칙

  • 공식 기록이나 법정 보관 데이터는 localStorage에 두지 않는다.
  • 중요한 데이터는 JSON 내보내기로 정기 백업한다.
  • 브라우저, PC, 파일 경로가 바뀌면 동작이 달라질 수 있음을 사용자에게 안내한다.
  • 공유 서비스가 필요해지는 순간 클라우드 데이터베이스로 전환해야 한다.

localStorage가 생각보다 까다로운 이유

MDN 문서는 localStorage가 현재 문서의 origin에 묶여 있다고 설명한다. 이 설명은 웹서비스를 배포할 때는 자연스럽지만, 단일 HTML 파일을 로컬에서 직접 여는 공공 실무 환경에서는 생각보다 큰 차이를 만든다. 특히 file: URL로 연 문서의 localStorage 동작은 표준적으로 보장되지 않으며 브라우저별로 달라질 수 있다고 MDN은 분명히 적고 있다. 즉 지금 당장은 잘 저장되는 것처럼 보여도, 브라우저 정책이나 파일 경로가 바뀌면 동작이 달라질 수 있다.

이 경고는 교재에서 반드시 강조해야 한다. 왜냐하면 단일 HTML 실습을 한 뒤 실제 업무에 쓰기 시작하는 순간, 사용자는 브라우저 저장소를 “자동으로 영구 보관되는 개인 DB”처럼 오해하기 쉽기 때문이다. 하지만 브라우저가 쿠키나 저장을 제한하는 정책을 쓰거나, 시크릿 모드로 열리거나, 파일 경로가 달라지면 저장이 예상과 다르게 동작할 수 있다. 따라서 localStorage는 편리하지만, 공식 기록이나 장기 보관의 기반이 아니라는 점을 반복해서 설명해야 한다.

IndexedDB를 언제 선택해야 하는가

MDN은 IndexedDB를 많은 양의 구조화된 데이터를 다루기 위한 저수준 클라이언트 저장소로 설명한다. localStorage가 문자열 키-값 저장에 가깝다면, IndexedDB는 객체 저장, 인덱스 검색, 트랜잭션, 다건 처리에 더 적합하다. 실무적으로는 저장 항목이 20개를 넘기고, 검색이나 정렬이 필요하며, 파일 메타데이터나 여러 레코드를 다뤄야 하는 순간부터 IndexedDB를 검토하는 편이 맞다. 반대로 메모장, 설정값, 임시 폼 저장처럼 작고 단순한 구조는 localStorage가 훨씬 빠르다.

중요한 점은 IndexedDB가 더 강력하다고 해서 무조건 더 좋은 것이 아니라는 사실이다. 코드 복잡도가 높아지고, 트랜잭션과 비동기 흐름을 이해해야 하므로 학습 난도가 올라간다. 따라서 C06의 교재에서는 “작으면 localStorage, 구조화가 필요하면 IndexedDB, 공유가 필요하면 서버 DB”라는 계단 구조를 보여 주는 것이 적절하다.

브라우저 저장소 선택 워크플로우

  1. 데이터가 개인용인가, 공유용인가를 먼저 묻는다.
    공유용이면 브라우저 저장소는 장기 해법이 아니다.
  2. 저장 구조가 단순한가, 검색과 목록이 필요한가를 묻는다.
    단순하면 localStorage, 복잡하면 IndexedDB를 검토한다.
  3. 브라우저 변경과 경로 변경을 감수할 수 있는가를 본다.
    안 된다면 내보내기·불러오기 기능이 필수다.
  4. 공식 기록 성격이 있는가를 묻는다.
    있다면 브라우저 저장소를 최종 저장소로 삼지 않는다.

이 네 질문만 반복해도, 개인 도구와 조직 시스템의 경계를 훨씬 덜 헷갈리게 된다.

개인 도구에서 공유 서비스로 옮겨 가는 실제 장면

브라우저 저장소는 종종 훌륭한 첫 단계가 된다. 예를 들어 부서 링크 라이브러리나 교육자료 메모장은 처음에는 개인용 localStorage로 충분하다. 하지만 동료 둘이 같은 자료를 보고 싶어지는 순간 문제가 바뀐다. 누가 최근에 무엇을 추가했는지, 누가 틀린 링크를 수정했는지, 삭제를 누가 했는지를 남겨야 하기 때문이다. 바로 이 시점이 브라우저 저장소를 졸업할 때다. 즉 M4는 저장 기술을 배우는 장이기도 하지만, “공유가 시작되면 서비스 설계가 달라진다”는 사실을 배우는 장이기도 하다.

개인 도구에서 공유 도구로 넘어가는 판단 기준

다음 조건 가운데 둘 이상이 맞으면 브라우저 저장소만으로는 부족하다고 판단하는 편이 좋다. 두 명 이상이 같은 데이터를 동시에 봐야 하는가. 작성 이력을 남겨야 하는가. 관리자 승인이나 상태 변경이 필요한가. 공식 기록에 준하는 보관이 필요한가. 이 네 가지는 공유형 서비스로 넘어갈 시점을 알려 주는 신호다. 즉 M4는 localStorage를 오래 쓰기 위한 장이 아니라, localStorage의 유효 범위를 정확히 아는 장이다.

Section 08. M5 엑셀 VBA와 LLM

VBA는 공공부문 실무에서 과소평가된 자동화 자산이다. Python이 강력하다는 사실과 별개로, 많은 조직에서는 별도 설치가 필요 없고 Office만 있으면 바로 쓸 수 있는 VBA가 훨씬 현실적이다. 엑셀, 워드, 아웃룩, 파일 시스템을 한 번에 다룰 수 있고, 기존에 쓰던 양식과 직접 결합되며, 보안검토도 비교적 단순하다. 문제는 진입장벽이 문법에 있었다는 점인데, LLM이 이 장벽을 크게 낮췄다.

이 모듈에서 가장 중요한 운영 철학은 “구조를 설명하고, 데이터는 내부에서 처리한다”는 원칙이다. 외부 LLM에는 시트명, 컬럼명, 샘플 헤더, 파일명 규칙 정도만 알려 주고 매크로 코드를 받는다. 실제 데이터를 읽고 통합하고 분리하는 작업은 내부 PC의 엑셀에서 실행한다. 이 구조는 Python과 동일하지만, 사용자 입장에서는 익숙한 Excel 창 안에서 바로 실행된다는 점 때문에 심리적 허들이 더 낮다.

실습 1. 보고서 자동 생성 매크로

이 실습의 목표는 사람이 매번 같은 수작업을 반복하는 구조를 끝내는 것이다. 보고서의 모양은 대부분 정해져 있고, 달라지는 것은 숫자와 입력 데이터뿐이다. 따라서 열 구조와 출력 위치만 명확하면 매크로는 매우 빠르게 효과를 낸다. 특히 팀별 파일을 본부 보고용 한 페이지로 합치거나, 경고 기준에 따라 색상을 바꾸는 작업에서 체감 효과가 크다.

실습 2. 다수 엑셀 파일 통합과 분리

많은 부서 업무는 사실상 제출 파일 관리다. 여러 팀에서 같은 양식으로 엑셀을 올리고, 본부가 이를 합쳐 보고서를 만든다. 이 과정은 늘 누락, 파일명 오류, 양식 미준수, 손작업 복사가 뒤섞인다. VBA 통합 매크로는 이 노동을 크게 줄여 준다. 반대로 본부 파일 하나를 팀별로 분리해 다시 내려보내는 작업도 자동화할 수 있다.

VBA를 쓸 때 놓치지 말아야 할 점

  • 매크로는 테스트용 복사본에서 먼저 검증한다.
  • 사용자에게 실행 버튼과 사용법을 명확히 제공한다.
  • 오류 메시지를 기록하고, 한 번에 너무 많은 기능을 넣지 않는다.
  • .xlsm 저장과 매크로 보안설정을 문서에 함께 적어 둔다.

VBA 객체 모델을 이해하면 두려움이 줄어든다

Microsoft Learn 문서는 VBA를 설명할 때 가장 먼저 객체 모델을 소개한다. Application, Document, Worksheet, Range 같은 객체가 계층적으로 연결되고, 이 객체에 메서드와 속성을 적용한다는 설명이다. 이 관점은 실무자에게 매우 중요하다. 왜냐하면 VBA를 신비한 매크로 언어로 보지 않고, “엑셀의 셀과 시트에 명령을 보내는 방법”으로 이해하게 해 주기 때문이다. 예를 들어 Worksheets(1).Range("A1").Value = "Hello" 같은 코드는 문법을 외우는 문제가 아니라, “첫 번째 시트의 A1 셀 값에 Hello를 넣는다”는 평범한 한국어 명령의 다른 표현일 뿐이다.

이 관점이 잡히면 VBA는 갑자기 쉬워진다. 사용자는 엑셀 화면에서 보이는 요소와 코드 속 요소를 대응시킬 수 있고, 어디를 바꾸면 어떤 결과가 나는지 추적할 수 있다. 반대로 이 개념이 없으면 매크로 기록기가 생성한 긴 코드를 읽을 때 금세 길을 잃는다. 따라서 C06 교재에서는 VBA 문법을 세세히 나열하기보다, 객체, 속성, 메서드, 반복문, 조건문이 실제 업무 자동화와 어떻게 연결되는지 설명하는 것이 훨씬 중요하다.

매크로 기록기에서 실무 자동화로 가는 4단계

Microsoft Learn이 강조하듯 Record Macro는 VBA 입문에서 매우 강한 다리다. 실제 업무 자동화로 옮길 때는 다음 네 단계를 권한다.

  1. 손으로 한 번 실행해 본다.
    예: 특정 열의 빈 행 삭제, 특정 서식 적용.
  2. 매크로 기록으로 코드를 얻는다.
  3. 불필요한 선택 동작을 줄이고 변수와 반복문으로 일반화한다.
  4. 버튼이나 메뉴로 묶고 사용법을 문서화한다.

이 네 단계를 거치면 “한 번 기록된 개인용 매크로”가 아니라 “다음 사람도 쓸 수 있는 부서 도구”로 바뀐다. 특히 기록된 코드는 모든 클릭을 다 담기 때문에 장황한 경우가 많다. 여기서 선택 동작을 줄이고 구조를 단순화하는 경험이 바로 VBA 학습의 핵심이다.

VBA 보안과 배포를 현실적으로 이해하기

Microsoft Learn은 매크로가 일반 문서 확장자에 그대로 저장되지 않으며, Word는 .docm, Excel은 .xlsm 같은 매크로 사용 확장자가 필요하다고 설명한다. 또한 Trust Center와 Trusted Location, Trusted Publisher 같은 보안 설정을 따로 봐야 한다고 안내한다. 이 설명은 공공조직에서는 특히 중요하다. 왜냐하면 많은 현장에서 “왜 파일이 열리는데 매크로는 안 돌아가느냐”가 첫 번째 장애이기 때문이다.

실무적으로는 세 가지를 반드시 문서화해야 한다. 첫째, 어떤 확장자로 저장해야 하는가. 둘째, 사용자가 처음 열 때 어떤 경고를 보게 되는가. 셋째, 신뢰할 수 있는 위치나 서명 정책이 필요한가. 이 세 가지를 적지 않으면 매크로 자체보다 보안 경고 때문에 도입이 좌초되기 쉽다. 따라서 VBA는 코드 작성만이 아니라, 사용자 실행 흐름까지 설계해야 하는 기술이라고 보는 편이 정확하다.

VBA 디버깅을 두려워하지 않는 법

Microsoft Learn은 문법 오류와 런타임 오류를 구분하고, 디버그 모드에서 실패 줄을 확인하며 고치라고 설명한다. 이 원칙은 C06에서도 그대로 중요하다. 실무자는 오류 메시지를 부끄러운 실패로 보기보다, 수정 단서를 주는 정보로 읽는 습관을 들여야 한다. 특히 LLM과 같이 작업할 때는 오류 메시지를 그대로 붙여 넣고, 어떤 시트에서 어떤 단계에서 실패했는지 함께 설명하는 것이 가장 빠르다.

즉 VBA에서도 핵심은 코드를 완벽히 처음부터 쓰는 능력이 아니라, 작게 시험하고, 오류를 읽고, 다시 수정하는 반복 루프다. 이 점에서 VBA는 오히려 생성형 AI와 궁합이 좋다. 레거시 도구이지만 수정 사이클은 현대적이기 때문이다.

왜 VBA는 여전히 유효한가

2026년에도 VBA가 의미를 잃지 않는 이유는 레거시 호환성 때문이다. 많은 공공부문 업무가 여전히 Excel과 Outlook 중심으로 돌아가고, 조직 내 표준 양식과 승인 흐름도 이에 맞춰져 있다. 이런 환경에서는 가장 최신 언어보다 가장 적게 거슬리는 자동화 수단이 더 강하다. VBA는 바로 그 지점을 차지한다. LLM이 문법 장벽을 낮춘 지금, 실무자는 매크로를 완전히 새로 배우기보다 "내가 하던 손작업을 어떤 단위로 나누어 설명할 것인가"에 더 집중하면 된다.

VBA와 외부 LLM의 역할 분담

실무적으로는 외부 LLM이 코드를 설계하고, 내부 Excel이 데이터를 처리하는 구조가 가장 안정적이다. 이때 중요한 것은 실제 값이 아니라 구조를 설명하는 것이다. 예를 들어 "A열은 부서명, B열은 접수일, C열은 상태"처럼 구조만 설명해도 상당수 매크로는 충분히 만들 수 있다. 이 원칙을 지키면 공공조직도 외부 지능과 내부 데이터를 비교적 안전하게 분리할 수 있다.

VBA 적용이 특히 유리한 현장

  • 이미 부서별 Excel 양식이 표준화되어 있는 경우
  • Outlook 메일 정리와 일정 관리가 반복되는 경우
  • 별도 프로그램 설치가 사실상 불가능한 경우
  • 사용자가 코드보다 버튼 실행에 익숙한 경우

이 네 조건 가운데 두세 가지가 겹치면 Python보다 VBA가 먼저 성과를 낼 가능성이 높다.

Section 09. M6 Ollama와 로컬 SLM

Ollama는 로컬 PC에서 오픈 모델을 실행하는 경험을 단순화한 도구다. 2026년 5월 기준 공식 사이트가 가장 앞에 내세우는 메시지도 로컬 실행과 데이터 통제다. 동시에 필요한 경우 더 큰 모델이나 병렬 처리에 cloud를 연결할 수 있다는 방향도 보여 준다. 이 두 메시지를 함께 읽어야 한다. 즉 Ollama의 본질은 모든 일을 클라우드로 보내는 것이 아니라, 가능한 한 로컬에서 시작하고 필요할 때만 확장하는 구조다.

공공부문에서 Ollama가 특히 의미 있는 이유는 외부 API 호출이 막혀 있는 환경에서도 LLM 활용 실험을 이어 갈 수 있게 해 주기 때문이다. 물론 로컬 SLM은 GPT급 대형 모델보다 성능이 낮고 느릴 수 있다. 그러나 간단한 분류, 짧은 문서 요약, 말투 정리, 내부 FAQ 초안, 규칙형 응답 생성 수준에서는 충분히 쓸 만하다. 오히려 작은 모델을 써 보면 프롬프트의 중요성을 더 직접적으로 체감하게 된다. 모호한 지시에는 모호한 결과가 나오고, 구체적 기준을 주면 작더라도 꽤 안정된 결과가 나온다.

M6가 중요한 이유는 단순히 모델을 내려받는 법을 배우기 때문이 아니다. 시스템 프롬프트, temperature, context 길이, 최대 출력 길이 같은 설정이 실제 품질을 어떻게 바꾸는지 직접 보기 때문이다. 특히 공공부문 응답은 톤과 형식이 일정해야 하므로, “3줄 이내로 답하라”, “존댓말을 쓰라”, “근거가 없으면 확인 후 안내드리겠다고 말하라” 같은 규칙을 시스템 프롬프트로 고정하는 방식이 유용하다.

2026년 현재 Ollama 공식 문서는 Windows, macOS, Linux를 모두 지원하는 빠른 시작 흐름을 제시하고, 로컬 API와 coding integration을 함께 강조한다. 특히 ollama 명령으로 대화형 메뉴를 열고 모델 실행이나 도구 실행으로 들어갈 수 있고, ollama launch claude, ollama launch codex 같은 통합도 제공한다는 점은 교육적으로 흥미롭다. 이것은 로컬 모델 실험이 단순 채팅을 넘어 코딩 보조 워크플로와도 연결될 수 있음을 보여 준다. 즉 Ollama는 “로컬 챗봇”이 아니라 “로컬 AI 실행 계층”으로 보는 편이 더 정확하다.

또한 공식 API 문서는 로컬 기본 주소가 http://localhost:11434/api이고, 동일한 형태의 API를 cloud에서도 쓸 수 있다고 설명한다. 이 사실은 교육 설계에 중요한 함의를 준다. 처음에는 로컬 API로 시작해 HTML, Python, VBA에서 동일한 호출 패턴을 익히고, 나중에 필요하면 더 큰 모델로 옮겨 가는 점진적 설계가 가능하다는 뜻이다. 즉 인터페이스를 먼저 익히고, 모델 위치는 나중에 바꾸는 전략을 취할 수 있다.

Ollama를 기존 도구에 붙이는 세 가지 방식

첫 번째 방식은 명령 프롬프트에서 직접 대화하며 모델 성격을 파악하는 것이다. 이것은 모델 선택과 시스템 프롬프트 실험에 유리하다. 두 번째 방식은 Python이나 단일 HTML에서 로컬 API를 호출하는 것이다. 같은 모델이라도 입력 형식을 통일하면 간단한 문서 요약 도구나 분류 도구를 빠르게 만들 수 있다. 세 번째 방식은 VBA나 Word 매크로에서 HTTP 호출을 통해 응답을 받아 오는 것이다. 이 방식은 레거시 환경과의 결합성이 높아, 폐쇄망 부서에서 로컬 SLM을 실제 업무 흐름에 끼워 넣는 데 특히 유용하다.

이 세 방식은 경쟁 관계가 아니라 단계 관계다. 처음에는 콘솔에서 모델을 시험해 보고, 다음에는 HTML이나 Python으로 작은 도구를 만든 뒤, 마지막에 Excel이나 Word 업무 흐름에 붙이는 것이 안전하다. 이런 순서를 따르면 모델 품질 문제와 연결 문제를 한꺼번에 섞지 않게 된다.

모델 선택 절차

로컬 모델 선택은 벤치마크 숫자를 외우는 일이 아니라, 내 PC와 내 업무에 맞는 조합을 찾는 일이다. 일반 사무 PC에서는 경량 모델부터 시작해 응답속도와 품질을 확인하는 편이 낫고, 메모리가 충분한 노트북이라면 조금 더 큰 모델을 시도해 볼 수 있다. 또 문장 생성이 필요한지, 분류만 필요한지, 출력 형식이 JSON인지 자연어인지에 따라 적합한 모델도 달라진다. 따라서 실습에서는 한 모델만 고집하기보다 동일한 과업을 둘 이상의 모델에 넣어 보고, 속도와 품질, 반복 가능성을 함께 평가하는 습관을 들여야 한다.

여기서 중요한 것은 정확도뿐 아니라 운영 편의다. 예를 들어 아주 작은 모델이 조금 덜 똑똑하더라도 응답이 빠르고 반복 가능하며 설치 부담이 낮다면, 부서 도구에는 오히려 더 적합할 수 있다. 반대로 한 번의 답변 품질은 좋지만 속도가 지나치게 느리거나 하드웨어 요구가 높은 모델은 일상 업무에서는 외면받기 쉽다. 공공부문에서 도구 채택은 벤치마크보다 반복 사용성이 더 중요한 경우가 많다.

Hugging Face와 GGUF 자산을 읽는 법

원본 슬라이드가 언급하듯 Hugging Face의 GGUF 자산은 Ollama 실험 폭을 넓혀 준다. 다만 공공부문 실무자는 새로운 모델을 시도할 때 라이선스와 파일 크기, 추론 자원, 사용 목적을 함께 확인해야 한다. 즉 모델을 많이 모으는 것이 목표가 아니라, 조직 안에서 반복적으로 돌릴 수 있는 안정적인 후보군을 정하는 것이 목표다.

로컬 SLM이 적합한 대표 시나리오

  • 엑셀 셀 값에 대한 짧은 분류 또는 라벨링
  • Word 문서 문장 다듬기와 맞춤법 보조
  • 내부 규정 문장 요약과 제목 추천
  • 반복 민원 문구의 초안 응답 생성

로컬 SLM 운영에서 중요한 기준

첫째, 작은 모델일수록 입력을 짧고 구조화해야 한다. 둘째, 환각을 줄이려면 선택형 과제나 규칙형 과제로 시작하는 것이 좋다. 셋째, 로컬이라고 해서 무검수로 운영하면 안 된다. 넷째, 로컬과 클라우드를 이분법적으로 보지 말고, 어떤 업무는 로컬이 맞고 어떤 업무는 외부 고성능 모델이 더 적합하다는 식으로 포트폴리오를 짜야 한다.

공공조직에서 권장하는 로컬 LLM 시작 순서

  1. 요약보다 분류부터 시작한다.
  2. 자유문 생성보다 JSON이나 선택형 출력부터 시작한다.
  3. 단독 모델 성능보다 사용자의 검수 부담을 같이 본다.
  4. 동일 과업을 외부 고성능 모델과 비교해 기대치를 조정한다.

이 순서를 따르면 로컬 모델 도입이 과장된 기대보다 통제 가능한 실험으로 남는다.

Section 10. M7 클라우드 노코드 풀스택 비교

Day 2에 들어서면 구현의 범위가 개인용 도구에서 공유 서비스로 넓어진다. 이 시점부터는 단일 HTML이나 브라우저 저장소만으로는 부족하다. 계정이 필요하고, 여러 사용자가 동시에 데이터를 쓰며, 권한별 화면이 달라지고, 배포된 URL로 접속해야 한다. M7은 바로 이 전환 지점에서 각 도구의 성격을 비교하는 모듈이다. 핵심은 어떤 도구가 최고인가를 가리는 것이 아니라, 어떤 도구가 어떤 문제에 더 적합한지 몸으로 느끼는 데 있다.

슬라이드의 제안처럼 같은 요구사항을 두 도구에 동시에 넣어 보는 방식은 매우 효과적이다. 표만 읽어서는 알 수 없는 차이가 실제로는 빠르게 드러난다. 어떤 도구는 화면은 좋지만 데이터 구조가 약하고, 어떤 도구는 풀스택 연결은 강하지만 미세한 디자인 보정이 번거롭다. 또 어떤 도구는 초보자에게 친절하지만 코드 소유권이나 협업 연결에서 제약이 있을 수 있다. 따라서 M7은 “읽고 비교”가 아니라 “만들고 비교”의 모듈이다.

2026년 현재 도구 지형을 읽는 다섯 축

2026년의 노코드 AI 도구 시장은 단순히 “어느 서비스가 더 멋진 화면을 뽑아 주는가”로 비교하기 어렵다. 실제 차이는 다섯 가지 축에서 벌어진다. 첫째는 코드 소유권이다. 생성 결과를 GitHub로 넘겨 장기 유지보수가 가능한지, 아니면 특정 플랫폼 안에 묶이는지가 중요하다. 둘째는 백엔드 자동화 수준이다. 데이터베이스, 인증, 파일 저장, 서버사이드 로직을 한 번에 다루는지 여부에 따라 실제 업무 전환 속도가 크게 달라진다. 셋째는 디자인 보정 가능성이다. 처음 만든 결과를 비개발 실무자가 얼마나 쉽게 수정할 수 있는지가 학습 곡선에 직접 영향을 준다. 넷째는 거버넌스와 보안 기능이다. 조직 사용에서는 2FA, SSO, 역할 분리, 감사 가능한 운영 구조가 생각보다 빨리 필요해진다. 다섯째는 핸드오프 용이성이다. 디자인 팀, 개발팀, 운영 담당자에게 결과를 넘길 때 문서와 코드가 얼마나 잘 정리되는지가 결국 조직 자산화의 관건이 된다.

이 다섯 축을 기준으로 보면 도구 선택은 취향 문제가 아니라 조직 구조 문제다. 개인이 단독으로 빠르게 시제품을 만들고 싶다면 화면 수정이 쉽고 진입 장벽이 낮은 도구가 맞다. 반대로 부서 단위 서비스로 이어 가려면 처음부터 GitHub 연동, 권한 구조, 데이터 모델링을 고려해야 한다. 같은 게시판 요구사항이라도 시연용이면 화면 생성 도구가 충분하지만, 실제 운영용이면 데이터와 권한을 함께 다루는 풀스택 도구가 더 적합하다. 따라서 학습자는 도구 데모에 감탄하기보다, “우리 조직에서 이 결과물을 누가 이어받을 것인가”를 먼저 물어야 한다.

공공부문에서 흔한 도구 선택 오판

실무에서는 종종 가장 눈에 띄는 화면을 만들어 준 도구를 곧바로 최선이라고 판단한다. 그러나 공공부문에서 더 흔한 실패는 예쁜 화면보다 운영 구조 부재에서 나온다. 예를 들어 행사 안내 페이지처럼 하루짜리 수명인 결과물은 정적 파일이면 충분하지만, 사용자 계정과 상태 변경이 들어가는 순간 설계 기준이 완전히 달라진다. 또한 시연 당시에는 한 사람이 모든 맥락을 알고 있어 문제가 없어 보여도, 실제 부서 서비스가 되면 승인권자, 운영 담당자, 보안 담당자, 다음 담당자가 각각 다른 질문을 던진다. 이 질문에 답하지 못하면 도구 선택은 잘못된 것이다.

그래서 M7의 비교 실습은 결과 화면 점수만 매기면 안 된다. 최소한 다음 항목을 같이 기록해야 한다. 첫째, 두 번째 수정이 얼마나 쉬웠는가. 둘째, 데이터 모델이 자연스럽게 보였는가. 셋째, 권한 요구를 적었을 때 결과가 어떻게 달라졌는가. 넷째, GitHub나 외부 배포 경로로 넘기기 쉬운가. 다섯째, 조직 표준 문서로 설명 가능한 구조인가. 이 다섯 항목을 기록하면 “어떤 도구가 더 신기했는가”가 아니라 “어떤 도구가 우리 조직에 덜 위험한가”를 판단할 수 있다.

비교 실습을 통해 얻어야 하는 것

  • 같은 문제를 서로 다른 도구로 풀 때 결과가 어떻게 달라지는지
  • 디자인 품질과 데이터 구조 가운데 무엇이 현재 조직에 더 중요한지
  • 비개발 실무자가 어디까지 직접 수정할 수 있는지
  • 나중에 개발자와 협업할 때 코드와 설정을 얼마나 넘겨주기 쉬운지

이 비교가 끝나야 M8에서 특정 도구를 중심으로 깊이 들어가도 흔들리지 않는다. 즉 M7은 도구 선택의 감각을 만드는 모듈이다.

Section 11. M8 Lovable과 Supabase 자동연동

Lovable은 자연어로 웹앱을 만들고, 반복 수정하며, 코드와 운영 구조를 함께 다루려는 사용자에게 가장 직관적인 경험을 제공하는 도구 중 하나다. 2026년 5월 기준 공식 문서가 강조하는 포인트도 분명하다. 자연어를 통해 실제 코드가 생성되고, 그 코드는 GitHub와 동기화할 수 있으며, 초기 아이디어부터 배포와 운영까지의 전체 생애주기를 지원한다는 점이다. 여기서 중요한 것은 단순히 “쉽게 만들어 준다”가 아니라, 코드 소유권과 워크플로 통합을 전제로 설계되었다는 점이다.

M8이 이 과목에서 핵심인 이유는 비개발 실무자가 처음으로 “풀스택이 한 묶음으로 작동하는 경험”을 하게 만들기 때문이다. 프론트엔드 화면, 데이터 저장, 인증, 파일 업로드, 배포를 각각 따로 배우는 대신, 하나의 서비스가 작동하기 위해 이 요소들이 어떻게 연결되는지를 동시에 본다. 특히 게시판, 결재 신청, 회의실 예약 같은 부서 서비스는 글 저장, 사용자 구분, 권한 분리, 목록 화면, 세부 화면, 수정과 삭제, 공개 URL이 함께 있어야 비로소 서비스가 된다. M8은 이 전체를 한 흐름으로 보여 준다.

Supabase와의 연결은 그 자체로도 교육적 가치가 크다. 공식 Auth 문서는 인증과 권한 부여를 명확히 구분하고, JWT 기반 인증과 RLS를 결합해 행 단위 접근 제어를 수행하는 구조를 강조한다. 공공부문 실무자에게 이 설명은 매우 중요하다. “로그인을 붙였다”와 “권한이 통제된다”는 전혀 다른 문제이기 때문이다. 사용자가 본인 데이터만 보게 하고, 승인자만 결재 상태를 바꿀 수 있게 하고, 관리자만 특정 메뉴를 볼 수 있게 하려면 결국 인증과 권한이 함께 설계되어야 한다.

Lovable 실습의 기본 흐름

  1. 시나리오를 한 문장으로 정리한다.
  2. 첫 화면과 핵심 기능을 자연어로 요청한다.
  3. 데이터 저장이 필요하면 테이블 구조를 함께 설명한다.
  4. 회원 구분이 필요하면 인증과 권한을 요청한다.
  5. 수정 후 GitHub 또는 배포 경로를 연결한다.

Lovable Cloud가 의미하는 것

원본 슬라이드가 강조하듯 Lovable의 강점은 화면 생성만이 아니라 백엔드 구성까지 한 묶음으로 연결된다는 데 있다. 인증, 데이터베이스, 파일 저장, 서버사이드 로직을 각각 따로 붙이는 대신, 필요한 기능을 자연어로 요청하면서 점진적으로 붙일 수 있다. 이 경험은 비개발 실무자에게 매우 중요하다. 서비스는 예쁜 화면 하나로 끝나지 않고, 결국 저장과 권한과 배포가 함께 있어야 유지되기 때문이다.

공공부문에서는 이 점이 더 직접적으로 와 닿는다. 예를 들어 결재 신청 화면만 있는 서비스는 실무에 바로 쓸 수 없다. 신청 내역 저장, 승인자 확인, 상태 변경, 첨부파일 보관, 사용자별 열람 범위가 함께 있어야 한다. Lovable 기반 실습은 이 요소들을 한 번에 보게 하므로, "웹앱을 만든다"는 말이 실제로 무엇을 뜻하는지 구체적으로 체감하게 한다.

2026년 현재 Lovable 공식 문서가 특히 강하게 내세우는 포인트는 세 가지다. 첫째, 자연어로 생산급 웹앱을 만들더라도 결과는 편집 가능한 실제 코드로 남는다는 점이다. 둘째, 공유 워크스페이스를 기반으로 팀 협업과 GitHub 연동을 전제로 삼는다는 점이다. 셋째, 보안과 거버넌스를 부가 기능이 아니라 기본 워크플로의 일부로 둔다는 점이다. 이 설명은 공공부문에 직접 번역할 가치가 있다. 왜냐하면 내부 도구를 만들 때도 결국은 코드 소유권, 협업, 운영 기준이 필요하기 때문이다.

즉 Lovable을 단순한 화면 생성기로 이해하면 절반만 이해한 셈이다. 오히려 중요한 것은 “아이디어에서 운영까지의 흐름을 얼마나 짧게 만들 수 있는가”다. 화면 초안, 데이터 저장, 계정 흐름, GitHub 보관, 공개 URL, 이후 운영 규칙 정리까지가 끊기지 않아야 한다. 이 흐름이 짧을수록 비개발 실무자도 구현의 전 과정을 조망할 수 있고, 나중에 개발팀과 협업할 때도 어디까지 자동으로 해결되고 어디서부터 수작업 검토가 필요한지 분명해진다.

공공 서비스 데이터 모델링의 최소 단위

많은 사용자가 자연어로 웹앱을 만들 때 화면 텍스트와 버튼 배치에만 집중한다. 그러나 실제 서비스는 결국 데이터 구조에서 무너지는 경우가 많다. 예를 들어 부서 게시판이라면 최소한 사용자, 게시글, 댓글, 첨부파일, 공지 여부, 작성일시, 수정일시 같은 엔터티를 생각해야 한다. 결재 신청 서비스라면 신청자, 승인자, 상태, 첨부파일, 결재 단계, 반려 사유, 수정 이력이 필요하다. 이 구조를 미리 생각하고 요청해야, 생성형 도구가 내놓는 결과가 단순 시연용을 넘어 운영 가능한 골격이 된다.

따라서 M8 실습에서는 화면을 만들기 전에 다음 질문을 쓰는 편이 좋다. 어떤 데이터가 새로 생성되는가. 누가 읽는가. 누가 수정하는가. 언제 삭제되는가. 파일 첨부가 필요한가. 알림이 필요한가. 검색과 정렬 기준은 무엇인가. 이 질문에 답하고 나면 프롬프트의 질이 높아지고, 나중에 RLS나 권한 설계로 넘어갈 때도 훨씬 덜 흔들린다.

Lovable이 자동으로 해결하지 않는 것

도구가 백엔드 연결과 코드 생성을 도와준다고 해서 조직 운영 문제가 자동으로 해결되지는 않는다. 개인정보 처리 목적과 보관 기간, 테스트 계정 운영 방식, 관리자 계정 승인 절차, 배포 승인 문서, 장애 시 수동 대체 절차는 여전히 사람이 정해야 한다. 공공부문에서 가장 흔한 오해는 “도구가 쉬우니 운영도 쉬울 것”이라는 기대다. 실제로는 구현 난도가 낮아질수록 오히려 운영 문서화와 책임 구조 정리가 더 중요해진다.

이 점에서 Lovable의 장점은 역설적으로 한계를 더 빨리 드러내 준다는 데 있다. 화면이 빨리 만들어지므로, 남는 시간 동안 데이터 기준과 권한 기준을 더 많이 토론할 수 있다. 즉 M8의 올바른 학습 방식은 화면 완성 속도에 감탄하는 것이 아니라, 남은 시간을 운영 논의에 투자하는 것이다.

협업과 인수인계를 전제로 한 결과물 정리

실습 결과를 부서 자산으로 남기려면 최소한 네 가지 문서가 있어야 한다. 서비스 개요 1장, 데이터 구조 요약 1장, 권한 규칙 1장, 배포와 수정 절차 1장이다. 여기에 README가 있으면 다음 담당자가 상당 부분 이어받을 수 있다. 반대로 화면은 화려하지만 이 네 문서가 없으면, 그 결과물은 사실상 만든 사람의 머릿속에만 존재하는 개인 자산에 머문다.

거버넌스와 보안 관점에서의 체크포인트

Lovable 공식 문서는 보안, 프라이버시, 거버넌스 지원을 강조한다. 그러나 조직 입장에서는 플랫폼이 무엇을 지원하느냐와, 사용자가 무엇을 설정하느냐를 구분해야 한다. 2FA, SSO, 권한 분리, 데이터 사용 통제 같은 기능이 있다고 해서 자동으로 올바른 운영이 되는 것은 아니다. 따라서 학습자는 기능 존재 여부보다도, 우리 조직에서 어떤 계정 정책을 쓸지, 어떤 데이터는 올리지 않을지, 어떤 프로젝트는 공개 저장소를 쓰지 않을지를 먼저 결정해야 한다.

실무용 요구사항을 더 잘 쓰는 방법

Lovable 같은 도구에 요청할 때는 "게시판 만들어줘"보다 "로그인한 사용자만 글을 쓰고, 작성자는 본인 글만 수정하며, 관리자는 공지글을 상단 고정할 수 있는 게시판"처럼 행동 규칙을 포함해 써야 한다. 화면 이름, 사용자 역할, 저장 대상, 금지해야 할 행동을 함께 적으면 결과 품질이 눈에 띄게 좋아진다. 결국 구현 품질은 프롬프트의 길이가 아니라, 운영 규칙의 명확도에 달려 있다.

공공부문에서 특히 유용한 시나리오

  • 부서 게시판과 공지 서비스
  • 결재 신청과 승인 현황 추적
  • 회의실 예약 또는 일정 공유 도구
  • 자료 공유와 댓글 기반 협업 보드
  • 민원 접수 상태 확인 도구

실무상 주의할 점

Lovable이 화면과 로직을 빠르게 만들어 준다고 해서 권한과 데이터 기준까지 자동으로 책임져 주는 것은 아니다. 예를 들어 “회원가입을 붙여 달라”는 요청만으로는 개인정보 수집 문구, 보관 기간, 관리자 열람 범위, 로그 보관 기준까지 해결되지 않는다. 따라서 공공부문 실무자는 생성 속도보다 운영 기준을 더 명확히 지시해야 한다. 이는 나중에 개발자와 협업할 때도 큰 자산이 된다.

코드 소유권과 협업

Lovable 공식 문서는 GitHub 동기화와 워크플로 결합을 강조한다. 이 점은 매우 중요하다. 교육 현장에서 빠르게 만든 결과물을 그대로 버리지 않고, 저장소에 남기고, README를 붙이고, 다음 담당자에게 인계할 수 있어야 도구가 부서 자산이 된다. 즉 M8의 진짜 성과는 화면이 예쁘게 나온 것보다, 그 결과물을 조직 안에서 계속 고칠 수 있는 형태로 남기는 것이다.

Lovable 실습을 90분 안에 끝내는 실제 순서

교육 현장에서 풀스택 도구를 처음 다룰 때 가장 흔한 실수는 한 번에 너무 큰 서비스를 요구하는 것이다. 그러면 화면은 멋지게 나오더라도, 데이터 구조와 권한이 불안정해져 끝까지 연결되지 못한다. 따라서 90분 실습 기준으로는 순서를 강하게 제한하는 편이 좋다. 처음 15분은 문제와 사용자를 문장으로 고정한다. 누가 쓰는지, 무엇을 신청하거나 조회하는지, 성공 기준이 무엇인지를 한 페이지에 적는다. 다음 15분은 엔터티와 상태값을 정리한다. 게시판이면 사용자, 게시글, 댓글, 첨부파일, 공지 여부가 필요하고, 예약이면 사용자, 예약 건, 시간대, 승인 상태가 필요하다.

그다음 20분 동안 첫 화면을 만든다. 이때는 반드시 전체 서비스가 아니라 핵심 화면 1개만 만든다. 예를 들어 회의실 예약 서비스라면 예약 신청 화면과 내 예약 목록까지만 먼저 본다. 이후 15분은 인증과 권한 요구를 추가한다. "로그인한 사용자만 신청 가능", "관리자만 승인 가능", "사용자는 자기 예약만 수정 가능"처럼 행동 규칙을 명시한다. 다음 15분은 데이터 흐름을 점검한다. 저장이 실제로 되는지, 목록이 갱신되는지, 상태가 변경되는지를 확인한다. 마지막 10분은 GitHub 동기화와 README 초안, 다음 수정 포인트를 남긴다. 이 마지막 10분을 생략하면 결과물은 교육 데모에 머문다.

이 순서를 지키는 이유는 도구를 제한하기 위해서가 아니라, 실무 감각을 보호하기 위해서다. 실제 현장에서도 서비스는 한 번에 완성되지 않는다. 핵심 기능, 저장 구조, 권한 구조, 배포와 문서가 순차적으로 정리된다. 따라서 교육에서도 같은 흐름을 밟는 편이 오히려 현실적이다.

사례 1. 회의실 예약 서비스를 Lovable로 설계하는 법

회의실 예약은 공공조직에서 매우 흔한 문제이지만, 실제로 설계해 보면 풀스택 개념이 빠짐없이 들어간다. 사용자는 신청자, 승인자, 관리자 세 부류로 나뉜다. 입력 데이터는 예약 일자, 시간, 회의실, 목적, 참석 인원, 장비 필요 여부 정도다. 출력은 예약 확정 여부, 내 신청 목록, 승인 대기 목록, 일정 충돌 경고다. 이 구조를 먼저 정리하고 Lovable에 요청하면, 화면 생성이 훨씬 안정적이다.

실습에서는 먼저 신청 화면과 목록 화면만 만든다. 그다음 승인 대기 화면을 추가하고, 관리자만 보이는 메뉴를 붙인다. 이후 상태값을 draft, requested, approved, rejected, cancelled처럼 최소 단위로 둔다. 여기서 중요한 것은 기능을 많이 넣는 것이 아니라 상태를 단순하게 유지하는 것이다. 상태값이 단순할수록 RLS와 화면 분기가 쉬워지고, 나중에 문서화도 쉬워진다. 반대로 처음부터 예외 상태를 많이 만들면 교육 시간 안에 흐름이 깨지기 쉽다.

또한 회의실 예약 서비스는 화면 문구가 매우 중요하다. 사용자는 단순히 버튼을 누르는 것이 아니라, "이 시간대가 확정되었는지", "승인 대기인지", "충돌인지"를 정확히 이해해야 한다. 따라서 Lovable 실습에서도 예쁜 카드 디자인보다 상태 문구와 경고 문장을 먼저 검토하는 편이 좋다. 이 점이 바로 공공부문형 풀스택 실습과 일반 데모 제작의 차이다.

사례 2. 부서 게시판 서비스에서 꼭 확인해야 할 것

부서 게시판은 가장 단순해 보여도 실제로는 운영 논의가 많이 필요한 서비스다. 게시글 작성, 수정, 삭제, 댓글, 공지 고정, 첨부파일, 검색, 작성자 표시가 기본 기능으로 보이지만, 실무에서는 추가 질문이 바로 따라온다. 퇴직자의 게시글은 어떻게 처리할 것인가, 관리자만 볼 수 있는 비공개 글이 필요한가, 파일 첨부는 어떤 형식을 허용하는가, 댓글 수정 이력은 남겨야 하는가 같은 문제다.

따라서 게시판을 Lovable로 만들 때는 화면보다 규칙을 먼저 적는 것이 좋다. 예를 들어 "로그인 사용자는 글 작성 가능", "작성자는 본인 글만 수정 가능", "관리자는 모든 글 삭제 가능", "공지글은 관리자만 작성 가능", "첨부파일은 PDF와 이미지로 제한"처럼 행동 규칙을 먼저 정리한다. 이 규칙이 분명하면 나중에 RLS, 파일 저장 정책, 관리자 메뉴 설계가 자연스럽게 따라온다.

또 하나 중요한 것은 검색과 정렬 기준이다. 공공조직 게시판은 시간이 지나면 글이 금세 쌓이므로 제목, 작성자, 태그, 작성일시, 공지 여부 같은 검색 기준이 없으면 곧 사용성이 떨어진다. 따라서 게시판은 단순 CRUD 예제가 아니라, 운영을 시작하면 무엇이 가장 먼저 부족해지는지 보여 주는 교육용 사례라고 볼 수 있다.

Lovable 결과를 부서 자산으로 남기는 최소 체크리스트

  • 서비스 한 줄 설명이 README 첫 문장에 적혀 있는가
  • 사용자 역할과 상태값이 문서로 정리되어 있는가
  • 주요 테이블과 컬럼이 한 장 표로 정리되어 있는가
  • 관리자 화면과 일반 화면이 구분되어 있는가
  • 다음 담당자가 수정해야 할 파일이나 위치를 설명할 수 있는가

이 다섯 가지가 남으면 실습 결과물은 단순 데모를 넘어 부서 자산으로 살아남을 가능성이 높다.

사용자별 하루 사용 시나리오로 결과를 검수하는 법

Lovable 같은 도구로 만든 결과물이 실제 업무에 가까운지 확인하려면 기능 목록보다 사용자 하루 흐름으로 검수하는 편이 훨씬 효과적이다. 예를 들어 회의실 예약 서비스를 만든다면, 신청자는 오전에 빈 시간대를 확인하고 예약을 넣고, 오후에 승인 여부를 다시 본다. 승인자는 점심 전후 승인 대기 목록을 보고 충돌 여부를 확인한 뒤 승인 또는 반려를 수행한다. 관리자는 주 1회 로그와 충돌 건수, 사용량을 점검한다. 이 세 사람의 하루 흐름을 문장으로 써 보고 화면을 따라가 보면, 빠진 기능과 모호한 문구가 곧바로 보인다.

신청자 관점에서는 세 가지가 핵심이다. 신청이 쉬운가, 현재 상태가 분명한가, 반려되었을 때 무엇을 해야 하는가다. 승인자 관점에서는 정보 과잉이 가장 큰 문제다. 승인에 필요한 정보만 빠르게 보여 주어야 하며, 신청자 연락처나 회의 목적, 장비 필요 여부 같은 판단 정보가 한눈에 들어와야 한다. 관리자 관점에서는 운영 메타정보가 중요하다. 최근 한 주간 승인 지연이 있었는지, 자주 충돌하는 시간대가 무엇인지, 누가 가장 많이 쓰는지를 보는 요약 화면이 필요할 수 있다. 즉 같은 서비스라도 사용자별로 "좋은 화면"의 정의가 달라진다.

이 검수 방식은 부서 게시판에도 그대로 적용된다. 일반 사용자는 글 작성과 검색이 쉬워야 하고, 관리자에게는 공지 고정과 부적절 게시글 관리가 쉬워야 하며, 운영 담당자는 첨부파일 정책과 게시판 이용 현황을 확인할 수 있어야 한다. 이런 식으로 사용자별 시나리오를 한 번만 써 봐도, 단순 CRUD 데모가 아닌 운영형 서비스로 한 단계 가까워진다.

실습 결과를 운영 후보로 승격시키는 마지막 질문

Lovable 실습이 끝난 뒤에는 다음 질문을 반드시 던지는 것이 좋다. 첫째, 이 서비스는 지금 누구 한 명의 머릿속에만 의존하고 있지 않은가. 둘째, 실제 사용자가 처음 접했을 때 3분 안에 핵심 기능을 이해할 수 있는가. 셋째, 승인자와 관리자 화면은 정말 필요한 정보만 보여 주는가. 넷째, 잘못 입력했을 때 복구 경로가 있는가. 다섯째, 운영 전 남은 논의가 무엇인지 문서로 남겼는가. 이 질문에 답하지 못하면 결과물은 여전히 시연용에 머물 가능성이 높다.

특히 다섯 번째 질문이 중요하다. 많은 팀이 "이 정도면 거의 완성"이라고 생각하지만, 실제로는 운영 문장과 책임 구조가 비어 있다. 공공부문에서는 바로 이 빈칸 때문에 좋은 데모가 실제 서비스로 이어지지 못하는 경우가 많다. 따라서 M8의 마지막 단계는 화면 완성이 아니라 남은 공백을 문장으로 적는 일이라고 보는 편이 맞다.

Section 12. M9 Stitch 디자인 파이프라인

M9는 구현보다 한 단계 앞선 설계 감각을 다룬다. 지금까지는 요구사항을 문장으로 적고 바로 코드나 화면을 만들었다면, Stitch는 의도와 분위기와 흐름을 먼저 시각화한 뒤, 그 결과를 다시 코드로 넘기는 파이프라인을 경험하게 한다. 이 과정은 비개발 실무자뿐 아니라 비디자이너에게도 큰 의미가 있다. 무엇을 만들고 싶은지 말로만 설명하는 것과, 실제 화면 흐름으로 상상한 뒤 구현으로 넘기는 것은 정확도가 전혀 다르기 때문이다.

원본 슬라이드는 Stitch를 “디자인 단계의 Lovable”처럼 설명한다. 이 비유는 적절하다. Stitch는 자연어, 이미지, 음성, 디자인 시스템을 받아 여러 화면을 일관된 톤으로 제안하고, 그 결과를 다른 도구로 넘기는 데 강점이 있다. 특히 민원 접수 흐름, 신청서 작성 단계, 대시보드 홈과 상세화면처럼 한 화면이 아니라 연결된 경험을 설계해야 하는 경우 유용하다. 단일 HTML이나 풀스택 도구만으로는 처음부터 전체 흐름을 상상하기 어려운 사용자에게 훌륭한 중간 단계가 된다.

디자인-투-코드 파이프라인의 의미

공공부문 서비스는 기능이 같아도 표현 방식에 따라 사용성이 크게 달라진다. 예를 들어 민원 접수 화면은 친절하고 단계적이어야 하고, 내부 승인 화면은 정보량이 많되 실수 유발 요소를 줄여야 한다. Stitch 같은 도구는 이런 감각적 요구를 문장과 시안 수준에서 먼저 조정하게 해 준다. 이후 AI Studio, Lovable, 코딩 에이전트로 넘기면 “코드를 만들기 전에 사용자 경험을 먼저 합의했다”는 장점이 생긴다.

Stitch를 쓸 때의 실무 원칙

  • 화면 목적과 사용자 감정을 함께 적는다.
  • 한 번에 모든 기능을 넣기보다 핵심 흐름 3~5화면을 먼저 만든다.
  • 기관 색상, 폰트, 금지 표현 같은 디자인 규칙을 미리 정리한다.
  • 시안을 바로 배포하지 말고 실제 업무 용어와 절차에 맞는지 검토한다.

왜 디자인 단계가 구현 품질을 좌우하는가

공공 서비스는 민간 서비스보다 설명성이 더 중요하다. 사용자가 버튼을 왜 눌러야 하는지, 어떤 데이터가 제출되는지, 처리 상태가 무엇을 의미하는지, 오류가 나면 어디로 가야 하는지가 화면 자체에서 드러나야 한다. 이 점 때문에 디자인은 단순 장식이 아니라 행정 절차의 번역이다. Stitch 같은 도구가 의미 있는 이유도 여기에 있다. 구현 전에 단계 화면을 먼저 합의하면, 나중에 개발 과정에서 생길 오해를 크게 줄일 수 있다.

특히 민원, 신청, 승인, 예약 같이 단계성이 강한 서비스는 화면 흐름 설계가 곧 업무 설계다. 입력 화면, 확인 화면, 처리 진행 화면, 완료 화면, 후속 안내 화면이 어떤 순서로 나와야 하는지, 각 단계에서 사용자 불안이 어디서 발생하는지, 어떤 문구가 과장되거나 오해를 낳는지 미리 점검할 수 있다. 이 과정을 건너뛰고 바로 코드로 가면, 기능은 동작해도 사용자 경험이 어색한 서비스가 나오기 쉽다.

디자인 시스템을 문서로 남겨야 하는 이유

원본 슬라이드가 말하는 DESIGN.md의 핵심도 결국 문서화다. 기관 색상, 헤더 스타일, 본문 문체, 버튼 강약, 경고 문구, 필수 고지 문구를 문서로 정리해 두면 이후 여러 화면이 생겨도 일관성을 유지하기 쉽다. 공공조직에서는 이 일관성이 곧 신뢰다. 부서별로 서로 다른 톤과 다른 버튼 문구를 쓰면 사용자는 서비스가 같은 조직의 것인지조차 헷갈릴 수 있다.

따라서 M9는 예쁜 시안을 만드는 수업이라기보다, 서비스의 언어와 톤을 먼저 정리하는 수업으로 이해하는 편이 좋다. 이 과정에서 만든 디자인 가이드는 이후 Lovable, 코딩 에이전트, 개발팀 핸드오프 단계에서 계속 재사용되는 기준 문서가 된다.

디자인 결과를 운영 코드로 넘길 때 확인할 것

  1. 화면 간 흐름이 실제 업무 절차와 맞는가
  2. 필수 입력과 선택 입력이 구분되어 있는가
  3. 모바일과 데스크톱 모두에서 읽기 쉬운가
  4. 이후 권한과 데이터 구조를 붙일 자리가 남아 있는가

Stitch 화면 설계 세션을 운영하는 실제 방법

Stitch나 유사 디자인 도구를 사용할 때는 종종 "예쁜 화면을 얼마나 빨리 뽑는가"에 시선이 쏠린다. 그러나 교육 효과를 높이려면 세션 진행 방식부터 바꿔야 한다. 첫 단계는 컴포넌트가 아니라 동사를 적는 것이다. 신청한다, 확인한다, 수정한다, 승인한다, 반려한다, 검색한다 같은 동사를 먼저 적으면 서비스 흐름이 보인다. 두 번째 단계는 이 동사에 대응하는 화면 수를 정한다. 신청 화면, 확인 화면, 승인 대기 화면, 상세 화면, 완료 화면처럼 3~5개 핵심 화면을 고른다. 세 번째 단계에서야 각 화면에 필요한 필드와 경고 문구를 붙인다.

이렇게 하면 화면 설계가 장식 단계가 아니라 업무 번역 단계가 된다. 반대로 버튼 위치나 카드 디자인부터 논의하면, 실제로는 어떤 사용자가 어떤 흐름으로 움직이는지에 대한 합의 없이 시안만 화려해진다. 공공 서비스는 사용자가 무엇을 기대해야 하는지를 명확히 보여 주는 것이 중요하므로, Stitch 세션에서도 문구와 단계 설계가 가장 먼저 나와야 한다.

신청형 서비스 화면을 설계할 때 보는 네 장면

신청형 서비스는 보통 네 장면만 잘 설계해도 전체 품질이 크게 올라간다. 첫째는 입력 화면이다. 무엇을 적어야 하는지, 어떤 항목이 필수인지, 저장 전 임시저장이 가능한지가 중요하다. 둘째는 확인 화면이다. 제출 전에 사용자가 스스로 검토할 수 있어야 하며, 이 장면에서 많은 오류를 줄일 수 있다. 셋째는 상태 확인 화면이다. 제출 후 현재 상태가 무엇인지, 누가 다음 조치를 하는지 설명해야 한다. 넷째는 완료 또는 반려 화면이다. 다음 행동을 안내해야 하며, 반려라면 수정 경로를 분명히 보여 주어야 한다.

이 네 장면을 Stitch에서 먼저 합의하면, 이후 Lovable이나 개발팀으로 넘길 때 요구사항 품질이 확 달라진다. 화면 수가 늘어나는 것이 문제가 아니라, 흐름이 분명해지는 것이 핵심이다. 특히 공공 서비스는 사용자가 한 번만 써 보고도 큰 불안을 느끼지 않아야 하므로, 확인과 완료 화면의 설명성이 매우 중요하다.

대시보드 화면은 예쁘기보다 해석 가능해야 한다

대시보드는 가장 흔하게 과잉 디자인되는 화면이다. 차트가 많고 색상이 화려할수록 좋아 보이지만, 실제 실무자는 그 안에서 빠르게 판단할 수 있어야 한다. 따라서 Stitch 단계에서는 먼저 "이 화면을 보는 사람이 첫 10초 안에 무엇을 알아야 하는가"를 적는 편이 좋다. 예를 들어 총괄 담당자는 누락 부서 수, 처리 대기 건수, 이번 주 변화량을 먼저 봐야 할 수 있다. 반면 관리자에게는 최근 오류 로그, 승인 대기, 비정상 데이터가 더 중요할 수 있다.

즉 같은 대시보드라도 사용자별 핵심 지표가 다르다. 이 차이를 반영하지 않으면 화면은 화려하지만 누구에게도 맞지 않는 중간값 UI가 나오기 쉽다. 따라서 Stitch 세션에서 사용자별 첫 화면 우선순위를 나누는 훈련은 매우 중요하다.

디자인 리뷰 회의에서 반드시 물어야 할 질문

  • 사용자는 이 화면에서 다음 행동을 직관적으로 이해할 수 있는가
  • 상태값과 경고 문구가 실제 행정 용어와 어긋나지 않는가
  • 모바일에서도 핵심 정보가 잘리지 않는가
  • 관리자 기능이 일반 사용자 흐름을 방해하지 않는가
  • 디자인 결과를 코드로 넘길 때 필요한 규칙 문서가 남아 있는가

이 질문에 답할 수 있으면, 디자인 단계는 단순 시안 생산이 아니라 서비스 품질 보증 단계가 된다.

Section 13. M10 인증과 권한 설계

많은 학습자가 로그인 화면을 붙이는 순간 서비스를 완성했다고 느낀다. 그러나 실제 운영은 그때부터 시작된다. 인증은 사용자가 누구인지 확인하는 문제이고, 권한은 그 사용자가 무엇을 볼 수 있고 무엇을 바꿀 수 있는지 결정하는 문제다. 이 둘을 구분하지 않으면 서비스는 바로 불안정해진다. 예를 들어 모든 로그인 사용자가 모든 데이터를 수정할 수 있다면, 그 서비스는 운영 가능한 상태가 아니다.

Supabase Auth 공식 문서가 강조하듯 인증과 권한은 서로 연결되어 있지만 역할이 다르다. 인증은 비밀번호, OTP, 소셜 로그인, SSO 등을 통해 신원을 확인하고, 권한은 JWT와 RLS 같은 기법을 통해 어떤 행과 기능에 접근할 수 있는지를 결정한다. 공공부문 실무에서 이 개념은 더욱 중요하다. 같은 화면을 쓰더라도 일반 사용자, 담당자, 승인자, 관리자에게 보이는 데이터와 버튼이 달라야 하기 때문이다.

표 9. 인증과 권한 설계 기본표

역할 가능한 행동 보이면 안 되는 것 구현 포인트
일반 사용자 본인 글 작성, 조회, 수정 타인 개인정보, 관리자 기능 본인 데이터만 RLS로 허용
담당자 접수 건 확인, 상태 변경, 내부 메모 관리자 설정 메뉴 역할 컬럼과 화면 분기 필요
승인자 승인 또는 반려, 이력 확인 시스템 설정 변경 승인 흐름 전용 권한 필요
관리자 사용자 관리, 설정 변경, 로그 확인 해당 없다 별도 메뉴와 감사 로그 필요

RLS를 실무 언어로 이해하기

Supabase의 RLS 문서는 정책을 사실상 모든 질의에 자동으로 붙는 보이지 않는 WHERE 절처럼 이해하라고 설명한다. 이 비유는 실무자에게 매우 유용하다. 예를 들어 "사용자는 자기 글만 본다"는 규칙은 단순한 화면 제어가 아니라 데이터 접근 자체의 규칙이 된다. 즉 화면에서 버튼을 숨기는 것만으로는 충분하지 않고, 데이터베이스 단계에서부터 해당 사용자가 볼 수 있는 행을 제한해야 한다는 뜻이다.

또한 공식 문서가 분명히 강조하는 몇 가지 기본 원칙이 있다. 외부에 노출된 스키마의 테이블은 RLS를 반드시 활성화해야 하고, 정책이 없으면 공개 키로는 아무 데이터도 접근되지 않는다. auth.uid()는 로그인하지 않은 상태에서는 null을 반환하므로, 인증 여부를 명시적으로 확인하는 것이 좋다. 그리고 service key는 브라우저에 절대 노출하면 안 된다. 이 네 가지는 단순 기술 팁이 아니라, 공공 서비스 운영에서 반드시 기억해야 하는 최소 수칙이다.

권한 모델을 만들 때 자주 빠지는 함정

첫 번째 함정은 역할 이름만 정하고 실제 정책을 세밀하게 분리하지 않는 것이다. 예를 들어 담당자와 승인자를 나눴다고 해도, 어떤 상태의 데이터만 바꿀 수 있는지, 이미 승인된 데이터는 누가 되돌릴 수 있는지까지 정의하지 않으면 실제 운영에서 충돌이 난다. 두 번째 함정은 읽기 권한과 수정 권한을 따로 보지 않는 것이다. 많은 서비스에서 조회는 넓게 허용해도 수정은 매우 제한해야 한다. 세 번째 함정은 테스트 계정과 운영 계정의 혼용이다. 공공조직에서는 교육이나 시연용 계정이 운영 데이터에 접근하지 못하도록 분리하는 것이 매우 중요하다.

성능과 운영까지 고려한 RLS 설계

Supabase 문서는 RLS가 강력한 대신 성능 영향을 고려해야 한다고 밝히고 있다. 특히 정책에 자주 쓰는 컬럼에는 인덱스를 두고, 질의에도 명시적인 필터를 추가하는 것이 좋다고 권고한다. 이 설명은 교육용 실습을 실제 운영 단계로 넘길 때 매우 중요하다. 초기에는 데이터가 적어 문제가 보이지 않다가도, 게시글이나 신청 건수가 늘어나면 갑자기 느려지는 경우가 있기 때문이다.

따라서 공공 서비스 설계에서는 "권한이 맞는가"와 함께 "권한 규칙이 커져도 견딜 수 있는가"를 같이 봐야 한다. 예를 들어 팀 단위 접근, 기관 단위 접근, 관리자 우회 조회처럼 복수 규칙이 붙는 순간 정책은 복잡해진다. 이때는 화면에서 숨기는 수준의 임시방편보다, 역할과 데이터 구조를 다시 단순화하는 것이 장기적으로 더 안전하다.

인증과 권한 테스트 체크리스트

  • 비로그인 상태에서 데이터가 전혀 보이지 않는가
  • 일반 사용자가 다른 사람의 데이터를 조회하거나 수정하지 못하는가
  • 승인자와 관리자의 화면이 실제로 다르게 보이는가
  • 삭제, 반려, 상태 변경 같은 위험 동작에 추가 검토 절차가 있는가
  • 만료된 세션이나 권한 변경 시 바로 동작이 달라지는가

실무에서 자주 빠지는 질문

  • 퇴직자나 전출자의 계정은 누가 비활성화하는가
  • 테스트 계정과 운영 계정을 어떻게 분리하는가
  • 관리자 계정의 2차 인증이 필요한가
  • 로그는 어디까지 남기고 누가 볼 수 있는가

계정 생애주기 관점에서 권한을 설계해야 하는 이유

인증과 권한을 설계할 때 많은 팀이 화면 기준으로만 생각한다. 그러나 실제 운영에서는 사용자의 생애주기를 같이 봐야 한다. 신규 사용자가 들어올 때 누가 계정을 만들고 어떤 기본 역할을 주는가, 부서 이동 시 어떤 권한을 회수하고 어떤 권한을 새로 주는가, 퇴직이나 전출 시 계정을 언제 비활성화하는가를 정하지 않으면 서비스는 금방 관리 불능 상태가 된다. 공공조직에서는 특히 부서 이동과 권한 변경이 잦기 때문에, 권한 설계는 화면 설계 못지않게 조직 흐름과 맞물려야 한다.

실무적으로는 Joiner, Mover, Leaver 세 장면으로 나누어 생각하면 쉽다. Joiner는 신규 사용자 등록 시 기본 역할과 초기 안내를 정하는 단계다. Mover는 역할 변경, 부서 이동, 승인권 이관이 일어나는 단계다. Leaver는 퇴직, 전출, 장기 미사용 계정을 비활성화하고 로그 열람 권한을 정리하는 단계다. 이 세 단계가 정리되어야 권한은 문서상 규칙이 아니라 실제 운영 규칙이 된다.

승인형 서비스 권한 설계를 단계별로 보면

승인형 서비스는 단순히 신청자와 승인자를 나누는 것으로 끝나지 않는다. 첫 단계는 조회 권한을 나누는 것이다. 신청자는 자기 건만 보고, 승인자는 담당 범위의 신청만 보고, 관리자는 전체를 본다. 두 번째 단계는 상태 변경 권한을 나누는 것이다. 신청자는 제출 전까지 수정 가능하고, 승인자는 승인과 반려만 가능하며, 관리자는 예외 조정이나 강제 종료 권한을 가진다. 세 번째 단계는 로그 권한을 나누는 것이다. 모든 상태 변경이 기록되고, 누가 언제 바꿨는지 관리자만 볼 수 있어야 한다.

이때 화면과 데이터 규칙이 어긋나지 않도록 주의해야 한다. 예를 들어 화면에서 버튼을 숨겼더라도, RLS나 서버 정책이 느슨하면 우회 접근이 가능할 수 있다. 따라서 권한은 항상 화면, API, 데이터베이스 세 층에서 함께 점검해야 한다. C06의 교육적 가치는 바로 이 다층 구조를 이해하게 만드는 데 있다.

감사 로그를 설계할 때 남겨야 할 최소 항목

  • 사용자 식별자
  • 변경한 대상과 이전 상태
  • 변경 이후 상태
  • 변경 시각
  • 어떤 화면 또는 기능에서 수행되었는지

이 다섯 항목만 남아도 운영 사고가 났을 때 원인 추적이 훨씬 쉬워진다. 반대로 로그가 없으면 권한 설계는 문서상으로는 좋아 보여도 실제 운영에서 설명 가능성이 크게 떨어진다.

인증과 권한은 나중에 붙이는 옵션이 아니라, 처음 데이터 구조를 설계할 때부터 함께 생각해야 하는 기본값이다. 특히 결재, 민원, 인사, 예산처럼 민감한 업무일수록 더 그렇다. 따라서 M10의 목표는 로그인 화면 구현 자체가 아니라, 운영 가능한 권한 구조를 머릿속에 넣는 데 있다.

Section 14. M11 GitHub와 배포

도구가 부서 자산이 되는 마지막 단계는 공개 가능한 형태로 정리되는 순간이다. 내 PC에서만 열리는 파일은 나에게만 의미가 있지만, URL이 생기고 버전이 남고 인수인계 문서가 생기면 조직 자산이 된다. M11은 이 전환을 다루는 모듈이다. 여기서 중요한 것은 화려한 DevOps를 배우는 것이 아니라, 최소한의 배포 경로와 변경 이력을 남기는 습관을 익히는 것이다.

배포 경로는 크게 두 가지로 나눌 수 있다. 하나는 Lovable 같은 도구에서 바로 Publish하는 방식이고, 다른 하나는 GitHub 저장소에 올린 뒤 Netlify나 Vercel로 연결하는 방식이다. 전자는 빠르고 단순하며, 후자는 코드와 이력 관리, 협업과 인수인계 측면에서 더 강하다. 어느 경로를 고를지는 결과물의 성격에 따라 달라진다. 단일 HTML이나 정적 사이트라면 GitHub와 정적 호스팅이 잘 맞고, Lovable 기반 풀스택 서비스라면 기본 Publish 흐름이 빠를 수 있다.

2026년 배포 운영에서 특히 중요한 것

Vercel의 2026년 배포 문서는 한 번의 배포가 단순 업로드가 아니라 고유한 URL을 가진 독립 실행 결과라는 점을 분명히 한다. Git 연결 배포, CLI 배포, Deploy Hook, REST API처럼 배포 방법이 다양하다는 사실보다 더 중요한 것은 Local, Preview, Production이라는 세 가지 환경 구분이다. 공공부문 서비스에 이 개념을 적용하면 매우 유용하다. 시연용 URL, 검수용 URL, 실제 공개 URL을 분리할 수 있기 때문이다.

이 구분은 곧 승인 절차와도 연결된다. 실무에서는 기능이 완성되었다고 바로 운영 URL에 반영하기 어렵다. 먼저 Preview에서 화면, 권한, 문구를 확인하고, 그다음 승인권자나 보안 담당자 검토를 거쳐 Production으로 올리는 편이 안전하다. 따라서 M11의 진짜 핵심은 배포 버튼을 누르는 법보다도, 변경을 점진적으로 검수하는 운영 흐름을 이해하는 데 있다.

Git 배포와 수동 배포를 어떻게 나눌 것인가

Git 기반 배포는 이력과 재현성이 강점이다. 어떤 수정이 어떤 커밋에서 들어갔는지 남고, 문제가 생기면 이전 상태로 돌아가기 쉽다. 반면 빠른 시연이나 교육 현장에서는 파일 몇 개만 올리는 수동 방식이 더 빠를 때도 있다. 실무적으로는 초기에 수동 배포로 감각을 잡더라도, 실제 운영을 생각하는 순간 Git 기반 흐름으로 전환하는 편이 좋다. 그래야 나중에 담당자가 바뀌어도 변경 이력을 추적할 수 있다.

운영 문서와 배포 로그를 함께 남겨야 하는 이유

배포는 성공했는데 왜 화면이 다르게 보이는지, 어떤 환경변수를 바꿨는지, 어떤 파일이 빌드에 포함되었는지는 나중에 반드시 문제가 된다. Vercel 문서가 배포별 리소스와 로그, 함수, 정적 자산을 확인할 수 있게 설명하는 이유도 여기에 있다. 즉 배포는 URL 생성으로 끝나는 일이 아니라, 이후 문제를 추적할 수 있는 흔적을 남기는 일이다. 공공부문 서비스에서는 이 흔적이 곧 책임소재를 줄이는 장치이기도 하다.

최소 롤백 절차

  1. 현재 운영 중인 URL과 최근 성공 배포를 기록해 둔다.
  2. 새 배포는 먼저 Preview에서 기능과 권한을 점검한다.
  3. Production 반영 전 README와 변경 요약을 갱신한다.
  4. 문제 발생 시 즉시 이전 성공 버전으로 되돌릴 수 있는 경로를 확보한다.

이 네 단계만 지켜도 “실수 한 번에 서비스 전체가 흔들리는” 상황을 크게 줄일 수 있다.

GitHub를 써야 하는 이유

GitHub는 단순 저장소가 아니라 변경 이력과 협업 기록을 남기는 기반이다. 누가 언제 어떤 파일을 바꾸었는지, 이전 버전으로 되돌릴 수 있는지, 다음 담당자가 구조를 이해할 수 있는지가 모두 여기에 걸려 있다. 비개발 실무자에게도 GitHub는 더 이상 선택이 아니라, 결과물을 조직 자산으로 남기기 위한 최소한의 도구라고 보는 편이 맞다.

GitHub 공식 문서는 repository를 코드와 파일, 그리고 각 파일의 revision history를 함께 담는 기본 단위로 설명한다. 또한 저장소는 협업자 여러 명을 둘 수 있고 공개 범위도 public 또는 private 등으로 관리된다고 밝힌다. 이 설명은 단순해 보이지만, 공공 실무에서는 매우 중요하다. 많은 팀이 결과물을 여전히 공유 폴더와 메신저 파일 전송으로 관리하기 때문이다. 이 방식은 “지금 최신본이 무엇인지”, “누가 무엇을 바꿨는지”, “이전 상태로 되돌릴 수 있는지”를 설명하기 어렵다. 저장소 개념이 들어오면 비로소 결과물이 프로젝트 단위로 관리되기 시작한다.

또한 GitHub 문서는 이슈, 토론, 풀리퀘스트, 프로젝트 보드 같은 협업 기능을 함께 소개한다. 공공조직에서는 이를 모두 쓸 필요는 없지만, 최소한 “수정 요청을 어디에 남길 것인가”와 “변경 제안을 어떻게 검토할 것인가”의 틀은 필요하다. 이 때문에 GitHub는 단순 백업 도구가 아니라 작업 과정 자체를 투명하게 남기는 도구라고 보는 편이 맞다.

저장소를 읽는 법과 운영 자산화

저장소를 잘 만든다는 것은 코드를 많이 넣는다는 뜻이 아니다. 오히려 다음 네 가지가 더 중요하다. 첫째, README가 목적과 실행법을 설명하는가. 둘째, 폴더 구조가 한눈에 이해되는가. 셋째, 샘플 데이터와 실제 데이터가 분리되어 있는가. 넷째, 공개 범위와 권한이 적절한가. GitHub 문서가 repository visibility와 permission을 강조하는 이유도 여기에 있다. 공공 프로젝트는 private가 기본일 수 있지만, 오픈 가능한 정적 도구나 학습용 샘플은 public 저장소가 오히려 확산에 유리할 수 있다. 중요한 것은 관성적으로 모두 비공개 또는 모두 공개로 가는 것이 아니라, 자료 성격에 맞게 나누는 일이다.

GitHub 협업 워크플로우를 교재형으로 정리하면

  1. 저장소를 만든다.
  2. README에 목적과 실행법을 먼저 적는다.
  3. 초안을 올린다.
  4. 수정은 브랜치나 별도 변경 단위로 관리한다.
  5. 검토 후 main에 반영한다.
  6. 배포 도구와 연결해 URL을 만든다.
  7. 변경 로그와 운영 메모를 갱신한다.

이 일곱 단계가 잡히면, AI로 급히 만든 초안도 조직 자산으로 전환될 수 있다. 반대로 이 흐름이 없으면 아무리 좋은 도구를 만들어도 결국 개인 PC 안의 산출물로 끝나기 쉽다.

공개 저장소와 비공개 저장소를 나누는 판단 기준

공개 저장소는 확산과 협업이 쉽고, 참고와 재사용이 빠르다. 그러나 GitHub 공식 문서가 설명하듯 공개 범위는 더 큰 보안 주의를 요구한다. 코드와 문서, 설정값이 외부에 드러날 수 있으므로 민감정보, 환경변수, 내부 경로, 실데이터 예시는 반드시 분리해야 한다. 반대로 비공개 저장소는 접근을 제한할 수 있지만, 그렇다고 보안이 자동으로 해결되는 것은 아니다. 권한을 누구에게 줄지, MFA와 계정 관리를 어떻게 할지, 퇴직자나 외부 협력사 계정을 어떻게 정리할지 같은 운영 기준이 필요하다.

따라서 C06의 교재에서는 GitHub를 단순히 “코드 올리는 곳”으로 설명하면 부족하다. 저장소는 공공부문에서 결과물의 공개 범위와 협업 구조를 정하는 그릇이기도 하다. 이 관점이 있어야 이후 Vercel, Netlify, Pages 같은 배포 단계도 자연스럽게 이해된다.

README를 왜 먼저 써야 하는가

README는 개발자를 위한 장식 문서가 아니다. 오히려 비개발 실무자에게 더 중요하다. README가 없으면 다음 담당자는 파일을 열어 봐야 하고, 어떤 것이 시작점인지, 어디를 수정해야 하는지, 무엇이 민감한지 스스로 추정해야 한다. 반대로 README가 있으면 저장소는 설명 가능한 결과물이 된다. 즉 GitHub에서 가장 먼저 작성해야 하는 파일은 종종 코드가 아니라 README다. 이것이 교재가 계속 문서화를 강조하는 이유다.

브랜치에서 운영 URL까지 가는 실제 배포 흐름

교육 현장에서 가장 실용적인 배포 흐름은 브랜치, 미리보기, 검토, 운영 반영의 네 단계다. 먼저 작업 브랜치에서 수정한다. 이 단계에서는 기능 실험과 화면 수정이 자유롭다. 다음으로 Preview URL을 확인한다. Netlify의 Branch Deploy나 Deploy Preview, Vercel의 Preview Deployment를 활용하면 운영 URL을 건드리지 않고 결과를 볼 수 있다. 그다음 검토자를 지정해 권한, 문구, 레이아웃을 확인한다. 마지막으로 운영 반영을 수행하고 변경 로그를 남긴다.

이 흐름의 장점은 수정과 공개를 분리한다는 점이다. 많은 실습 결과물이 시연만 되고 끝나는 이유는 "수정하는 사람"과 "승인하는 사람"의 단계가 분리되지 않기 때문이다. Preview 단계를 두면 기능 완성, 정책 검토, 공개 반영이 서로 다른 단계가 되므로 조직 적용성이 훨씬 높아진다.

환경변수와 비밀정보를 다루는 규칙

배포가 본격화되면 가장 먼저 생기는 문제가 비밀정보 관리다. API 키, 데이터베이스 키, 서비스 토큰, 관리자 초기 계정 정보를 코드에 직접 넣으면 안 된다. Netlify와 Vercel이 배포 맥락별 환경변수 설정을 제공하는 이유도 여기에 있다. 공공 실무에서는 이 규칙을 간단히 세 문장으로 고정하는 편이 좋다. 첫째, 비밀정보는 저장소에 넣지 않는다. 둘째, 환경별로 अलग 값을 둔다. 셋째, 문서에는 키 값이 아니라 어디서 설정하는지만 적는다.

이 규칙이 지켜지지 않으면 공개 저장소는 물론 비공개 저장소에서도 문제가 커질 수 있다. 특히 실습용 키와 운영용 키가 섞이면, 시연은 성공했는데 운영에서 장애가 나는 일이 생긴다. 따라서 배포 교육은 URL 생성보다 환경변수 규율을 먼저 가르치는 편이 오히려 실무적이다.

배포 실패를 줄이는 점검 루틴

  1. Preview에서 먼저 권한과 레이아웃을 확인한다.
  2. 이미지 경로와 외부 라이브러리 로딩을 점검한다.
  3. 모바일 화면과 데스크톱 화면을 각각 확인한다.
  4. README와 변경 요약을 갱신한다.
  5. 운영 반영 후 즉시 첫 화면과 핵심 기능을 재확인한다.

이 다섯 단계는 단순해 보여도 실제로 배포 사고 대부분을 줄여 준다. 특히 정적 파일 기반 도구는 상대 경로 하나만 어긋나도 화면이 깨질 수 있으므로, 배포 후 5분 점검 루틴을 운영 규칙으로 두는 편이 좋다.

GitHub에서 배포까지 이어지는 최소 절차

첫째, 저장소를 만들고 결과물 파일을 올린다. 둘째, README를 작성해 실행 방법과 수정 포인트를 남긴다. 셋째, Netlify나 Vercel 같은 정적 호스팅 도구에서 저장소를 연결한다. 넷째, 첫 배포 후 URL을 열어 실제로 이미지 경로와 링크, 모바일 표시가 정상인지 확인한다. 다섯째, 수정이 생기면 다시 저장소에 반영하고 재배포되는 흐름을 익힌다. 이 정도만 익혀도 "파일 전달" 수준을 넘어 "버전 있는 서비스 운영"으로 사고가 바뀐다.

여기에 Netlify의 2026 배포 개념을 같이 보면 배포 품질을 이해하기 더 쉽다. Netlify는 atomic deploy를 강조한다. 즉 파일 하나씩 덮어쓰는 방식이 아니라, 새 배포본이 완전히 준비된 뒤 한 번에 교체되어 사이트가 중간 상태로 노출되지 않는다. 이 설명은 공공부문 실무자에게 특히 유용하다. 화면 파일은 바뀌었는데 CSS나 이미지가 아직 안 올라와서 잠시 깨지는 상황을 줄일 수 있기 때문이다. 또한 Deploy Preview와 Branch Deploy 같은 개념은 승인권자나 동료에게 운영 URL과 분리된 검토용 링크를 제공하는 데 적합하다.

Netlify 문서가 설명하는 deploy context 개념도 실무적으로 중요하다. production, deploy-preview, branch-deploy, dev처럼 배포 맥락에 따라 환경변수와 빌드 설정을 달리할 수 있다. 이것은 교육용 예제를 실제 조직용 도구로 바꿀 때 큰 차이를 만든다. 예를 들어 샘플 데이터용 환경과 운영 환경을 분리하고, 공개 URL과 검토 URL을 अलग 두는 방식이 가능해진다. 결국 배포는 결과물을 올리는 일이 아니라, 어떤 맥락에서 어떤 설정으로 실행할지를 구분하는 일이다.

Vercel과 Netlify를 고를 때의 실무 기준

Vercel은 Git 중심의 Preview와 Production 흐름이 분명하고, 각 배포가 고유 URL을 갖는다는 점이 강점이다. Netlify는 atomic deploy, Branch Deploy, Deploy Preview, 검토 협업 기능이 잘 드러나 있다. 따라서 React나 코드 중심 워크플로에는 Vercel이 자연스러울 수 있고, 정적 사이트와 배포 미리보기 협업에는 Netlify가 편할 수 있다. 다만 어떤 플랫폼을 고르든 핵심은 같다. 운영 URL, 검토 URL, 개발 URL을 분리하고, 환경변수는 저장소가 아니라 플랫폼 설정에서 관리하며, 실패한 배포의 로그를 남겨야 한다.

README에 반드시 들어가야 할 항목

  • 서비스 목적과 대상 사용자
  • 실행 방법과 필요한 계정 또는 환경변수
  • 핵심 파일 구조와 수정이 잦은 위치
  • 관리자 또는 다음 담당자를 위한 점검 순서
  • 개인정보와 샘플 데이터 관련 주의사항

배포 전에 반드시 확인할 것

  • 샘플 데이터가 실제 개인정보처럼 보이지 않는가
  • 환경변수와 API 키가 코드에 직접 들어가 있지 않은가
  • README에 실행법, 수정 포인트, 관리자 정보를 적었는가
  • 모바일과 데스크톱에서 기본 동작이 무너지지 않는가

Section 15. M12 종합 워크숍

원본 슬라이드는 종합 워크숍의 제목만 제시하지만, 이 모듈은 사실상 앞선 모든 실습을 하나의 서비스 경험으로 묶는 단계다. 가장 좋은 종합 과제는 복잡한 신규 아이디어보다, 이미 Day 1과 Day 2에서 만든 산출물을 조합해 실제 업무 흐름처럼 보이게 만드는 것이다. 예를 들어 Day 1의 Python 자동화로 데이터를 정리하고, Day 2의 웹앱에서 그 결과를 조회하게 하거나, VBA로 만든 통합 결과를 Lovable 기반 승인 화면과 연결하는 방식이 가능하다.

이 단계에서는 “더 많은 기능”보다 “끝까지 설명 가능한 구조”가 중요하다. 입력은 어디서 들어오고, 누가 어떤 화면에서 무엇을 보고, 어떤 권한으로 수정하며, 결과는 어디에 저장되고, 장애가 나면 누가 무엇을 확인하는지까지 설명할 수 있어야 한다. 즉 종합 워크숍은 구현 실습인 동시에 발표와 운영 설계 실습이다.

  • 교육 신청 폼 + 참가자 명단 관리 + 결과 안내 페이지

종합 과제를 설계하는 실제 순서

좋은 종합 과제는 기능이 많은 과제가 아니라, Day 1과 Day 2의 산출물이 논리적으로 이어지는 과제다. 예를 들어 Python 자동화로 매일 들어오는 CSV를 정리하고, Lovable로 만든 웹 화면에서 그 결과를 조회하게 하는 과제는 구현 수준이 지나치게 높지 않으면서도 통합 경험을 준다. 반대로 처음부터 신규 데이터 수집, 로컬 LLM, 승인 흐름, 외부 배포를 한 번에 다 넣으면 대부분 중간에서 끊긴다. 따라서 설계 순서는 “이미 만든 것 가운데 무엇을 이어 붙일 수 있는가”에서 시작해야 한다.

현장 적용성이 높은 과제는 대체로 세 가지 유형으로 나뉜다. 첫째는 반복 보고 자동화형이다. Python이나 VBA로 문서와 표를 정리하고, 그 결과를 조회 화면으로 연결한다. 둘째는 신청·승인형이다. 입력 폼, 상태 변경, 권한 분리, 결과 조회가 핵심이다. 셋째는 지식 정리형이다. 자료 등록, 검색, 댓글 또는 메모, 사용자 구분이 주요 기능이 된다. 이 세 유형은 공공조직에서 가장 흔한 업무 흐름이므로 교육 종료 후 실제 확산 가능성이 높다.

발표보다 더 중요한 검수 문장

종합 실습에서 발표를 준비할 때는 시연 순서만 짜지 말고, 검수 문장도 함께 써 두는 것이 좋다. 예를 들어 “비로그인 상태에서는 조회만 가능하다”, “신청자는 자기 건만 수정할 수 있다”, “관리자는 로그를 확인할 수 있다”, “실제 운영 전에는 샘플 데이터만 사용했다” 같은 문장은 서비스의 완성도를 높여 보이게 한다. 공공부문에서는 기술적 화려함보다 이런 통제 문장이 더 설득력 있게 들리는 경우가 많다.

통합 서비스 평가 기준

종합 과제가 좋은지 판단하는 기준은 네 가지다. 첫째, 문제와 사용자가 분명한가. 둘째, 입력에서 결과까지 실제로 이어지는가. 셋째, 권한과 운영 메모가 있는가. 넷째, 다음 단계 확장 계획이 현실적인가. 이 네 기준이 충족되면 장수가 많지 않아도 좋은 산출물이고, 반대로 기능은 많아도 이 네 기준이 비어 있으면 데모에 머문다.

통합 워크플로우 사례 1. 보고 자동화에서 조회 대시보드까지

가장 교육 효과가 높은 종합 과제는 새롭고 거대한 문제보다, 이미 모든 부서가 겪고 있는 반복 업무를 끝까지 연결하는 과제다. 예를 들어 각 팀이 매주 제출하는 CSV를 통합해 총괄 대시보드로 보여 주는 과제를 생각해 보자. Day 1에서는 Python이 폴더 안의 CSV를 읽어 구조를 통일하고, 누락 팀을 찾아 로그를 남기며, 집계 엑셀과 요약 CSV를 만든다. 이 단계에서 핵심은 “파일을 읽는다”가 아니라 “어떤 파일이 빠졌는지까지 알려 준다”는 점이다. 그래야 총괄 담당자의 확인 노동이 실제로 줄어든다.

Day 2에서는 이 결과를 단일 HTML 또는 Lovable 기반 조회 화면으로 옮긴다. 여기서 입력 데이터는 이미 정리된 CSV 또는 테이블이므로, 화면은 목록, 필터, 요약 카드, 차트처럼 읽기 기능에 집중할 수 있다. 이 구조의 장점은 구현 위험이 낮다는 데 있다. 신청과 승인, 계정 관리까지 한 번에 다루지 않아도 되므로, 2일 교육 안에서도 끝까지 연결된 서비스처럼 보이는 결과를 만들 수 있다. 또한 실무 확산도 빠르다. 처음에는 총괄자 혼자 보는 대시보드로 시작하고, 필요하면 이후 팀장 열람 권한을 붙이는 식으로 확장할 수 있기 때문이다.

이 과제를 발표할 때는 “우리가 AI로 대시보드를 만들었다”보다 “기존에는 월 4회, 회당 2시간 걸리던 파일 통합 업무를 자동화해 총 8시간을 줄였다”처럼 말하는 편이 훨씬 설득력이 있다. 공공조직은 기술 데모보다 시간, 누락 감소, 보고 일관성 같은 지표에 더 잘 반응한다. 따라서 종합 과제의 발표 문장도 기술 중심보다 업무 효과 중심으로 설계해야 한다.

통합 워크플로우 사례 2. 신청서에서 승인 화면까지

두 번째 대표 과제는 신청·승인형 서비스다. 예를 들어 회의실 예약, 교육 신청, 간단한 물품 요청처럼 신청자와 승인자가 분명한 업무가 있다. 이 과제는 Day 1에서 입력 양식과 상태값을 정리하는 것으로 시작한다. 신청자는 어떤 필드를 입력해야 하는가, 승인자는 무엇을 확인하고 어떤 상태로 바꾸는가, 관리자만 볼 수 있는 정보는 무엇인가를 먼저 표로 적는다. 이 단계가 불분명하면 이후 어떤 도구를 써도 권한 설계가 흔들린다.

Day 2에서는 Lovable과 Supabase 조합이 특히 잘 맞는다. 신청 폼, 내 신청 목록, 승인 대기 목록, 상태 변경 화면을 하나의 흐름으로 구성하고, RLS로 사용자별 접근을 나눈다. 여기서 중요한 것은 예쁜 폼을 만드는 것이 아니라 상태값 설계를 단순하게 유지하는 것이다. 예를 들어 draft, submitted, approved, rejected 정도로 시작하면 충분한데, 처음부터 예외 상태를 지나치게 많이 만들면 교육 시간 안에 흐름이 무너진다. 따라서 종합 과제에서는 권한과 상태를 최소 단위로 시작하고, 운영 확장 단계에서 세분화하는 편이 안전하다.

이 사례가 좋은 이유는 C06의 거의 모든 개념을 실제로 묶어 주기 때문이다. 프롬프트 설계가 필요하고, 데이터 모델링이 필요하며, 인증과 권한 분리가 필요하고, README와 배포도 필요하다. 즉 “노코드 AI 서비스 구현”이라는 과목명이 왜 단순 화면 생성이 아닌지 가장 잘 보여 주는 과제다. 발표할 때도 “신청 화면”만 보여 주면 부족하고, 반드시 “일반 사용자 화면”, “승인자 화면”, “관리자 관점의 운영 포인트”를 함께 설명해야 완성도가 살아난다.

통합 워크플로우 사례 3. 지식 정리 서비스와 검색형 업무 보조

세 번째 사례는 지식 정리형 서비스다. 부서에서 자주 쓰는 양식, 링크, 규정, FAQ, 업무 메모를 한곳에 저장하고 검색하는 도구는 생각보다 파급력이 크다. 많은 공공조직에서 업무 적응 속도를 늦추는 원인은 시스템이 없어서가 아니라, 필요한 문서와 링크가 흩어져 있기 때문이다. 이 과제는 초기에는 localStorage 또는 IndexedDB 기반 단일 HTML로도 충분히 시작할 수 있고, 여러 사람이 함께 쓰기 시작하면 Supabase나 다른 DB 구조로 확장할 수 있다.

이 유형의 장점은 사용자가 금방 가치를 느낀다는 점이다. 신입 직원은 자주 쓰는 링크와 양식을 찾는 시간이 줄고, 기존 직원은 반복 설명을 덜 하게 된다. 또 태그와 메모 구조를 잘 설계하면 단순 즐겨찾기를 넘어 업무 지식 맵이 생긴다. 이 과정에서 LLM은 문서 요약, 제목 추천, 태그 초안 생성처럼 보조 역할을 맡을 수 있다. 다만 최종 등록과 공개 범위는 사람이 결정해야 한다. 특히 규정이나 민감한 내부 지침이 섞여 있을 수 있으므로, 어떤 항목이 공개 가능한지와 어떤 항목은 내부 전용인지 구분하는 운영 기준이 필요하다.

발표에서는 “자료를 모아 두었다”보다 “신입이 첫 주에 꼭 찾는 20개 자료를 3분 안에 찾을 수 있게 했다”는 식의 사용자 관점 설명이 더 좋다. 이 과제는 화려한 화면 없이도 실무 효용을 강하게 증명할 수 있다.

종합 과제를 끝까지 밀어붙이는 발표 시나리오

종합 과제에서 자주 벌어지는 문제는 구현 자체보다 설명 부족이다. 팀은 열심히 만들었지만, 왜 이 도구 조합을 골랐는지, 어떤 권한 구조를 가정했는지, 실제 운영에서는 무엇을 더 정리해야 하는지를 말하지 못한다. 이를 막으려면 발표를 다음 순서로 준비하는 것이 좋다.

  1. 문제를 한 문장으로 말한다.
    예: 매주 부서별 CSV 통합과 누락 확인에 2시간이 걸린다.
  2. 사용자를 말한다.
    예: 각 팀 주무관이 입력하고 총괄 담당자가 확인한다.
  3. 입력과 출력을 말한다.
    예: CSV 입력, 통합 엑셀과 조회 대시보드 출력.
  4. 왜 이 도구 조합을 골랐는지 말한다.
    예: Python은 파일 통합에, HTML은 조회에 적합했다.
  5. 운영 전환 전 남은 과제를 말한다.
    예: 관리자 계정과 배포 승인 절차를 추가로 정리해야 한다.

이 다섯 줄만 분명해도 시연은 훨씬 교재다운 산출물처럼 보인다.

2일 종합 워크숍을 실제로 운영하는 시간표

종합 과제가 얇아지는 가장 큰 이유는 시간이 부족해서가 아니라, 시간을 어디에 써야 하는지 합의가 없어서다. Day 1 오전은 반드시 문제 정의와 입력 구조 정리에 써야 한다. 이 시간에 문제를 좁히지 못하면 Day 2까지 가도 화면만 남는다. Day 1 오후는 데이터 처리 또는 단일 HTML 초안처럼 가장 작은 기능을 끝까지 연결하는 데 써야 한다. Day 2 오전은 인증, 권한, 화면 보강, 배포 준비에 쓰고, Day 2 오후는 README, 발표 문장, 운영 메모를 정리하는 데 써야 한다.

이 시간표를 지키면 종합 과제는 "기능 수가 많은 과제"보다 "끝까지 설명 가능한 과제"가 된다. 공공 실무 교육에서는 후자가 훨씬 중요하다. 왜냐하면 실제 확산은 기능 갯수보다 문서와 설명 가능성에 더 크게 좌우되기 때문이다.

팀 역할 분담을 미리 정하면 완성도가 달라진다

팀 실습에서는 최소 네 역할을 두는 편이 좋다. 첫째는 문제 정의 담당이다. 사용자, 입력, 출력, 상태값을 정리한다. 둘째는 구현 담당이다. Python, HTML, Lovable 같은 실제 구현을 밀어붙인다. 셋째는 검수 담당이다. 버튼, 문구, 권한 흐름, 오류를 점검한다. 넷째는 문서 담당이다. README, 발표 문장, 운영 메모를 정리한다. 이 네 역할이 있으면 팀 전체가 구현에만 몰리지 않고, 운영과 발표까지 자연스럽게 챙기게 된다.

많은 팀이 구현 담당 한 명에게 모든 것을 몰아주는데, 그러면 시연은 그럴듯해도 설명 문서가 비기 쉽다. 반대로 문서 담당이 따로 있으면, 기능이 조금 적더라도 교재다운 결과물이 나온다. C06이 원하는 것은 바로 이런 완결성이다.

종합 워크숍 검수 질문 10개

  2. 입력 데이터는 어디서 오고 샘플로 대체 가능한가
  3. 서비스가 없는 현재 절차는 무엇인가
  4. 가장 먼저 줄어드는 수작업은 무엇인가

  7. 결과물이 저장되는 위치는 어디인가

이 질문에 답할 수 있으면 종합 과제는 상당히 성숙한 수준으로 올라온다.

5분 발표 구조 예시

종합 워크숍 발표는 기능 시연보다 문제 정의와 운영 관점을 먼저 보여 주는 편이 좋다. 첫 1분에는 누구의 어떤 불편을 줄이려 했는지 설명하고, 다음 2분에는 실제 동작 흐름을 보여 준다. 그다음 1분에는 왜 이 도구 조합을 골랐는지 말하고, 마지막 1분에는 권한, 유지보수, 다음 확장 계획을 설명한다. 이 구조를 따르면 "신기한 데모"가 아니라 "실무에 올릴 수 있는 서비스"로 인식되기 쉽다.

종합 과제에서 자주 발생하는 실패

  • 문제 범위가 너무 커서 핵심 기능이 끝까지 연결되지 않는다
  • 로그인은 붙였지만 권한 분리가 없다
  • 자동화는 되지만 결과 검수 단계가 없다
  • 배포 URL은 있으나 README와 인수인계 문서가 비어 있다

따라서 M12의 핵심은 기능 추가가 아니라 완결성이다. 적더라도 끝까지 연결된 기능, 설명 가능한 권한, 남겨진 문서가 있는 팀이 더 좋은 결과를 만든다.

Section 16. 운영·보안·거버넌스 체크리스트

교육 현장에서 결과물을 만들었다고 해서 바로 운영에 넣을 수 있는 것은 아니다. 특히 공공부문은 개인정보, 공문, 민원, 내부 규정, 예산 정보처럼 설명 책임이 큰 데이터를 다루므로, 기술 시연과 운영 투입 사이에 반드시 점검 단계가 필요하다. 이 장은 바로 그 점검을 위한 최소 기준을 정리한다.

  • 관리자 화면은 일반 사용자 화면과 분리하고, 최소 권한 원칙을 적용한다.

운영 전환 워크플로우를 단계별로 보면

교육 산출물을 실제 도구로 넘길 때는 보통 다섯 단계를 거친다. 첫 단계는 사용성 점검이다. 실제 사용자가 막히는 지점을 찾고, 화면 문구를 바꾸고, 입력 오류를 줄인다. 둘째는 데이터 점검이다. 샘플 데이터와 실제 데이터를 분리하고, 내보내기 파일 형식을 확정하며, 저장 위치를 정리한다. 셋째는 권한 점검이다. 일반 사용자, 관리자, 검토자 역할을 최소한으로라도 나누고, 누가 무엇을 볼 수 있는지 문서화한다. 넷째는 배포 점검이다. 공개 URL인지 내부 URL인지, 롤백 경로가 있는지, 환경변수가 안전한지 확인한다. 다섯째는 인수인계 점검이다. README, 운영 메모, 테스트 기록, 변경 로그를 남긴다.

이 다섯 단계는 거창한 DevOps 절차가 아니라, 공공 실무에서 가장 자주 빠지는 구멍을 메우는 순서다. 대부분의 실패는 기술 부족보다 이 다섯 단계를 건너뛰는 데서 나온다. 즉 운영 전환은 배포 버튼을 누르는 순간이 아니라, 이 다섯 점검이 끝나는 순간이라고 보는 편이 더 정확하다.

실제 운영 투입 전 2주 점검 시나리오

첫째 주에는 실사용자 2~3명을 선정해 실제 업무처럼 써 보게 한다. 이때는 기능을 더 넣지 말고 막히는 지점을 기록하는 데 집중한다. 둘째 주에는 기록된 문제를 우선순위별로 고친다. 예를 들어 버튼 위치가 헷갈렸는지, 상태 이름이 모호했는지, 관리자만 봐야 할 정보가 섞였는지, 모바일에서 레이아웃이 깨지는지 등을 점검한다. 이 두 주만 제대로 보내도 교육 결과물의 품질은 크게 달라진다.

실무에서는 자꾸 새 기능을 붙이고 싶어지지만, 교육 종료 직후 2주는 오히려 기능 동결 기간처럼 운영하는 것이 좋다. 왜냐하면 이 시기에 필요한 것은 확장이 아니라 안정화이기 때문이다. 특히 공공조직 도구는 오류가 한 번 나면 신뢰를 잃기 쉽다. 따라서 처음에는 기능 수보다 안정적 사용 경험을 우선하는 편이 장기적으로 더 유리하다.

운영 승인 회의에서 실제로 나오는 질문

상급자나 정보보호 담당자가 운영 승인 전 던지는 질문은 대체로 비슷하다. 이 도구가 정말 필요한가, 기존 절차보다 무엇이 줄어드는가, 실데이터는 어디에 저장되는가, 관리자 권한은 누가 갖는가, 오류가 나면 수동으로 어떻게 대체하는가, 다음 담당자는 누구인가가 핵심이다. 기술 설명만 준비하고 가면 이 질문 앞에서 곧바로 막히기 쉽다. 따라서 승인 회의 전에는 기능 시연보다 이 질문에 대한 문장을 먼저 준비하는 편이 좋다.

예를 들어 "이 도구는 부서별 CSV 통합 시간을 회당 2시간에서 20분으로 줄이기 위한 것임", "실데이터는 내부 드라이브에 저장하고 외부 전송은 하지 않음", "운영 장애 시 기존 수동 통합 절차로 즉시 전환 가능함" 같은 문장은 매우 실무적이다. 공공조직은 기술적 신기함보다 관리 가능성과 책임 범위를 먼저 본다. 그러므로 운영 승인 문장은 짧고 명확할수록 좋다.

장애 발생 첫 24시간 대응 흐름

운영 중 작은 장애가 나면 그 순간의 대응이 신뢰를 좌우한다. 첫째, 현재 증상을 기록한다. 어떤 화면에서 어떤 입력으로 실패했는지 남긴다. 둘째, 직전 성공 버전으로 되돌릴 수 있는지 확인한다. 셋째, 사용자에게 임시 안내 문구를 제공한다. 넷째, 수동 대체 절차를 가동한다. 다섯째, 원인과 조치 결과를 간단한 로그로 남긴다. 이 다섯 단계는 단순하지만 실제 운영팀에게 매우 중요하다.

교육용 산출물은 종종 "장애가 날 정도로 복잡하지 않다"고 생각하기 쉽다. 그러나 작은 도구일수록 담당자가 적고 대응 문서가 비어 있어 오히려 더 취약할 수 있다. 따라서 C06 교재는 큰 서비스뿐 아니라 작은 부서 도구에도 장애 대응 메모가 필요하다는 점을 분명히 적어 두는 편이 맞다.

운영 문서가 실제로 줄여 주는 것

운영 문서는 형식적 산출물이 아니라, 설명 시간을 줄여 주는 장치다. README가 있으면 새 담당자가 구조를 빨리 이해하고, 운영 메모가 있으면 장애 시 우왕좌왕할 가능성이 줄며, 변경 로그가 있으면 왜 지금 화면이 이렇게 생겼는지 설명할 수 있다. 즉 문서는 일을 늘리는 것이 아니라, 반복 설명과 반복 실수를 줄이는 방법이다. 공공부문에서는 담당자 이동이 잦고 승인 절차가 많기 때문에 이 효과가 특히 크다.

다음 담당자가 1시간 안에 이해할 수 있어야 한다

운영 전환의 마지막 기준은 결국 인수인계 가능성이다. 지금 담당자가 모든 맥락을 알고 있으니 잘 돌아가는 것처럼 보여도, 다음 담당자가 1시간 안에 구조를 이해하지 못하면 그 도구는 아직 조직 자산이 아니다. 따라서 실제 인수인계 시나리오를 한 번 상상해 보는 것이 좋다. 새 담당자가 들어와 README를 읽고, 운영 URL을 열고, 관리자 계정을 확인하고, 최근 변경 로그를 보고, 가장 자주 수정하는 화면이나 문구를 찾아볼 수 있어야 한다. 이 다섯 단계를 막힘없이 따라갈 수 있으면 인수인계 품질은 상당히 높다고 볼 수 있다.

반대로 새 담당자가 처음 열자마자 "어디서 시작해야 하지", "이 키는 어디에 있지", "왜 이 상태값이 이렇게 많지", "누가 관리자지" 같은 질문을 던지게 되면, 그 결과물은 아직 개인 프로젝트에 가깝다. 공공조직에서는 기술적 완성도 못지않게 이런 인수인계 가능성이 중요하다. 담당자 이동과 역할 변경이 자주 일어나기 때문이다. 따라서 C06에서 말하는 운영 문서는 단순 첨부물이 아니라, 다음 사람을 위한 진입로라고 이해하는 편이 정확하다.

인수인계 전 마지막 자체 점검

  • 새 담당자에게 먼저 읽어야 할 문서 순서를 적어 두었는가
  • 운영 URL, Preview URL, 로컬 실행 경로가 각각 구분되어 있는가
  • 관리자 계정 발급 방식과 권한 변경 절차가 적혀 있는가
  • 최근 3회 변경 사항과 이유가 로그로 남아 있는가
  • 가장 자주 발생하는 오류와 임시 대응법이 한 문단으로 정리되어 있는가

이 다섯 항목이 정리되어 있으면, 작은 도구라도 조직 안에서 이어질 확률이 크게 높아진다.

운영 전환 후 첫 30일을 어떻게 보낼 것인가

실무에서는 도구를 배포한 뒤 바로 다음 기능을 붙이고 싶어지지만, 실제로 가장 중요한 시기는 첫 30일이다. 이 기간은 기능 확장보다 사용 패턴을 읽는 시기다. 첫 주에는 누가 얼마나 자주 쓰는지, 어디에서 멈추는지, 어떤 문구가 헷갈리는지 기록해야 한다. 둘째 주에는 입력 오류와 상태 표현을 다듬어야 한다. 셋째 주에는 관리자와 승인자의 화면이 정말 필요한 정보만 보여 주는지 다시 정리해야 한다. 넷째 주에는 README, 운영 메모, 자주 묻는 질문을 갱신해 다음 담당자가 이어받을 수 있는 상태로 만들어야 한다.

이 30일을 잘 보내면 작은 도구는 조직 자산으로 자리 잡기 시작한다. 반대로 이 시기에 기록을 남기지 않으면, 한 달 뒤에는 어떤 문제가 반복되었는지 기억에만 의존하게 되고, 수정도 즉흥적으로 이뤄지기 쉽다. 따라서 운영 전환은 배포 시점의 성공 여부보다, 첫 30일 동안 얼마나 차분히 기록하고 보정했는지에 더 크게 좌우된다고 볼 수 있다.

상급자 보고용 1페이지 요약은 어떻게 써야 하는가

운영 승인이나 확산 보고를 할 때는 긴 기술 문서보다 1페이지 요약이 먼저 읽힌다. 이 문서는 다섯 문단이면 충분하다. 첫째, 현재 어떤 수작업 문제가 있는지 적는다. 둘째, 이번 도구가 무엇을 자동화하거나 지원하는지 적는다. 셋째, 누가 쓰고 누가 관리하는지 적는다. 넷째, 데이터 저장 위치와 보안 통제 방식을 적는다. 다섯째, 운영 전 남은 과제를 적는다. 이렇게 쓰면 상급자는 기술 세부를 몰라도 도입 가치와 위험 범위를 빠르게 이해할 수 있다.

예를 들어 보고 자동화 도구라면 "현재 각 부서 CSV 통합에 회당 2시간이 소요됨", "본 도구는 파일 구조 검증과 통합표 생성까지 자동화함", "총괄 담당자 1인이 운영하며 부서별 입력자는 별도 교육 없이 사용 가능함", "실데이터는 내부 공유 드라이브에 저장하고 외부 전송은 하지 않음", "운영 전 권한 분리와 장애 대응 메모 보완이 필요함"처럼 적을 수 있다. 이런 문장은 화려하지 않지만 실제 보고에서 매우 강하다.

확산 전 마지막 자체 점검표

  • 이 도구의 핵심 기능을 처음 사용자에게 3문장 안에 설명할 수 있는가
  • 관리자 부재 시 누가 최소 운영을 이어받을 수 있는가
  • 오류가 났을 때 사용자가 볼 안내 문구가 준비되어 있는가
  • 실사용자 피드백이 최소 2건 이상 기록되어 있는가
  • 다음 한 달 안에 고칠 우선순위 3개가 정리되어 있는가

이 체크리스트를 통과하면, 교육 결과물은 적어도 조직 안에서 "한 번 더 써 볼 만한 도구" 단계로 올라선다. 공공조직의 확산은 대개 거창한 론칭보다, 이런 작은 신뢰의 축적에서 시작된다.

운영 책임자를 위한 점검 문장 예시

  • 현재 버전은 샘플 데이터 기준으로 검증되었고, 실데이터 전환 전 추가 검토가 필요하다
  • 일반 사용자와 관리자 화면은 분리되었으나 관리자 계정 발급 절차는 별도 문서화 필요
  • 배포 URL은 검토용이며, 운영 URL 전환 전 최종 승인 필요
  • 장애 발생 시 이전 성공 배포로 즉시 복귀 가능하도록 경로를 확보한다
  • README와 운영 메모를 기준 문서로 삼고, 변경 시 반드시 로그를 남김

이 문장들은 발표를 그럴듯하게 꾸미기 위한 표현이 아니라, 실제로 운영 승인과 보고서 문안에 바로 넣을 수 있는 표현들이다. C06 교재는 이런 문장을 직접 제공해야 실무 활용도가 높아진다.

“작게 시작하고 남기기”가 결국 가장 빠르다

많은 팀이 처음부터 넓게 만들려다가 끝내 아무것도 운영에 올리지 못한다. 반대로 작은 자동화 하나, 작은 HTML 도구 하나, 작은 승인 흐름 하나를 끝까지 정리한 팀은 실제 부서에 훨씬 빨리 확산된다. 이유는 단순하다. 작을수록 설명이 쉽고, 수정이 빠르며, 책임 범위도 분명하기 때문이다. 따라서 운영 전환의 핵심 전략은 “크게 만들기”가 아니라 “작게 완결하기”다. 이 문장은 C06 전체를 관통하는 결론이기도 하다.

확산 전략

좋은 도구를 만들었다면 바로 전 조직에 확산하려 하지 말고, 가장 가까운 팀에서 작게 써 본 뒤 피드백을 반영하는 것이 좋다. 반복적으로 수정된 도구만이 표준이 된다. 따라서 교육 종료 후 첫 4주는 기능 추가보다 사용자 피드백, 오류 기록, README 보완, 권한 정리에 집중하는 편이 더 생산적이다.

도입 실패 패턴을 미리 아는 것이 중요한 이유

공공조직에서 AI 도구 도입이 실패하는 이유는 기술 부족보다 운영 과신에 있는 경우가 많다. 첫 번째 패턴은 시연 성공을 운영 성공으로 착각하는 것이다. 교육장에서는 잘 돌아갔지만, 실제 데이터와 실제 사용자, 실제 승인 절차가 들어오면 곧바로 문제가 생긴다. 두 번째 패턴은 만든 사람이 모든 맥락을 알고 있다는 사실을 잊는 것이다. 문서화가 없으면 담당자가 바뀌는 순간 서비스 품질이 급격히 떨어진다. 세 번째 패턴은 작은 도구의 한계를 인정하지 않는 것이다. localStorage 기반 도구를 공유 시스템처럼 쓰려 하거나, 단일 HTML을 장기 운영 시스템처럼 다루면 반드시 무리가 온다.

네 번째 패턴은 권한을 나중 문제로 미루는 것이다. 화면과 기능은 완성됐는데, 누가 무엇을 볼 수 있는지 정하지 않으면 결국 운영 투입이 지연된다. 다섯 번째 패턴은 로그와 롤백 절차가 없는 상태로 공개하는 것이다. 작은 조직일수록 장애 대응 문서가 없어도 될 것 같지만, 실제로는 사람이 적을수록 문서가 더 필요하다. 여섯 번째 패턴은 도구 선택을 유행 중심으로 하는 것이다. 조직에 맞는 최소한의 구조보다 최신 브랜드를 우선하면, 몇 달 뒤 유지보수 비용이 더 커질 수 있다.

30일, 60일, 90일 실행 로드맵

과정 이후 확산은 한 번의 배포로 끝나지 않는다. 첫 30일은 안정화 단계다. 오류를 모으고 README를 보완하며 실제 사용자 반응을 기록한다. 다음 60일은 구조 정리 단계다. 권한 분리, 입력 검증, 관리자 화면, 로그 기준을 붙인다. 90일 시점부터는 표준화 단계로 들어간다. 부서 내 다른 팀이 재사용할 수 있게 템플릿화하고, 운영 승인과 인수인계 문서를 정리한다. 이 로드맵을 갖고 있으면 교육 결과물이 “한 번 해 본 실습”에서 “반복 가능한 자산”으로 바뀐다.

운영 승인 전 최종 점검 질문

  • 이 도구가 없을 때의 수동 절차가 아직 남아 있는가
  • 실제 개인정보와 샘플 데이터가 명확히 분리되어 있는가
  • 관리자 권한과 일반 권한이 화면과 데이터 양쪽에서 나뉘는가
  • 배포 후 되돌릴 수 있는 이전 버전이 존재하는가
  • 다음 담당자가 1시간 안에 구조를 이해할 수 있는가

이 다섯 질문에 답할 수 없다면, 아직은 운영보다 추가 정리가 먼저다.

Section 17. 예상 질문과 답변

Q1. 코드를 잘 몰라도 정말 여기까지 만들 수 있는가

가능하다. 다만 전제는 코드를 몰라도 된다는 뜻이 아니라, 코드를 처음부터 직접 작성하지 않아도 된다는 뜻이다. 이 과정에서 필요한 역량은 문법 암기보다 요구사항을 구조화하고 결과를 검수하는 능력이다. 실제로 작은 자동화와 단일 HTML, VBA, 노코드 풀스택은 이 능력만으로도 충분히 시작할 수 있다.

Q2. 폐쇄망 조직인데 외부 LLM을 쓰면 위험하지 않은가

위험은 무엇을 보내느냐에 달려 있다. 실제 민감정보를 외부에 넣는 것은 위험하지만, 파일 구조와 샘플, 요구사항만 설명하고 코드를 받아 내부에서 실행하는 방식은 위험 범위를 크게 줄일 수 있다. 그래서 이 과정은 처음부터 데이터와 지식을 분리하는 비대칭 구조를 훈련한다.

Q3. Python과 VBA 가운데 무엇부터 시작하는 것이 좋은가

설치와 운영 자유도가 있으면 Python이 범용성이 높다. 반대로 조직 환경이 엄격하고 Office 중심 업무가 많다면 VBA가 더 빨리 효과를 낸다. 어느 쪽이 더 우월한지가 아니라, 현재 조직에서 가장 적은 저항으로 돌아가는 방식이 무엇인지가 기준이다.

Q4. localStorage로 만든 도구를 공식 업무에 써도 되는가

개인용 또는 임시 보조도구 수준에서는 가능하지만, 공식 기록 시스템으로 쓰기에는 부적합하다. 브라우저 저장소는 삭제, 브라우저 변경, PC 교체에 취약하기 때문이다. 따라서 공유와 장기 보관이 필요한 시점부터는 데이터베이스 기반 구조로 전환해야 한다.

Q5. 로컬 SLM은 성능이 낮은데 왜 배워야 하는가

로컬 SLM은 최고 성능을 위한 도구가 아니라, 외부 API 호출이 어려운 환경에서 통제 가능한 AI 활용 실험을 지속하기 위한 도구다. 또한 작은 모델을 다뤄 보면 프롬프트와 검수의 중요성을 더 잘 이해하게 된다. 이것은 이후 외부 고성능 모델을 쓸 때도 그대로 도움이 된다.

Q6. Lovable 같은 도구로 만든 앱을 실제 운영에 올려도 되는가

가능하지만, 화면이 만들어졌다는 사실만으로 운영 준비가 끝난 것은 아니다. 계정 정책, 권한 분리, 로그 기준, 개인정보 문구, 관리자 책임자, README 같은 운영 요소를 함께 정리해야 한다. 즉 빠른 개발과 안전한 운영은 다른 층위의 문제라는 점을 기억해야 한다.

Q7. GitHub를 꼭 써야 하는가

혼자 잠깐 쓰는 파일이라면 꼭 필요하지 않을 수도 있다. 그러나 부서 자산으로 남기려면 사실상 필요하다. 저장소가 있어야 이전 버전을 되돌릴 수 있고, 담당자가 바뀌어도 구조를 이어받을 수 있으며, 배포와 문서화를 체계적으로 관리할 수 있다.

Q8. 도구가 너무 많아 무엇을 선택해야 할지 모르겠다

문제의 크기를 기준으로 줄이면 된다. 혼자 쓰는 계산기라면 단일 HTML, 문서와 파일 자동화라면 Python 또는 VBA, 내부망 AI 보조라면 Ollama, 여러 사용자가 함께 쓰는 서비스라면 Lovable과 데이터베이스 구조를 생각하면 된다. 도구를 먼저 고르지 말고 문제의 범위와 운영 범위를 먼저 고르는 것이 순서다.

Q9. 구현은 했는데 다음 담당자에게 넘기기 어렵다

이것은 코드 문제라기보다 문서 문제인 경우가 많다. 실행 방법, 필수 계정, 수정 위치, 자주 나는 오류, 관리자 연락 포인트를 README에 적어 두면 인수인계 난도가 크게 내려간다. 따라서 과정 종료 후 첫 번째 후속 작업은 기능 추가가 아니라 문서화다.

Q10. 과정 이후 가장 먼저 해야 할 일은 무엇인가

가장 좋은 다음 행동은 작은 실제 사용자 테스트다. 동료 2~3명에게 도구를 써 보게 하고, 막히는 지점과 필요한 기능을 기록한 뒤 우선순위를 정한다. 그다음에 권한과 README를 보완하면, 교육용 결과물이 조직용 도구로 바뀌기 시작한다.

Q11. 시연용 도구와 운영용 도구의 경계는 어디에서 갈리는가

가장 큰 차이는 기능 수가 아니라 책임 구조에 있다. 시연용 도구는 잘 동작하는 한 번의 흐름을 보여 주면 충분한 경우가 많다. 반면 운영용 도구는 누가 쓰는지, 누가 고치는지, 오류가 났을 때 어떻게 되돌리는지, 어떤 데이터가 남는지까지 설명할 수 있어야 한다. 즉 운영용 여부는 화면 화려함보다 권한, 로그, 문서, 복구 경로가 있는지에서 갈린다.

실무적으로는 다음 질문에 답할 수 있으면 운영 후보로 볼 수 있다. 첫째, 실제 사용자와 관리자 역할이 나뉘는가. 둘째, 저장 위치와 보관 기준이 정리되어 있는가. 셋째, 장애 시 수동 대체 절차가 있는가. 넷째, 다음 담당자가 1시간 안에 구조를 이해할 수 있는가. 이 네 가지가 비어 있으면 결과물이 아무리 좋아 보여도 아직은 시연용에 가깝다.

Q12. 외부 LLM과 로컬 모델을 함께 쓰는 혼합 전략도 가능한가

가능할 뿐 아니라, 공공부문에서는 오히려 그 방식이 현실적인 경우가 많다. 예를 들어 공개 가능한 자료를 바탕으로 보고서 초안을 잡거나 복잡한 추론이 필요할 때는 외부 고성능 LLM을 쓰고, 내부 분류나 문장 정리, 짧은 질의응답처럼 민감도가 높거나 폐쇄망 적합성이 중요한 작업은 로컬 SLM을 쓰는 식으로 나눌 수 있다. 중요한 것은 "어느 모델이 최고인가"가 아니라 "어떤 업무를 어디까지 맡길 것인가"를 분리하는 일이다.

이 혼합 전략을 쓰면 품질과 통제를 동시에 잡기 쉬워진다. 다만 운영 문서에는 반드시 역할 분담을 적어 두어야 한다. 어떤 단계에서 외부 호출이 일어나는지, 어떤 데이터는 절대 밖으로 나가지 않는지, 로컬 모델의 한계는 무엇인지가 문장으로 남아 있어야 한다. 그래야 기술 선택이 투명해지고, 나중에 보안 검토나 사업 문서 작성 시에도 설명이 가능하다.

Q13. 작은 자동화 도구를 RFP 언어로 바꾸려면 어떻게 해야 하는가

작은 도구를 RFP로 옮긴다는 것은 기능을 부풀리는 일이 아니다. 오히려 현재 도구가 줄여 주는 업무 부담을 공식 문장으로 번역하는 일에 가깝다. 예를 들어 CSV 통합 도구가 있다면, 이를 "월간 보고자료 취합 및 형식 검증 자동화 기능"처럼 적을 수 있다. 단일 HTML 신청 도구가 있다면, "간단한 신청 접수 및 상태 확인용 웹 기반 사용자 인터페이스"라고 정리할 수 있다. 즉 이미 만든 것을 더 크게 포장하는 것이 아니라, 어떤 문제를 어떤 범위에서 해결하는지 행정 언어로 재기술하는 것이다.

RFP 전환에서 특히 중요한 것은 범위와 단계다. 지금 당장 구현된 것은 무엇이고, 다음 단계에서 추가되어야 할 것은 무엇인지 구분해 적어야 한다. 그래야 시연본과 발주 문서 사이에 불필요한 과장이 줄어든다. 공공사업 문서에서 가장 위험한 것은 "가능할 것 같다"를 "이미 준비되었다"처럼 쓰는 일이다. 따라서 작은 도구일수록 현재 수준과 확장 계획을 분리해 적는 태도가 중요하다.

Q14. 운영 승인 전에 꼭 남겨야 할 로그와 문서는 무엇인가

최소 단위로 보면 네 가지면 충분하다. README, 운영 메모, 사용자 테스트 기록, 변경 로그다. README는 목적과 실행법을 설명하고, 운영 메모는 URL과 관리자, 계정과 수동 대체 절차를 남긴다. 사용자 테스트 기록은 실제 사용자가 어디서 막혔는지 보여 주고, 변경 로그는 무엇을 왜 고쳤는지 남긴다. 이 네 문서가 있으면 작은 도구라도 설명 가능성과 재현 가능성이 크게 올라간다.

로그는 기능마다 거창할 필요가 없다. 누가 언제 무엇을 바꿨는지, 어떤 오류가 났는지, 직전 성공 버전이 무엇인지 정도만 남아도 충분히 유용하다. 공공부문에서는 도구의 규모와 무관하게 "나중에 설명할 수 있는가"가 중요하기 때문에, 문서와 로그는 사후 장식이 아니라 도입 조건에 가깝다.

Q15. 2일 교육 결과물을 실제 부서 프로젝트로 키울 때 가장 좋은 순서는 무엇인가

첫 단계는 작은 반복 사용이다. 한 명 또는 한 팀이 실제로 반복해서 써 보며 막히는 지점을 기록해야 한다. 두 번째 단계는 안정화다. 문구 수정, 권한 분리, 입력 검증, README 보완처럼 기능보다 사용성을 먼저 정리한다. 세 번째 단계는 운영 승인 준비다. 저장 위치, 관리자, 수동 대체 절차, 로그 기준을 문서로 정리한다. 네 번째 단계에서야 다른 팀으로 확산하거나 공식 사업화, RFP 전환을 검토하는 것이 좋다.

많은 팀이 교육 직후 곧바로 전면 확산을 떠올리지만, 공공조직에서는 작은 성공을 몇 번 반복하는 쪽이 훨씬 안전하다. 한 번이라도 실제 사용자 피드백을 반영한 도구는 설득력이 달라진다. 따라서 과정 이후의 최선의 성장 경로는 "작게 사용하고, 기록하고, 고치고, 그다음 넓힌다"로 요약할 수 있다.

Section 18. 부록 A. 프롬프트 패턴

이 부록은 각 도구별로 바로 써 볼 수 있는 프롬프트 패턴을 정리한 것이다. 중요한 것은 문장을 그대로 외우는 것이 아니라, 어떤 항목을 빼먹지 말아야 하는지 이해하는 데 있다.

1. Python 자동화 프롬프트 패턴

다음 요소를 포함해 요청하면 좋다.

  • 목적: 한 폴더의 CSV 파일들을 하나의 엑셀 보고서로 통합하는 Python 스크립트 작성
  • 환경: Windows, Python 3.x, pandas 사용 가능
  • 입력 폴더: ./input
  • 파일 구조: 날짜, 부서명, 항목, 금액
  • 원하는 출력: 월별 통합 시트 1개, 부서별 합계 시트 1개, 누락 파일 목록 로그 1개
  • 주의사항: 실제 데이터는 외부로 보내지 않으므로 샘플 구조만 기준으로 작성하고, 함수 단위로 나누어 오류 수정이 쉽게 해 달라고 요청

2. 단일 HTML 프롬프트 패턴

다음 요소를 포함해 요청하면 좋다.

  • 목적: 부서 내부에서 더블클릭으로 바로 실행 가능한 출장비 계산기 HTML 파일 생성
  • 입력: 출장 일수, 교통수단, 직급, 숙박 여부
  • 출력: 항목별 금액 표, 합계, 인쇄 버튼
  • 디자인: 공공기관 문서처럼 단정한 톤
  • 주의사항: HTML, CSS, JavaScript를 한 파일에 넣고 외부 의존성 없이 작동하도록 요청

3. localStorage 프롬프트 패턴

다음 요소를 포함해 요청하면 좋다.

  • 대상: 단일 HTML 메모 앱
  • 요구사항: 입력 즉시 자동 저장, 페이지 재방문 시 복원, 개별 삭제, 전체 초기화, JSON 내보내기, JSON 불러오기
  • 데이터 항목: 제목, 내용, 작성일, 완료 여부

4. VBA 프롬프트 패턴

다음 요소를 포함해 요청하면 좋다.

  • 기준 파일 구조: 시트명은 원본데이터
  • 컬럼: 부서명, 접수일, 상태, 처리일, 담당자
  • 원하는 결과: 요약 시트에 부서별 건수와 처리율, 상태별 막대차트 생성
  • 조건: 실제 데이터는 제공하지 않고 구조만 설명한다는 전제로 작성
  • 출력 형식: 엑셀 VBA 모듈에 바로 붙여넣을 수 있는 코드

5. Ollama 프롬프트 패턴

다음 요소를 포함해 요청하면 좋다.

  • 과업: 로컬 SLM이 민원 문장을 4개 분류 중 하나로 분류하도록 프롬프트 설계
  • 분류 기준: 교통, 환경, 복지, 기타
  • 출력 형식: JSON 한 줄
  • 보정 조건: 모델이 모르면 기타로 분류하고 설명은 20자 이내로 제한

6. Lovable 프롬프트 패턴

다음 요소를 포함해 요청하면 좋다.

  • 대상 서비스: 부서 내부 게시판 웹앱
  • 필수 기능: 회원가입, 로그인, 글 작성, 댓글, 검색
  • 권한: 로그인한 사용자만 글 작성 가능, 작성자는 본인 글만 수정 가능, 관리자는 모든 글 삭제 가능
  • 추가 요청: GitHub 동기화를 고려해 구조를 명확히 하고 README 초안에 들어갈 핵심 기능 목록도 함께 정리해 달라고 요청

Section 19. 부록 B. 실행 워크시트와 4주 계획

Section 20. 부록 C. 서비스 설계 캔버스

교육 현장에서는 구현 자체가 성과처럼 느껴지지만, 실제 조직 확산은 설계 캔버스가 있을 때 훨씬 빨라진다. 서비스 설계 캔버스는 거창한 기획서를 대신하는 문서가 아니라, 누구의 어떤 문제를 어떤 데이터와 어떤 권한 구조로 풀 것인지 한 장에 정리하는 작업 도구다. 비개발 실무자가 AI와 함께 일할 때 특히 유용한 이유는, 프롬프트로 흩어진 요구사항을 구조화된 문장으로 되돌려 주기 때문이다. 캔버스가 있으면 코딩 에이전트나 노코드 도구에 무엇을 요청해야 할지 선명해지고, 이후 개발자와 협업할 때도 설명 비용이 크게 줄어든다.

다음 캔버스는 C06 전 과정을 실제 업무 전환 문서로 이어 주는 최소 형식이다.

캔버스를 쓰는 올바른 순서

첫째, 문제와 사용자를 먼저 적는다. 둘째, 입력과 출력 구조를 적는다. 셋째, 권한과 운영 방식을 적는다. 넷째, 마지막에야 도구를 고른다. 많은 팀이 이 순서를 거꾸로 밟는다. 먼저 Lovable을 쓸지, Python을 쓸지, VBA를 쓸지 정한 뒤 문제를 끼워 맞춘다. 그러나 실제로는 문제와 운영 범위가 정리되고 나면 도구 선택은 꽤 자연스럽게 좁혀진다. 이 순서를 지키면 유행하는 도구보다 적합한 도구를 고르게 된다.

예시 1. 보고 자동화형 캔버스

보고 자동화형 과제는 가장 현실적인 첫 적용 사례다. 예를 들어 각 부서가 보내는 주간 CSV를 모아 통합 엑셀과 간단한 웹 조회 화면을 만드는 경우를 보자. 입력은 부서별 CSV와 제출 시점이고, 출력은 통합표, 누락 부서 목록, 요약 차트다. 작성 권한은 각 부서에 있고, 조회 권한은 총괄부서와 관리자에게 있다. 이 경우 Day 1의 Python 자동화와 Day 2의 정적 HTML 또는 Lovable 조회 화면을 조합하면 비교적 짧은 시간 안에 실제 업무와 유사한 결과가 나온다.

이 캔버스의 장점은 기술 선택도 분명하게 만든다는 데 있다. 입력 데이터가 파일 중심이고 승인보다 집계가 핵심이므로, 처음부터 풀스택 데이터베이스를 쓰지 않아도 된다. 먼저 Python으로 통합 로직을 만들고, 그 결과를 파일 기반 화면으로 보여 준 뒤, 필요해지면 데이터베이스로 옮기는 순서가 합리적이다. 즉 캔버스는 과도한 구현을 막는 장치이기도 하다.

예시 2. 신청·승인형 캔버스

회의실 예약, 교육 신청, 간단한 내부 요청처럼 상태 변경과 승인자가 분명한 업무는 신청·승인형 캔버스로 정리하는 편이 좋다. 입력은 신청 폼이고, 출력은 상태 목록과 승인 결과다. 사용자 역할은 신청자, 승인자, 관리자 세 가지로 단순화할 수 있다. 이런 구조에서는 localStorage나 단일 HTML만으로는 곧 한계가 드러난다. 데이터 공유와 권한 분리가 필요하기 때문이다. 따라서 이 유형은 Lovable과 Supabase, 또는 최소한 인증과 저장 구조를 가진 웹앱으로 넘어갈 근거가 분명하다.

또한 신청·승인형 과제는 화면 개수도 자연스럽게 나온다. 신청 화면, 내 신청 목록, 승인 대기 목록, 승인 상세, 관리자 설정 화면 정도가 기본 뼈대가 된다. 이렇게 화면을 먼저 나누면 Stitch 같은 디자인 도구를 쓰기에도 수월하고, 이후 권한 분리도 명확해진다. 캔버스는 결국 화면 구조를 낳는 출발점이다.

예시 3. 지식 정리형 캔버스

부서 내 자주 쓰는 양식, 업무 지침, FAQ, 참고 링크를 구조화하는 과제는 지식 정리형 캔버스로 접근하는 것이 좋다. 입력은 문서, 링크, 태그, 설명 메모이고, 출력은 검색 가능한 목록과 상세 보기다. 처음에는 브라우저 저장소만으로도 충분할 수 있지만, 여러 사람이 함께 쓰거나 댓글과 수정 이력이 필요해지면 데이터베이스 기반으로 전환해야 한다. 이 유형은 빠른 데모와 장기 운영 가능성 사이의 경계가 비교적 선명해, 교육용 과제로 매우 적합하다.

30분 팀 워크숍 진행법

  1. 10분 동안 팀별로 문제와 사용자를 적는다.
  2. 10분 동안 입력, 출력, 권한을 채운다.
  3. 5분 동안 어떤 도구 조합이 맞는지 결정한다.
  4. 5분 동안 실패 위험과 다음 단계 한 줄을 적는다.

이 정도만 해도 종합 워크숍의 완성도가 눈에 띄게 달라진다. 도구 시연만 하던 팀도 서비스 설명이 가능해지고, 반대로 아이디어만 풍부하던 팀도 구현 범위를 줄일 수 있다.

Section 21. 부록 D. 운영 전환 문서 예시

결과물을 조직 자산으로 남기려면 코드보다 문서가 먼저 읽혀야 한다. 운영 전환 문서는 크게 어렵지 않다. 핵심은 다음 담당자가 1시간 안에 구조를 이해하고, 첫 수정 1회를 무리 없이 수행할 수 있게 만드는 것이다. 이를 위해 필요한 문서는 README, 운영 메모, 사용자 테스트 기록, 변경 로그 네 가지다. 이 네 문서가 없으면 교육 결과물은 시연용으로 남기 쉽고, 반대로 이 네 문서가 있으면 기능이 다소 단순해도 조직 자산으로 살아남을 가능성이 높다.

README 최소 구성 예시

README는 화려할 필요가 없다. 다음 다섯 항목이면 충분하다.

  1. 서비스 목적: 어떤 문제를 해결하는 도구인지 한 문단으로 설명한다.
  2. 실행 방법: 로컬 실행인지, 배포 URL인지, 필요한 계정이 무엇인지 적는다.
  3. 핵심 기능: 사용자가 실제로 할 수 있는 동작만 적는다.
  4. 수정 포인트: 자주 바뀌는 파일, 문구, 환경변수 위치를 적는다.
  5. 주의사항: 샘플 데이터 사용 원칙, 권한 기준, 관리자 연락 포인트를 적는다.

README를 이렇게 쓰면 비개발자도 문서를 통해 구조를 파악할 수 있고, 개발자도 첫 진입 장벽을 낮출 수 있다. 중요한 것은 멋진 문장이 아니라, 실제로 다음 작업자가 첫 10분 안에 궁금해할 정보를 먼저 주는 것이다.

운영 메모에 들어가야 할 내용

운영 메모는 README보다 조금 더 실무적이다. 현재 운영 주소, 마지막 성공 배포일, 테스트 계정 존재 여부, 관리자 권한 부여 방법, 장애 시 연락 경로, 수동 대체 절차를 적어 둔다. 특히 공공부문에서는 담당자 이동이 잦을 수 있으므로, 운영 메모가 개인 메신저 대화를 대신할 수 있어야 한다. 예를 들어 “배포 오류 시 우선 Preview에서 재현 여부 확인 후, 최근 성공 배포 URL로 임시 복귀” 같은 문장이 있으면 실제 장애 대응 속도가 크게 달라진다.

사용자 테스트 기록 양식

작은 도구라도 사용자 테스트 기록이 있으면 신뢰도가 올라간다. 복잡한 설문이 필요하지 않다. 누가 언제 어떤 화면을 썼는지, 어디서 막혔는지, 무엇을 고쳐 달라고 했는지 한 표로 남기면 충분하다.

테스트 일시 사용자 유형 수행 과업 막힌 지점 수정 여부
2026-06-10 일반 사용자 신청서 1건 작성 임시저장 위치를 찾지 못한다 수정 완료
2026-06-10 승인자 승인 대기 목록 검토 상태 필터가 모호한다 수정 예정
2026-06-11 관리자 사용자 권한 확인 관리자 메뉴 진입 경로 불명확 수정 완료

이 표가 있으면 기능 추가보다 사용성 수정이 우선인지, 권한 문제가 먼저인지 빠르게 판단할 수 있다. 또한 상급자에게도 “실제 사용자 검증을 거쳤다”는 설명 근거가 생긴다.

변경 로그를 남기는 간단한 규칙

변경 로그는 소프트웨어 개발팀만 쓰는 문서가 아니다. 공공부문 도구라도 어떤 수정이 언제 반영되었는지 남겨야 한다. 다만 형식은 간단해도 된다. 날짜, 수정 내용, 이유, 영향 범위 네 줄이면 충분하다. 예를 들어 “2026-06-12 / 승인자 화면에 상태 필터 추가 / 목록 검수 속도 개선 / 승인 화면만 영향” 정도면 실무에 필요한 맥락이 남는다. 이런 로그가 쌓이면 담당자가 바뀌어도 도구의 변화 방향을 쉽게 파악할 수 있다.

인수인계 메모 예시

인수인계 메모는 다음 사람에게 말로 설명할 내용을 미리 적어 두는 문서다. 권장 항목은 다음과 같다.

  • 이 도구의 주 사용자와 관리자
  • 실제 운영 중인 URL 또는 파일 위치
  • 데이터 입력 주기와 담당 부서
  • 자주 나는 오류와 임시 대응법
  • 당장 고쳐야 할 우선순위 3개

이 다섯 항목만 있어도 다음 담당자는 도구를 처음부터 다시 해석하지 않아도 된다.

운영 승인 전에 상급자에게 보여 줄 한 페이지 요약

실무에서는 기술 문서보다 승인용 요약이 더 빨리 필요해지는 경우가 많다. 이때는 한 페이지에 다음 다섯 줄만 정리해도 충분하다. 무엇을 자동화하는가. 누가 쓰는가. 어떤 데이터가 들어가는가. 어떤 보안 통제가 있는가. 장애 시 어떻게 되돌리는가. 상급자는 대부분 이 다섯 질문에 대한 답을 먼저 원한다. 따라서 운영 전환 문서 세트에는 기술 README와 별개로, 관리자용 요약 1페이지가 함께 있는 편이 좋다.

문서가 많아 보일 때 우선순위를 정하는 법

모든 문서를 완벽하게 쓰려다 시작도 못 하는 경우가 많다. 이때는 우선순위를 다음 순서로 잡으면 된다. 첫째 README, 둘째 운영 메모, 셋째 테스트 기록, 넷째 변경 로그다. 이 네 문서가 돌아가기 시작하면 나머지 문서는 필요에 따라 붙이면 된다. 핵심은 문서량이 아니라 재사용 가능성이다.

노코드 AI 서비스 구현은 개발자를 대체하는 교육이 아니다. 문제를 가장 잘 아는 사람이 가장 작은 구현을 직접 시작할 수 있도록 만들어 주는 교육이다. 공공부문에서 이 역량은 특히 중요하다. 작은 자동화 하나가 팀의 시간을 아끼고, 단일 HTML 하나가 반복 계산을 없애며, 권한이 붙은 웹앱 하나가 부서 협업 구조를 바꾼다. 이 과목의 최종 목표는 최신 도구를 많이 아는 사람이 되는 것이 아니라, 문제를 구조화하고 구현하고 검수하고 운영으로 넘길 수 있는 사람으로 바뀌는 데 있다.
