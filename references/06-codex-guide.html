1. Codex를 일반인 관점에서 한 문장으로 설명하면

Codex는 단순히 답만 해주는 챗봇이 아니라, 파일을 읽고, 수정하고, 터미널 명령도 실행하고, 필요하면 외부 도구까지 연결해서 실제 작업을 수행하는 코딩 에이전트입니다.

즉, 아래가 한 도구 안에 합쳐져 있다고 보면 이해가 쉽습니다.

  • 채팅형 AI
  • 코드 편집기 보조
  • 터미널 자동화 도구
  • 프로젝트 규칙을 기억하는 작업자
  • 외부 서비스와 연결되는 자동화 허브

2. 먼저 꼭 구분해야 하는 7가지

초보자가 가장 많이 헷갈리는 부분입니다.

구분뜻예시쉽게 비유하면
프롬프트그냥 Codex에게 말로 시키는 요청로그인 오류 원인 찾아줘사람에게 부탁하기
슬래시 명령Codex 대화창 안에서 /로 시작하는 제어 명령/status, /review리모컨 버튼
셸 명령운영체제 터미널 명령git status, npm test컴퓨터 자체 명령
CLI 명령터미널에서 codex로 시작하는 명령codex review --uncommittedCodex 프로그램 실행법
설정 파일Codex 기본 동작을 저장해 두는 파일~/.codex/config.toml환경설정 메뉴
AGENTS.md프로젝트별 작업 지침 문서테스트 규칙, 금지사항, 코딩 스타일팀 작업 수칙 메모
MCP / Skills / PluginsCodex 능력을 확장하는 구조Figma 연결, Docs 검색, 재사용 워크플로우확장팩

이 7개만 분리해서 이해해도 절반은 끝납니다.

3. Codex는 어디에서 쓰는가

Codex는 하나의 제품이지만 여러 "표면"에서 작동합니다.

사용 위치핵심 특징추천 대상
CLI터미널 중심, 강력한 제어, 슬래시 명령 풍부개발자, 파워유저
IDE ExtensionVS Code/JetBrains 같은 편집기 안에서 사용코드를 보면서 바로 수정하고 싶은 사람
AppGUI 중심, 작업 관리와 시각적 흐름이 편함일반 사용자, 팀 단위 사용
Web / Cloud원격 환경에서 긴 작업 수행로컬 PC 성능/환경에 덜 의존하고 싶은 경우

핵심 차이는 이렇습니다.

  • CLI는 "터미널 중심 제어"가 강합니다.
  • IDE 확장은 "지금 보고 있는 파일/에디터 맥락"이 강합니다.
  • App과 Cloud는 "작업 관리"와 "원격 실행"이 강합니다.

4. 초보자용 핵심 개념 지도

Codex를 제대로 쓰려면 아래 흐름으로 이해하면 됩니다.

  1. 프로젝트를 연다.
  2. Codex가 그 프로젝트를 믿어도 되는지 판단한다.
  3. 어떤 모델로 작업할지 정한다.
  4. 얼마나 자동으로 행동하게 할지 정한다.
  5. 필요하면 외부 도구를 붙인다.
  6. 프로젝트 규칙을 AGENTS.md나 설정 파일에 적는다.
  7. 작업 중에는 슬래시 명령으로 상태를 바꾸고 점검한다.
  8. 결과는 diff, review, 테스트로 확인한다.

이 과정에서 가장 중요한 세 축은 아래입니다.

  • 명령: 지금 당장 무엇을 하게 할 것인가
  • 설정: 앞으로도 계속 어떻게 행동하게 할 것인가
  • 안전장치: 어디까지 자동으로 허용할 것인가

5-1. CLI 슬래시 명령 전체 표

명령뜻언제 쓰나비고
/permissions승인/권한 수준 변경자동 실행 범위를 넓히거나 줄일 때매우 중요
/sandbox-add-read-dir추가 읽기 경로 허용현재 작업 폴더 밖 절대경로를 읽어야 할 때Windows 네이티브 CLI 전용
/agent활성 서브에이전트 전환하위 에이전트 작업 확인멀티에이전트 관련
/apps연결 앱 목록 삽입GitHub, Drive, Slack 등 사용 전프롬프트에 앱 멘션 삽입
/clear화면과 대화 새로 시작완전한 새 채팅이 필요할 때Ctrl+L와 다름
/compact긴 대화를 요약 압축컨텍스트가 길어졌을 때토큰 절약
/copy마지막 응답 복사결과를 바로 붙여넣고 싶을 때클립보드 복사
/diffGit 변경사항 보기수정 내용 검토추적 안 된 파일도 포함
/exitCLI 종료Codex 종료/quit와 동일
/experimental실험 기능 토글앱, 스마트 승인, 서브에이전트 등재시작 필요할 수 있음
/feedback피드백 전송버그 제보, 로그 전달유지보수팀에 전달
/initAGENTS.md 초안 생성프로젝트 규칙 파일 시작매우 유용
/logout로그아웃공용 PC 사용 후인증 정보 제거
/mcpMCP 도구 목록 확인외부 툴 연결 상태 점검설정 확인용
/mention파일/폴더를 대화에 직접 첨부특정 파일을 꼭 보게 만들 때파일 경로 지정
/model현재 모델 변경더 빠르게/더 깊게 작업하고 싶을 때모델 선택
/fastFast 모드 토글속도 우선 작업공식 문서상 GPT-5.4 관련
/plan계획 모드 전환코딩 전에 실행 계획만 받고 싶을 때작업 중엔 잠시 불가
/personality응답 스타일 변경더 친절하게, 더 실무적으로 등friendly, pragmatic, none
/ps백그라운드 터미널 보기오래 걸리는 명령 상태 점검unified_exec 관련
/fork현재 대화 복제다른 접근법을 병렬로 시험할 때새 스레드 생성
/resume이전 대화 재개저장된 작업 이어서 할 때세션 선택
/new같은 세션 안에서 새 대화 시작화면은 유지하고 맥락만 새로 시작/clear와 다름
/quitCLI 종료종료/exit와 동일
/review현재 작업트리 코드 리뷰수정 후 문제점 점검테스트 누락, 리스크 확인
/status현재 상태 보기모델, 토큰, 권한, 쓰기 경로 확인가장 자주 씀
/debug-config설정 계층 진단왜 설정이 안 먹는지 디버깅할 때매우 중요
/statusline하단 상태줄 항목 설정모델, 토큰, 브랜치 등 표시 조정config.toml에 저장

추가 참고:

  • /approvals는 예전 이름의 별칭(alias) 으로 여전히 동작하지만, 공식 문서상 슬래시 팝업 기본 목록에는 보이지 않습니다.

5-2. 특히 자주 쓰는 핵심 슬래시 명령 12개

/status

가장 먼저 익혀야 하는 명령입니다.

  • 현재 모델
  • 승인 정책
  • 쓰기 가능한 경로
  • 토큰 사용량
  • 현재 세션 상태

를 빠르게 확인할 수 있습니다.

/permissions

초보자에게 가장 중요한 안전 명령입니다.

  • 너무 자동으로 움직여서 불안할 때는 더 엄격하게
  • 너무 자꾸 물어봐서 답답할 때는 더 완화되게

바꿀 수 있습니다.

/model

현재 작업에 맞게 모델을 바꿉니다.

  • 빠른 초안 작업
  • 깊은 분석
  • 리뷰 중심 작업

처럼 상황별로 모델 선택을 바꾸는 용도입니다.

/plan

바로 코드 수정하지 말고 먼저 계획만 받고 싶을 때 씁니다.

예:

/plan 이 프로젝트의 로그인 구조를 먼저 분석하고 수정 계획만 세워줘

/review

지금 바뀐 코드에 대해 "문제점 중심"으로 점검받는 명령입니다.

  • 동작 버그
  • 회귀 가능성
  • 테스트 누락
  • 위험한 변경

을 찾는 데 유용합니다.

/diff

Codex가 실제로 뭘 바꿨는지 눈으로 확인하는 명령입니다.

/compact

긴 대화가 누적되면 문맥이 무거워집니다. 그럴 때 앞부분을 요약 압축해서 중요 정보는 남기고 길이만 줄이는 명령입니다.

/mention

Codex가 특정 파일을 놓치지 않게 하려면 직접 파일을 집어서 맥락에 넣는 방식입니다.

예:

/mention src/auth/login.ts

/new 와 /clear 차이

  • /new: 화면은 유지하고 새 대화만 시작
  • /clear: 화면도 비우고 대화도 새로 시작
  • Ctrl+L: 화면만 지우고 현재 대화는 유지

이 셋은 완전히 다릅니다.

/debug-config

설정이 안 먹을 때 거의 정답 같은 명령입니다.

아래를 확인할 수 있습니다.

  • 어떤 config 파일이 읽혔는지
  • 어떤 설정이 최종적으로 이겼는지
  • 조직 정책 때문에 막힌 값이 있는지
  • MCP, rules, network 관련 진단

/mcp

외부 도구가 붙어 있는지 확인하는 명령입니다.

예:

  • 문서 검색 서버
  • 브라우저 도구
  • Figma
  • 사내 도구

/apps

8. 설정 파일은 어디에 있고 어떤 순서로 적용되는가

Codex 설정의 핵심은 config.toml 입니다.

8-1. 기본 위치

위치의미
~/.codex/config.toml사용자 전체 기본 설정
<프로젝트>/.codex/config.toml그 프로젝트 전용 설정
/etc/codex/config.toml시스템 전체 설정, 유닉스 계열

8-2. 적용 우선순위

공식 문서 기준 높은 우선순위부터 낮은 우선순위까지 아래 순서입니다.

  1. CLI 플래그와 --config 직접 덮어쓰기
  2. --profile <name> 로 선택한 프로필 값
  3. 프로젝트 .codex/config.toml 들
  4. 사용자 ~/.codex/config.toml
  5. 시스템 config
  6. 내장 기본값

이 말은 곧 아래를 의미합니다.

  • 지금 한 번만 바꾸고 싶으면 --config
  • 내 개인 기본 취향이면 ~/.codex/config.toml
  • 팀/프로젝트 규칙이면 .codex/config.toml

8-3. 왜 프로젝트 설정이 무시될 수 있는가

공식 문서상 Codex는 신뢰(trust)된 프로젝트만 프로젝트 범위 .codex/ 설정을 읽습니다.

즉, 프로젝트가 신뢰되지 않으면 아래가 무시될 수 있습니다.

  • .codex/config.toml
  • 프로젝트 쪽 일부 설정 레이어

따라서 "왜 프로젝트 설정이 안 먹지?" 싶으면 제일 먼저 의심해야 합니다.

9. config.toml에서 자주 만지는 핵심 설정들

공식 샘플 설정과 로컬 도움말을 기준으로, 일반 사용자가 알아야 하는 핵심 키들을 먼저 정리하면 아래와 같습니다.

9-1. 모델 관련

model = "gpt-5.4"
# personality = "pragmatic"
# review_model = "gpt-5.4"
model_provider = "openai"
# service_tier = "flex"

뜻:

  • model: 기본 모델
  • personality: 말투/응답 스타일
  • review_model: /review 전용 모델 지정
  • model_provider: 어떤 제공자를 쓸지
  • service_tier: 속도/서비스 티어 관련

9-2. 승인과 샌드박스

approval_policy = "on-request"
sandbox_mode = "read-only"
allow_login_shell = true

뜻:

  • approval_policy: 언제 사용자에게 물어볼지
  • sandbox_mode: Codex가 기술적으로 어디까지 접근 가능한지
  • allow_login_shell: 로그인 셸 허용 여부

9-3. 작업공간 쓰기 + 네트워크

[sandbox_workspace_write]
writable_roots = []
network_access = false
exclude_tmpdir_env_var = false
exclude_slash_tmp = false

뜻:

  • writable_roots: 추가 쓰기 허용 경로
  • network_access: 샌드박스 안 명령의 외부 네트워크 허용 여부

9-4. 웹 검색

web_search = "cached"

가능 값:

  • disabled
  • cached
  • live

쉬운 설명:

  • cached: OpenAI가 유지하는 검색 캐시 기반
  • live: 실시간 웹
  • disabled: 웹 검색 끔

9-5. 프로젝트 문서와 AGENTS 관련

project_doc_max_bytes = 32768
project_doc_fallback_filenames = []

뜻:

  • project_doc_max_bytes: AGENTS류 문서를 얼마나 읽을지
  • project_doc_fallback_filenames: AGENTS.md 말고 다른 파일명도 지침 문서로 인정할지

예:

project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md"]
project_doc_max_bytes = 65536

file_opener = "vscode"
hide_agent_reasoning = false
show_raw_agent_reasoning = false
disable_paste_burst = false
check_for_update_on_startup = true

범주예시 키의미

초보자라면 이 범주를 한 번에 다 건드릴 필요는 없습니다.

하지만 아래 상황에서는 알아두면 좋습니다.

  • 회사 정책 때문에 로그인 방식이 고정되어야 할 때
  • 특정 폴더를 신뢰 프로젝트로 자동 인식시키고 싶을 때
  • 연결 앱별로 동작을 제한하고 싶을 때
  • CLI 하단 상태줄을 팀 공통 표준으로 맞추고 싶을 때

10. 초보자에게 추천하는 실제 설정 예시 3가지

10-1. 가장 안전한 입문형

model = "gpt-5.4"
approval_policy = "on-request"
sandbox_mode = "read-only"
web_search = "cached"

추천 대상:

  • 처음 쓰는 사람
  • 실수로 파일이 바뀌는 것이 불안한 사람

10-2. 가장 무난한 실전형

model = "gpt-5.4"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "cached"

[sandbox_workspace_write]
network_access = false

추천 대상:

  • 로컬 코드 수정까지 맡기고 싶은 사람
  • 하지만 네트워크는 아직 막고 싶은 사람

10-3. 자동화 비중이 큰 고급형

model = "gpt-5.4"
approval_policy = "never"
sandbox_mode = "workspace-write"
web_search = "live"

[sandbox_workspace_write]
network_access = true

주의:

  • 편하지만 위험도가 올라갑니다.
  • 공용 PC, 회사 PC, 민감한 저장소에는 신중해야 합니다.

11. 승인 정책과 샌드박스는 무엇이 다른가

이 둘은 비슷해 보여도 완전히 다릅니다.

11-1. 승인 정책 approval_policy

이건 "실행 전에 사람에게 물어볼지" 를 정합니다.

공식 샘플과 로컬 도움말 기준 주요 값:

값뜻
untrusted안전하다고 알려진 읽기성 명령만 자동, 나머지는 물어봄
on-requestCodex가 필요하다고 판단하면 물어봄
never아예 묻지 않음
on-failure예전 방식, 로컬 도움말상 deprecated 안내가 있음
granular 계열 설정상황별로 더 세밀하게 제어하는 고급 방식

11-2. 샌드박스 sandbox_mode

이건 "기술적으로 어디까지 접근 가능한가" 를 정합니다.

값뜻
read-only읽기 위주, 수정 거의 불가
workspace-write현재 작업공간에 한해 수정 가능
danger-full-access사실상 제한 없음, 매우 위험

쉽게 정리하면:

  • 승인 정책 = 행동 전에 물어볼지
  • 샌드박스 = 애초에 가능한 행동 범위가 어디까지인지

둘은 따로 존재하지만 같이 움직입니다.

12. 네트워크와 웹 검색은 또 별개다

초보자가 많이 헷갈리는 두 번째 포인트입니다.

12-1. 샌드박스 안 네트워크

공식 문서상 workspace-write 기본 상태에서는 네트워크가 꺼져 있습니다.

켜려면 예를 들어:

[sandbox_workspace_write]
network_access = true

12-2. 웹 검색

웹 검색은 또 별도 제어가 가능합니다.

web_search = "cached"

또는 CLI에서:

codex --search

핵심은 아래입니다.

  • 네트워크 허용 = 셸 명령이 외부 인터넷에 나갈 수 있음
  • 웹 검색 = Codex가 검색 도구를 쓸 수 있음

즉, 둘은 같은 개념이 아닙니다.

13. AGENTS.md는 왜 중요한가

일반 사용자 입장에서 AGENTS.md는 "이 프로젝트에서 Codex가 꼭 지켜야 할 상시 지침 메모" 입니다.

예를 들면:

  • 자바스크립트 파일을 바꾸면 항상 테스트 돌리기
  • pnpm만 쓰기
  • 운영 의존성 추가 전에 확인받기

13-1. 어디에 둘 수 있는가

공식 문서 기준:

  • 전역: ~/.codex/AGENTS.md
  • 전역 임시 덮어쓰기: ~/.codex/AGENTS.override.md
  • 프로젝트 루트: <repo>/AGENTS.md
  • 하위 디렉터리 특수 규칙: <repo>/some/path/AGENTS.override.md

13-2. 탐색 순서 핵심

공식 문서 요약:

  1. 전역 범위에서는 AGENTS.override.md가 있으면 그걸 우선
  2. 프로젝트 루트에서 현재 작업 디렉터리까지 내려오며 탐색
  3. 각 디렉터리마다 AGENTS.override.md가 AGENTS.md보다 우선
  4. 한 디렉터리에서는 최대 한 파일만 채택

즉, 가까운 위치의 override가 더 강합니다.

13-3. 아주 실용적인 예시

# AGENTS.md

## Repository expectations

- 수정 후에는 최소 lint를 실행한다.
- 공개 API 동작이 바뀌면 docs도 함께 수정한다.
- 새 의존성 추가 전에는 반드시 사용자 확인을 받는다.

이 한 장만 있어도 Codex의 일관성이 눈에 띄게 좋아집니다.

14. Rules, Hooks, Skills, Plugins, MCP, Subagents를 한 번에 이해하기

이 부분은 이름이 많아서 초보자들이 제일 어렵다고 느낍니다.

14-1. 한 줄 정의

개념한 줄 설명
Rules샌드박스 밖 명령을 어떤 조건에서 허용/질문할지 정하는 규칙
HooksCodex 생명주기 중 특정 순간에 자동 스크립트를 실행하는 장치
Skills특정 작업을 잘 하도록 만든 재사용 워크플로우
PluginsSkills + Apps + MCP를 묶어 배포하는 확장 패키지
MCP외부 도구/문맥 서버를 Codex에 연결하는 표준
SubagentsCodex가 하위 전문 작업자를 병렬로 돌리는 방식

14-2. Rules

공식 문서상 Rules는 샌드박스 밖에서 어떤 명령을 어떻게 다룰지 제어하는 기능입니다.

예시 개념:

  • gh pr view 계열은 승인 후 허용
  • 특정 위험 명령은 항상 차단

즉, 보안과 통제용입니다.

14-3. Hooks

공식 문서상 Hooks는 대화/툴 사용 전후에 자동 스크립트를 실행하는 확장 구조입니다.

예시:

  • API 키가 프롬프트에 들어왔는지 검사
  • 대화 종료 시 자동 요약 저장
  • 특정 디렉터리에서만 별도 검증 스크립트 실행

현재 문서상 Hooks는 실험적이고, Windows 지원은 일시 비활성화 안내가 있습니다.

14-4. Skills

공식 문서상 Skill은 아래를 묶는 작업 레시피입니다.

  • SKILL.md
  • 필요하면 scripts/
  • 필요하면 references/
  • 필요하면 assets/

초보자 관점에서는 이렇게 보면 됩니다.

  • AGENTS.md = 프로젝트 공통 규칙
  • Skill = 특정 작업을 잘하게 만드는 전문 매뉴얼

14-5. Plugins

공식 문서상 Plugin은 아래를 묶습니다.

  • Skills
  • Apps
  • MCP servers

즉, 배포 단위입니다.

쉽게 말해:

  • Skill은 요리법
  • Plugin은 요리법 + 재료 + 도구 세트

참고:

  • 공식 Plugins 문서에는 CLI 안에서 codex 실행 후 /plugins로 플러그인 목록을 여는 예시가 있습니다.

14-6. MCP

MCP는 Model Context Protocol의 약자입니다.

쉽게 설명하면:

  • Codex가 외부 도구를 "표준 방식"으로 붙이는 방법

예:

  • 문서 검색 서버
  • 브라우저 제어
  • Figma
  • 사내 위키/DB

설정은 주로 아래 둘 중 하나입니다.

  1. codex mcp ... CLI로 관리
  2. config.toml 직접 편집

14-7. Subagents

공식 문서상 Codex는 사용자가 명시적으로 요청할 때만 서브에이전트를 띄웁니다.

쉽게 말하면:

  • 메인 Codex가 팀장
  • 서브에이전트는 병렬 전문 작업자

유용한 상황:

  • 대형 코드베이스 탐색
  • 독립된 두 기능을 병렬 분석
  • 리뷰와 구현을 분리

단점:

  • 토큰 사용량 증가
  • 과한 병렬화는 오히려 복잡해질 수 있음

15. Codex와 다른 코딩툴은 무엇이 다른가

여기서는 일반적인 구조 차이를 설명합니다. 다른 툴은 버전별 변화가 크므로, 아래 비교는 개념 중심으로 보는 것이 맞습니다.

15-1. 큰 분류부터

도구 유형대표 예중심 철학
에이전트형 코딩 도구Codex, Claude Code 계열일을 맡기고 결과를 받는 방식
IDE 보조형 도구Cursor, GitHub Copilot 계열편집 중 실시간 보조
생성형 UI/앱 빌더v0, Lovable 계열프롬프트로 화면/앱 생성
검색/리서치형 보조Perplexity류자료 찾기와 요약 중심

15-2. Codex가 특히 다른 점

항목Codex의 특징
제어 방식프롬프트 + 슬래시 명령 + CLI 옵션 + 설정 파일이 함께 존재
지속 지시AGENTS.md와 config.toml을 함께 사용
안전 구조승인 정책과 샌드박스를 분리해서 제어
확장 방식MCP, Skills, Plugins, Subagents가 체계적으로 나뉨
사용 위치CLI, IDE, App, Cloud를 넘나듦
작업 방식단순 답변보다 "실행 가능한 작업 흐름"에 가깝다

15-3. 다른 툴과 감각적 차이

GitHub Copilot 계열과 비교

  • Copilot 계열은 보통 "코드 자동완성/짧은 제안"이 강합니다.
  • Codex는 "프로젝트 전체를 읽고 수정하고 검토하는 에이전트성"이 더 강합니다.

Cursor/Windsurf 같은 IDE 중심 도구와 비교

  • IDE 중심 도구는 에디터 안 흐름이 매우 자연스럽습니다.
  • Codex도 IDE 확장이 있지만, CLI/App/Cloud까지 하나의 체계로 묶인 점이 특징입니다.
  • Codex는 설정을 config.toml, AGENTS.md, 슬래시 명령으로 매우 명시적으로 제어하는 편입니다.

Claude Code 같은 터미널 에이전트와 비교

  • 둘 다 대화형 에이전트라는 공통점이 있습니다.
  • Codex는 공식 문서상 MCP, Skills, Plugins, Cloud, App/IDE/CLI 통합 문서 체계가 명확한 편입니다.

v0 같은 UI 생성 도구와 비교

  • v0류는 화면 생성과 프론트엔드 초안에 특화됩니다.
  • Codex는 UI만이 아니라 코드베이스 수정, 테스트, 리뷰, 도구 연동까지 범위가 더 넓습니다.

15-4. 설정 방식 차이만 따로 보면

도구 성격설정을 주로 어디서 하나
Codexconfig.toml, AGENTS.md, 슬래시 명령, CLI 옵션
IDE 보조형 도구IDE 설정 메뉴, 프로젝트 규칙 파일
UI 생성형 도구웹 UI 설정, 프로젝트 연결 옵션
단순 챗봇형 도구프롬프트와 약간의 사용자 설정

즉, Codex는 "설정이 많은 대신 통제력도 높은 도구" 쪽에 가깝습니다.

16. 일반 사용자가 가장 많이 하는 실수

16-1. 슬래시 명령과 셸 명령을 섞어 쓰는 실수

예:

  • /review 는 Codex 대화 안 명령
  • git status 는 셸 명령

둘은 입력 위치가 다릅니다.

16-2. AGENTS.md와 config.toml 역할을 섞는 실수

  • AGENTS.md = 작업 지침
  • config.toml = 프로그램 동작 설정

예:

  • "테스트 꼭 돌려라"는 AGENTS.md
  • "샌드박스는 read-only"는 config.toml

16-3. 권한과 샌드박스 차이를 모르는 실수

  • 묻는지 안 묻는지는 approval_policy
  • 기술적으로 가능한 범위는 sandbox_mode

16-4. 프로젝트 설정이 안 먹는데 이유를 안 찾는 실수

보통 원인은 아래입니다.

  • 프로젝트가 trust 상태가 아님
  • 현재 작업 디렉터리가 예상과 다름
  • 상위/하위 config 우선순위를 잘못 이해함

이때는 /debug-config가 가장 중요합니다.

16-5. /new, /clear, Ctrl+L를 같은 것으로 생각하는 실수

셋은 다릅니다.

16-6. /compact를 안 써서 대화가 너무 길어지는 실수

긴 작업에서는 /compact가 매우 중요합니다.

17. 처음 쓰는 사람을 위한 추천 사용 순서

17-1. 가장 쉬운 시작 순서

  1. codex 실행
  2. /status로 현재 상태 확인
  3. /permissions로 너무 공격적이지 않게 설정
  4. 간단한 읽기 작업부터 시키기
  5. 특정 파일이 중요하면 /mention
  6. 수정 작업 후 /diff
  7. 다시 /review

17-2. 프로젝트에 오래 붙여 쓸 때

  1. ~/.codex/config.toml에 개인 기본 취향 저장
  2. 프로젝트 루트에 AGENTS.md 작성
  3. 필요하면 .codex/config.toml로 프로젝트 전용 설정 추가
  4. 외부 도구가 필요하면 codex mcp 또는 [mcp_servers] 설정
  5. 팀 표준 자동화가 필요하면 Hooks/Rules/Skills 고려

18. 정말 자주 쓰는 실전 치트시트

지금 상태 확인

/status

자동 실행 범위 바꾸기

/permissions

모델 바꾸기

/model

/new

/clear

/debug-config

19. 추천 결론

일반 사용자가 Codex를 가장 잘 쓰는 방법은 복잡한 기능을 한 번에 다 켜는 것이 아니라, 아래 순서로 단계적으로 익히는 것입니다.

  1. 프롬프트, 슬래시 명령, 셸 명령 차이 이해
  2. /status, /permissions, /diff, /review 먼저 익히기
  3. config.toml은 최소한만 만지기
  4. 프로젝트에 AGENTS.md 추가하기
  5. 필요할 때만 MCP, Skills, Plugins, Hooks, Subagents 확장하기

한 문장으로 정리하면 이렇습니다.

Codex는 "채팅형 AI"라기보다, 설정과 규칙을 먹고 움직이는 "작업형 코딩 에이전트"에 가깝습니다.

그래서 잘 쓰는 핵심은 아래입니다.

  • 무엇을 시킬지 잘 말하기
  • 어디까지 허용할지 잘 정하기
  • 어떤 규칙을 기억시킬지 문서화하기
