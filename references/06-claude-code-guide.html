1. Claude Code가 뭔가요?

한마디로: 터미널(검은 화면)에서 대화하면서 코딩을 대신 해주는 AI 비서입니다.

일반적인 AI 코딩 툴과의 차이:

  • ChatGPT에 코드를 복붙하는 게 아닙니다
  • Claude Code는 직접 여러분의 프로젝트 파일을 읽고, 수정하고, 명령어를 실행합니다
  • 마치 "옆에 앉은 시니어 개발자"가 키보드를 잡고 직접 일하는 것과 같습니다

쓸 수 있는 곳:

환경설명
터미널 (CLI)가장 기본. 터미널에 claude 입력
VS Code 확장VS Code 안에서 패널로 사용
JetBrains 플러그인IntelliJ, PyCharm 등에서 사용
데스크탑 앱Windows/Mac 전용 앱
웹claude.ai/code 에서 브라우저로 사용

3. 설치하고 처음 시작하기

설치 (한 번만)

npm install -g @anthropic-ai/claude-code
쉬운 설명: npm은 Node.js 패키지 관리자입니다. Node.js가 없으면 먼저 nodejs.org에서 설치하세요.

Mac이라면 더 쉽게

brew install claude-code

처음 실행

cd 내프로젝트폴더     # 작업할 폴더로 이동
claude                # Claude Code 시작!

처음 실행하면:

  1. 로그인 화면이 나옵니다 (Anthropic 계정 또는 API 키)
  2. 로그인하면 바로 대화를 시작할 수 있습니다
  3. 프로젝트 폴더에 .claude/ 폴더가 생깁니다 (설정 저장용)

바로 질문하면서 시작하기

claude "이 프로젝트가 뭘 하는 코드인지 설명해줘"
사례: 남이 만든 프로젝트를 처음 봤을 때, 전체 구조를 파악하고 싶을 때

4. 터미널에서 실행할 때 옵션들

claude 명령어 뒤에 붙이는 옵션(플래그)들입니다.

자주 쓰는 옵션

-p 또는 --print — 질문하고 답만 받기

claude -p "package.json에 있는 의존성 목록 알려줘"
쉬운 설명: 대화 모드로 들어가지 않고, 질문 → 답변 → 자동 종료. 스크립트에서 자동화할 때 유용.

>

사례: CI/CD 파이프라인에서 "코드 리뷰 자동화" 같은 것을 만들 때 사용

-c 또는 --continue — 아까 하던 대화 이어하기

claude -c
사례: 점심 먹고 돌아와서 "아까 버그 고치던 거 이어서 해줘"

-r 또는 --resume — 특정 과거 대화 다시 열기

claude -r              # 최근 대화 목록에서 선택
claude -r abc123       # 특정 세션 ID로 바로 열기
사례: "3일 전에 로그인 기능 만들던 대화 다시 열어야 해"

--model — AI 모델 고르기

claude --model opus     # 가장 똑똑한 모델 (비쌈)
claude --model sonnet   # 균형 잡힌 모델 (기본값)
claude --model haiku    # 가장 빠르고 저렴한 모델
사례: - 복잡한 아키텍처 설계 → opus - 일반 코딩 → sonnet - 간단한 질문, 파일 이름 바꾸기 → haiku

--fallback-model — 백업 모델

claude --model opus --fallback-model sonnet
쉬운 설명: opus가 바쁘거나 에러 나면 자동으로 sonnet으로 전환

--permission-mode — 권한 모드 지정

claude --permission-mode plan          # 읽기만 가능 (안전)
claude --permission-mode acceptEdits   # 파일 수정 자동 승인
사례: 처음 쓰는 프로젝트에서는 plan 모드로 시작해서 뭘 할지 먼저 확인

--dangerously-skip-permissions — 모든 허락 건너뛰기

claude --dangerously-skip-permissions
주의: 모든 파일 수정, 명령어 실행을 묻지 않고 바로 합니다. 테스트 환경에서만 쓰세요!

--allowed-tools — 특정 도구만 허용

claude --allowed-tools "Read Grep Glob"
사례: "읽기만 하고 아무것도 수정하지 마" 라고 제한하고 싶을 때

--disallowed-tools — 특정 도구 금지

claude --disallowed-tools "Bash Write"
사례: "터미널 명령 실행이랑 파일 새로 만드는 건 하지 마"

--add-dir — 추가 작업 폴더

claude --add-dir ../shared-library
사례: 메인 프로젝트 말고 공유 라이브러리 폴더도 같이 봐야 할 때

--mcp-config — MCP 설정 파일 지정

claude --mcp-config ./my-mcp-servers.json
사례: 프로젝트마다 다른 외부 도구 연결이 필요할 때

--output-format — 출력 형식 (스크립트용)

claude -p "버그 목록" --output-format json
사례: 다른 프로그램이 Claude의 결과를 파싱해야 할 때. text, json, stream-json 중 선택

--verbose / --debug — 상세 로그

claude --verbose    # 자세한 로그
claude --debug      # 디버그 수준 로그
사례: "왜 안 되지?" 싶을 때 내부에서 무슨 일이 일어나는지 확인

--append-system-prompt — 숨은 지시 추가

claude --append-system-prompt "항상 한국어로 대답해"
사례: 자동화 스크립트에서 Claude의 동작 규칙을 추가로 넣고 싶을 때

--settings — 커스텀 설정 파일

claude --settings ./team-settings.json
사례: 팀원마다 다른 설정 파일을 쓰고 싶을 때

--session-id — 세션 ID 지정

claude --session-id "my-debug-session-001"
사례: 자동화에서 세션을 추적·관리할 때

서브커맨드 (claude 뒤에 붙는 하위 명령)

claude config — 설정 관리

claude config set model sonnet    # 기본 모델을 sonnet으로
claude config get model           # 현재 기본 모델 확인
claude config list                # 모든 설정 보기
사례: "매번 --model sonnet 치기 귀찮아. 기본값으로 저장하자"

claude mcp — MCP 서버 관리

claude mcp add        # 새 MCP 서버 추가 (대화형)
claude mcp list       # 설치된 서버 목록
claude mcp remove     # 서버 제거
사례: GitHub API 연결하거나, DB 직접 쿼리 기능 추가할 때

claude doctor — 환경 진단

claude doctor
사례: "설치했는데 뭔가 이상해" → 이 명령으로 문제 자동 탐지

claude update — 업데이트

claude update
사례: 새 기능이 나왔을 때. 주기적으로 실행 권장

claude login / claude logout — 계정

claude login       # 로그인
claude logout      # 로그아웃 (계정 전환 시)

5. 슬래시 명령어 완전 목록

대화 중에 / 를 치면 나오는 명령어입니다. 일반 대화와 구분하기 위해 슬래시(/)로 시작합니다.

세션/대화 관리

/help — 도움말

/help
언제: Claude Code 사용법이 기억 안 날 때. 간단한 가이드가 표시됩니다.

/clear — 대화 초기화

/clear
쉬운 설명: 지금까지의 대화를 전부 잊어버립니다. 새 주제를 시작할 때 유용합니다.

>

사례: "로그인 기능 작업이 끝났으니, 이제 결제 기능 작업을 시작하자" → /clear 하고 새로 시작

/compact — 대화 압축

/compact
/compact 현재까지의 버그 수정 내용만 기억해
쉬운 설명: 대화가 너무 길어지면 AI가 앞부분을 잊어버릴 수 있습니다. /compact는 중요한 내용만 요약해서 기억 공간을 확보합니다.

>

사례: 2시간째 작업 중인데 Claude가 앞에서 한 말을 잊어버릴 때

>

팁: 뒤에 힌트를 주면 ("현재까지의 DB 스키마 변경만 기억해") 해당 내용을 우선 보존

/exit 또는 /quit — 종료

/exit
/quit
사례: 작업 끝났을 때. Ctrl+D나 Ctrl+C 두 번도 같은 효과.

/resume — 과거 대화 목록에서 재개

/resume

8. 4가지 모드

Claude Code에는 "Claude가 얼마나 자유롭게 행동할 수 있는지" 결정하는 4가지 모드가 있습니다.

Shift+Tab으로 빠르게 전환하거나, /mode 명령으로 선택합니다.

Default 모드 (기본)

"모든 행동 전에 허락을 구합니다"
  • 파일 수정 전: "이 파일 수정해도 될까요?" → Yes/No
  • 명령 실행 전: "npm install 실행해도 될까요?" → Yes/No
  • 장점: 가장 안전
  • 단점: 매번 허락하는 게 귀찮음
사례: 중요한 프로덕션 코드 작업할 때. 실수 방지.

Auto-Accept Edits 모드

"파일 수정은 알아서 하고, 위험한 명령만 물어봅니다"
  • 파일 읽기/수정: 자동 허용
  • git push, rm 같은 위험 명령: 여전히 물어봄
  • 장점: 편리하면서도 적당히 안전
  • 단점: 원치 않는 파일 수정이 될 수 있음
사례: 일반적인 개발 작업. 가장 많이 쓰는 모드.

Plan 모드 (읽기 전용)

"아무것도 수정하지 않고, 계획만 세웁니다"
  • 파일 읽기: 가능
  • 파일 수정: 불가능
  • 명령 실행: 불가능
  • 장점: 먼저 계획을 검토한 후 실행할 수 있음
  • 단점: 계획 확인 후 직접 모드를 전환해야 함
사례: - "이 리팩토링 어떻게 할 건지 먼저 계획 보여줘" → Plan 모드에서 계획 검토 - 검토 후 Shift+Tab으로 Auto-Accept로 전환 → "실행해"

>

다른 툴에는 없는 기능! "먼저 생각, 나중에 행동" 패턴을 강제할 수 있습니다.

Bypass Permissions 모드

"아무것도 묻지 않고 전부 실행합니다"
  • 모든 파일 수정: 자동
  • 모든 명령 실행: 자동
  • 장점: 가장 빠름
  • 단점: 위험! 잘못된 명령도 바로 실행됨
사례: 격리된 테스트 환경, Docker 안, 또는 CI/CD에서만 사용. 실제 개발에서는 비추천.

9. Thinking 모드 — 더 깊이 생각하게 하기

프롬프트에 특정 키워드를 넣으면 Claude가 더 오래, 더 깊이 생각합니다.

키워드사고 깊이비유언제 쓰나
think보통잠깐 생각약간 복잡한 질문
think hard깊음진지하게 고민여러 파일에 걸친 버그
think harder매우 깊음풀 집중아키텍처 설계
ultrathink최대모든 가능성 검토가장 어려운 문제

사용법

think: 이 함수가 왜 느린지 분석해줘

think hard: 이 에러의 근본 원인을 찾아줘. 여러 파일에 걸쳐서 추적해봐.

think harder: 이 마이크로서비스 아키텍처를 모노레포로 전환하는 방법을 설계해줘

ultrathink: 이 코드베이스 전체의 성능 병목을 찾고 최적화 전략을 세워줘
비유: 일반 모드는 "5초 생각하고 대답", ultrathink는 "5분 숙고하고 대답"

>

주의: 더 깊이 생각할수록 토큰(비용)을 더 많이 씁니다.

>

다른 툴에는 없는 기능! Cursor나 Copilot에서는 사고 깊이를 조절할 수 없습니다.

10. 메모리 시스템

Claude Code는 대화가 끝나도 기억을 유지하는 영구 메모리가 있습니다. CLAUDE.md 파일에 저장됩니다.

4단계 계층

1. 엔터프라이즈 정책 (회사 관리자가 배포, 최상위 우선순위)
    ↓
2. 사용자 메모리:  ~/.claude/CLAUDE.md
   → 내가 하는 모든 프로젝트에 공통 적용
    ↓
3. 프로젝트 메모리: ./CLAUDE.md
   → 이 프로젝트에만 적용, 팀원과 공유 (git 커밋)
    ↓
4. 로컬 메모리:     ./CLAUDE.local.md
   → 이 프로젝트에만, 나만 적용 (git에 안 올라감)

실제 활용 사례

사용자 메모리 (~/.claude/CLAUDE.md):

# 나의 전역 설정
- 항상 한국어로 대답해
- 커밋 메시지는 영어로 써
- console.log 대신 logger 쓰기

프로젝트 메모리 (./CLAUDE.md):

# 프로젝트 규칙 (팀 전원 적용)
- TypeScript strict 모드 필수
- API 응답은 camelCase
- 테스트 커버리지 80% 이상 유지
- Tailwind CSS 사용, 인라인 스타일 금지

@docs/architecture.md    ← 다른 파일도 import 가능!

로컬 메모리 (./CLAUDE.local.md):

# 내 개인 설정 (팀원에게 안 보임)
- 내 로컬 API 키: .env.local 참조
- 현재 작업 중: 결제 모듈 리팩토링

메모리 만드는 3가지 방법

  1. /init — Claude가 프로젝트를 분석해서 자동 생성
  2. /memory — 에디터에서 직접 편집
  3. #메시지 — 대화 중 빠르게 추가
다른 툴과의 차이: - Cursor의 .cursorrules는 단일 파일, 계층 없음 - Claude Code는 4단계 계층 + 파일 import + 하위폴더별 분리 가능

12. 권한 시스템

"Claude가 뭘 할 수 있고 뭘 못하게 할지" 정하는 시스템입니다.

패턴 문법

Bash                 → 모든 터미널 명령
Bash(git:*)          → git으로 시작하는 모든 명령 (git push, git pull 등)
Bash(git diff:*)     → git diff 계열만
Bash(rm:*)           → rm으로 시작하는 삭제 명령
Bash(npm publish:*)  → npm publish 명령

Read                 → 모든 파일 읽기
Read(~/.ssh/**)      → SSH 키 폴더 읽기
Read(src/**)         → src 폴더 내 읽기

Edit                 → 모든 파일 수정
Edit(src/**)         → src 폴더만 수정 허용

Write                → 새 파일 생성
Write(**/.env)       → .env 파일 생성

mcp__github__*       → GitHub MCP 서버의 모든 도구

3가지 동작

동작설명사례
allow물어보지 않고 자동 허용git 명령, 파일 읽기
deny무조건 차단 (Claude가 시도조차 못함)rm -rf, .env 파일 수정
ask매번 물어봄npm publish, DB 마이그레이션

실제 설정 예시

{
  "permissions": {
    "allow": [
      "Read",              // 모든 파일 읽기 OK
      "Edit(src/**)",      // src 폴더 수정 OK
      "Bash(git:*)",       // git 명령 OK
      "Bash(npm test:*)",  // 테스트 실행 OK
      "Bash(npm run:*)"    // npm 스크립트 실행 OK
    ],
    "deny": [
      "Bash(rm -rf:*)",     // 재귀 삭제 절대 금지
      "Write(**/.env*)",    // .env 파일 생성 금지
      "Read(~/.ssh/**)"     // SSH 키 읽기 금지
    ],
    "ask": [
      "Bash(npm publish:*)", // 패키지 배포는 항상 확인
      "Bash(git push:*)"     // 푸시는 항상 확인
    ]
  }
}

대화 중 동적 권한

Claude가 뭔가 하려고 물어볼 때:

Claude wants to run: git push origin main
[Y] Yes  [N] No  [A] Yes, and always allow Bash(git push:*)

A를 선택하면 settings.json의 allow에 자동 저장됩니다.

13. Hooks

한마디로: "Claude가 뭔가 할 때 자동으로 실행되는 내 스크립트"

다른 AI 코딩 툴에는 없는 Claude Code만의 핵심 기능입니다.

9가지 이벤트

이벤트발동 시점사례
PreToolUseClaude가 도구를 쓰기 직전"위험한 명령 차단", "로그 기록"
PostToolUseClaude가 도구를 쓴 직후"파일 수정 후 자동 포맷팅"
UserPromptSubmit내가 메시지를 보낼 때"입력 검증", "자동 컨텍스트 추가"
Notification알림이 뜰 때"슬랙으로 알림 전송"
StopClaude 응답 완료 시"작업 완료 알림", "자동 테스트 실행"
SubagentStop서브에이전트 완료 시"서브에이전트 결과 로깅"
SessionStart세션 시작 시"환경 체크", "상태 표시"
SessionEnd세션 종료 시"세션 요약 저장"
PreCompact컨텍스트 압축 전"중요 정보 백업"

실제 사용 사례들

사례 1: 파일 수정 후 자동 포맷팅

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write $CLAUDE_FILE_PATHS"
          }
        ]
      }
    ]
  }
}
Claude가 파일을 수정할 때마다 자동으로 prettier가 돌아갑니다. 수동으로 포맷팅할 필요 없음!

사례 2: 위험한 명령 차단

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/scripts/block-dangerous.sh"
          }
        ]
      }
    ]
  }
}
스크립트가 exit code 2를 리턴하면 Claude의 행동이 차단됩니다. "rm -rf /" 같은 명령 원천 봉쇄.

사례 3: 작업 완료 시 슬랙 알림

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "curl -X POST https://hooks.slack.com/... -d '{\"text\": \"Claude 작업 완료!\"}'"
          }
        ]
      }
    ]
  }
}
Claude가 응답을 끝내면 자동으로 슬랙에 알림. 긴 작업 맡기고 다른 일 할 때 유용.

사례 4: 세션 시작 시 환경 체크

{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo '현재 브랜치: '$(git branch --show-current)"
          }
        ]
      }
    ]
  }
}

훅 차단 메커니즘

  • 훅 스크립트가 exit code 2를 리턴하면 → 해당 동작이 차단됨
  • stderr로 출력한 메시지가 → Claude에게 전달됨 (이유 설명용)

14. Subagents

한마디로: "Claude의 분신. 전문 분야별 도우미를 따로 부릴 수 있음"

왜 필요한가?

비유로 설명:

  • Claude = 팀장
  • Subagent = 팀원 (각자 전문 분야가 있음)
  • 팀장이 "리뷰는 네가 해, 테스트는 네가 해" 하고 각자에게 시키는 것

실제 이점

  1. 메인 대화 보호 — 서브에이전트가 파일 100개를 검색해도, 그 결과가 메인 대화를 오염시키지 않음
  2. 병렬 작업 — 여러 서브에이전트가 동시에 다른 작업 가능
  3. 전문화 — 코드 리뷰어, 디버거, 테스터 등 역할 분리

에이전트 파일 만들기

위치:

  • ~/.claude/agents/*.md — 모든 프로젝트에서 사용
  • .claude/agents/*.md — 이 프로젝트에서만 사용

예시: 코드 리뷰 에이전트 (.claude/agents/reviewer.md)

---
name: code-reviewer
description: 코드 리뷰. 스타일, 보안, 성능 체크.
tools: Read, Grep, Glob
model: sonnet
---

당신은 코드 리뷰 전문 에이전트입니다.
다음 기준으로 리뷰하세요:
1. 보안 취약점 (SQL 인젝션, XSS 등)
2. 성능 문제 (N+1 쿼리, 불필요한 렌더링)
3. 코드 스타일 (네이밍, 함수 길이)
심각도를 상/중/하로 표시하세요.

예시: 테스트 작성 에이전트 (.claude/agents/tester.md)

---
name: test-writer
description: 테스트 코드 작성 전문
tools: Read, Write, Bash(npm test:*)
model: sonnet
---

당신은 테스트 작성 전문가입니다.
- 단위 테스트는 Vitest로 작성
- 엣지 케이스 포함 필수
- 테스트 실행까지 확인

사용법

Claude에게 자연어로 요청하면 자동으로 적절한 에이전트를 호출합니다:

이 PR을 코드 리뷰해줘
→ code-reviewer 서브에이전트 자동 실행

이 함수에 대한 테스트를 작성해줘
→ test-writer 서브에이전트 자동 실행

또는 /agents 명령으로 직접 관리:

/agents

15. Skills

16. MCP 서버

한마디로: "Claude Code에 외부 도구를 플러그인처럼 연결하는 표준 규격"

MCP가 뭔가?

MCP (Model Context Protocol) = Anthropic이 만든 오픈 표준으로, AI와 외부 도구를 연결하는 "USB 포트" 같은 것입니다.

예를 들어:

  • GitHub MCP → Claude가 직접 GitHub PR을 만들 수 있음
  • PostgreSQL MCP → Claude가 직접 DB 쿼리를 날릴 수 있음
  • Figma MCP → Claude가 디자인 파일을 읽을 수 있음

설치 방법

대화형 (가장 쉬움):

21. Git Worktree

한마디로: "같은 저장소를 여러 폴더에 복사해서 동시에 다른 브랜치 작업"

일반 Git의 문제

하나의 폴더 = 하나의 브랜치만 작업 가능
브랜치 전환하면 현재 작업물이 사라짐 (stash 필요)

Worktree의 해결

# main 브랜치는 원래 폴더에서
# feature 브랜치는 새 폴더에서 동시에!
git worktree add ../my-project-feature feature-branch
cd ../my-project-feature
claude    # 여기서 feature 작업

Claude Code에서의 활용

일부 플러그인은 자동으로 worktree를 생성합니다:

서브에이전트가 worktree에서 독립적으로 작업
→ 메인 작업을 방해하지 않음
→ 완료 후 merge
사례: - "이 버그 수정하는 동안 새 기능도 동시에 개발하고 싶어" - 하나의 Claude Code 세션에서 여러 브랜치를 동시 작업
