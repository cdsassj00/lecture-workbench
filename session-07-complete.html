RAG 파이프라인의 네 단계 · 넣기·쪼개기·찾기·답하기
RAG(Retrieval-Augmented Generation, 외부 문서 저장소에서 질문과 관련된 조각을 검색해 LLM 프롬프트에 함께 넣어 답을 생성하게 하는 구조)는 LLM이 모르는 지식을 외부 문서 저장소에서 검색해 답변에 넣어 주는 구조입니다. 기획자 관점에서는 네 단계로 기억하면 충분합니다. 첫째, 넣기는 PDF·한글·워드·엑셀 등 부서 문서를 모아 검색 가능한 저장소에 올리는 단계입니다(DeepSeek-OCR·한컴 PDF 파서·LangChain 로더 활용). 둘째, 쪼개기는 문서를 의미 단위의 청크(Chunk, 문서를 의미 단위로 나눈 작은 조각)로 나누고, 각 청크를 임베딩(Embedding, 텍스트를 수백~수천 차원 벡터 숫자로 바꾸는 변환)으로 바꿔 검색 색인에 넣습니다. 셋째, 찾기는 질문이 들어오면 임베딩 유사도로 관련 청크 top-k(검색 시 유사도 상위 몇 개 청크를 가져올지 정하는 파라미터)를 꺼내는 벡터 DB(임베딩된 청크를 저장하고 유사도로 검색하는 데이터베이스, 예: Chroma·FAISS) 조회입니다. 넷째, 답하기는 찾아온 청크를 LLM 프롬프트에 붙여 최종 답변을 생성하게 합니다.
01 · 정의RAG가 하는 일
RAG = 질문 시점에 외부 문서를 검색해 프롬프트에 끼워 답을 생성하는 구조

Claude·GPT는 학습 시점까지의 일반 지식만 안다. 부서 규정·민원 응대 DB·회의록은 학습되지 않은 데이터다. RAG는 질문이 들어올 때마다 부서 문서에서 관련 조각을 검색해 프롬프트에 함께 넣고 답을 생성한다.

STEP 01 · 넣기INGEST

STEP 02 · 쪼개기CHUNK

의미 단위 청킹 — 조항·안건·민원 1건. 임베딩은 텍스트를 수백~수천 차원 벡터로 바꿔 의미 유사도 비교가 가능하게 만드는 변환.

STEP 03 · 찾기RETRIEVE
벡터 DB에서 유사 조각 검색

Chroma · FAISS 같은 벡터 DB로 질문과 유사한 조각을 뽑는다. 상위 3~5개. 고급 옵션은 명칭만 — 리랭크·하이브리드·GraphRAG.

STEP 04 · 답하기GENERATE
검색된 조각을 AI에 전달해 답 생성

찾아온 조각을 프롬프트에 붙여 AI에 전달한다. "이 조각만 참고해 답하라". 근거가 없으면 "규정에 없음"으로 출구.

03 · 최소 구조30줄 RAG 감상
구현은 에이전트 몫 — 거점리더는 4단계의 입력·출력만 읽는다
# 1. 넣기 — PDF 10건을 텍스트로 import fitz, pathlib docs=[] for p in pathlib.Path("./규정").glob("*.pdf"): d=fitz.open(p) txt="\n".join(pg.get_text() for pg in d) docs.append({"src":p.name,"text":txt}) # 2. 쪼개기 — 500자 청크, 겹침 50 chunks=[] for d in docs: for i in range(0,len(d["text"]),450): chunks.append({"src":d["src"], "text":d["text"][i:i+500]}) # 3. 찾기 — 키워드 포함수 def search(q,k=3): s=[(c,sum(w in c["text"] for w in q.split())) for c in chunks] return sorted(s,key=lambda x:-x[1])[:k] # 4. 답하기 — Ollama 호출 def answer(q): hits=search(q) ctx="\n\n".join(h[0]["text"] for h in hits) return ollama.chat( f"다음만 참고:\n{ctx}\n\n질문:{q}")

이 30줄에 4단계가 모두 들어있다. 벡터 DB·임베딩 없이 키워드 포함 횟수만으로도 동작한다. 30줄은 학습용 최소 구조이며 운영본은 임베딩·벡터 DB·메타 필터를 더한다.

LangChain RecursiveCharacterTextSplitter는 줄바꿈 → 문단 → 문장 → 단어 순으로 기회 분할. 경계 존중 + 크기 제한을 동시에.

02 · SIZE · OVERLAP크기·겹침 전략
500자 ± 100 · 겹침 50자 — 공공 경험 기본값

너무 작으면 맥락 부족 → 답변이 얕다. 너무 크면 LLM이 관련 부분을 놓친다. 500자는 한국어 기준 3~5문단로 맥락과 집중의 균형점.

# 겹침이 있는 재귀 분할 (간이 버전) def chunk(text, size=500, overlap=50): out=[] i=0 while i < len(text): end=min(i+size,len(text)) # 문단 경계까지 당기기 if end < len(text): nl=text.rfind("\n\n",i,end) if nl > i+size*0.5: end=nl out.append(text[i:end]) i=end-overlap return out

01 · WHY SVG이미지 아닌 코드

RAG 품질 점검 4지표
RAG 운영의 최소 계기판은 정답 세트 20건과 네 가지 지표입니다. 측정되지 않는 시스템은 개선되지 않는다는 원칙을 제도화합니다. 첫째, 정확성(Accuracy)은 정답 20건 중 기대 답을 포함하고 기대 파일을 인용한 비율로 80% 이상이 운영 기준입니다. 둘째, 검색성(Recall@K)은 상위 K개 청크(주로 K=5)에 기대 청크가 포함된 확률이며 90% 이상이 목표입니다. 셋째, 재현성(Consistency)은 같은 질문을 3번 돌려 답이 일치하는 비율로 70% 이상을 유지하기 위해 temperature를 낮추고 프롬프트를 구조화합니다. 넷째, 안전성(Safety)은 근거 없는 단정·개인정보 패턴·금칙어 발생률로 1% 이하를 유지해야 합니다.
01 · ACCURACY · RECALL정확성·검색성
① 정확성 (Accuracy)
정답 20건 중 맞춘 비율 — 목표 80%+

답변이 기대 답을 포함하는가 + 출처가 기대 파일을 가리키는가. 단순 문자열 포함으로 시작해 LLM 평가로 확장.

② 검색성 (Recall@K)
상위 K 청크에 기대 청크가 포함될 확률

Recall@5 · Recall@10 측정. 검색성이 낮으면 답변성은 그 이하가 된다. 병목을 찾는 1순위.

# eval.py — 자동 평가 (발췌) def eval_row(q, expected_text, expected_src): hits = retrieve(q, k=5) a = generate(q, hits) acc = expected_text in a.text rec = any(expected_src in h.src for h in hits) return {"acc":acc, "recall5":rec}
02 · CONSISTENCY · SAFETY재현성·안전성
③ 재현성 (Consistency)
같은 질문을 3번 돌려서 답 일치율

LLM 무작위성 때문에 답이 출렁일 수 있다. 일치율 70%+ 목표. temperature 낮추고 프롬프트 구조화.

④ 안전성 (Safety)
근거 없는 단정·유출·금칙어 발생률

자동 검사: 출처가 비어 있거나 답변에 개인정보 패턴이 보이면 차단. 0%에 가까워야 운영 허용.

RAG 답변은 항상
출처 메타를 함께 반환한다.

RAG 답변 객체는 네 필드로 구조화한다 — answer · citations · confidence · abstain. citations에는 파일명·조항·페이지를 담고, 답변 본문 끝에 "근거: [파일명/조항]"을 강제 표기한다. 근거가 없으면 abstain=true로 두고 "규정에 없음" 또는 "담당자 확인 권장"으로 출구를 만든다. 감사 로그에는 이 네 필드와 hits·latency_ms·feedback을 함께 기록한다.
