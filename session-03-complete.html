메타 프롬프트는 "프롬프트를 만드는 프롬프트"다
직접 지시문을 짜는 대신, AI에게 업무 상황을 기술하면 AI가 역할·규칙·포맷·예시를 구조화해 돌려준다. 프롬프트 작성의 핵심 역량이 "기법 암기"에서 "상황 기술"로 이동한다.
CONCEPT메타 프롬프트의 구조
메타 프롬프트는 2단계로 작동한다. 1단계에서 AI에게 "나는 ○○ 업무를 하는데, 이 업무에 쓸 시스템 프롬프트를 만들어줘"라고 상황을 기술한다. 2단계에서 AI가 만든 프롬프트를 실제 업무에 돌려보고 수정한다.
# 메타 프롬프트 · 입력 예시 상황: ○○구 교통정책과 주무관. 버스 노선 관련 민원을 처리한다. 답변은 공문 형식이다. 요청: 이 업무에 쓸 시스템 프롬프트를 만들어줘. 역할·규칙·포맷·예시를 모두 포함해줘.
WHY IT WORKS
상황만 바꾸면 프롬프트가 자동으로 바뀐다

같은 메타 프롬프트 양식에 부서·업무·제약만 교체하면 새 프롬프트가 나온다. 기법을 외울 필요가 없어진다.

OUTPUT STRUCTUREAI가 만들어주는 프롬프트의 구조
섹션내용
PERSONA누구로서 답할지 — "○○구 민원 답변 담당자"
CONTEXT배경 · 조직 · 제약 — "공문 형식 · 법령 인용 필수"
RULES지킬 것 · 하지 말 것 — "가정적 표현 금지"
FORMAT출력 양식 — "접수확인 → 사안요약 → 근거 → 답변 → 안내"
EXAMPLES좋은 예 · 나쁜 예 — 모델이 따라할 기준
CRITICAL DIFFERENCE
기법 암기 vs 메타 생성
기법 암기: "역할을 먼저 정하고, 규칙을 나열하고, 포맷을 지정하고…" — 매번 기억해야 한다.
메타 생성: "상황을 기술하면 AI가 위 5섹션을 알아서 채운다" — 기억할 필요가 없다. 수정만 한다.

PART A-2 · 교차 검증 — 같은 메타 프롬프트를 3개 모델에 던진다 · 10MIN
교차 검증은 누락·편향·불필요 수사를 잡는 구조다
모델마다 "잘하는 프롬프트 스타일"이 다르다. ChatGPT는 구조 설계에 강하고 Claude는 논리·제약 반영에 강하고 Gemini는 간결·실용적이다. 한 모델에만 맡기면 그 모델의 편향이 프롬프트에 박힌다.
WHY교차 검증을 하는 3가지 이유
REASON 01
누락 방지 — 한 모델이 놓친 제약을 다른 모델이 잡아낸다

"500자 이내"를 ChatGPT는 반영하지만 출력 형식을 빠뜨릴 수 있다. Claude는 반대 패턴. 교차하면 양쪽 누락이 드러난다.

REASON 02
편향 제거 — 모델별 관용 표현이 달라 편향을 의식하게 된다

ChatGPT는 목록을 선호하고 Claude는 문단을 선호하는 식. 셋을 비교하면 "왜 이 형식이지?"를 자연스럽게 물어보게 된다.

REASON 03
감사 대응 — "왜 이 프롬프트를 썼나" 근거가 남는다

3개 결과를 표로 비교한 기록 자체가 의사결정 근거다. 감사·개선·인수인계 시 재현 가능.

PITFALL
3개 돌린 뒤 "그냥 하나 골라서 쓴다"
이러면 교차 검증이 아니라 3배 비용. 비교표를 채우고 조합해야 값어치가 있다.
HOW교차 검증 매트릭스 · 5관점
관찰 항목CHATGPTCLAUDEGEMINI

제약 반영도______
불필요 수사______
누락 지시______
COMBINE RULE
A의 역할 + B의 제약 + C의 출력 형식 = 최종본. 3개 결과의 가장 좋은 부분만 뽑아 조합한다. 모든 프롬프트에 교차 검증을 할 필요는 없다 — 재사용할 핵심 프롬프트에만. 1회성 프롬프트는 메타 생성만으로 충분하다.

TYPE 03
문서 변환
같은 원본 → 독자별 다른 형태
TYPE 04
논문 리서치
트렌드 → 논문 → 학습·정리
TYPE 05
이미지 콘텐츠
포스터 · 카드뉴스 · 배너

영상 파이프라인
스크립트 → TTS → 영상 → 편집

TYPE 01 — 문서 취합형 워크플로우의 구조
여러 부서·출처에서 수집된 문서를 분류 기준에 따라 정리하고 하나의 통합 문서로 생산하는 타입. 외부 체인과 내부망 체인을 병렬로 설계하는 것이 기본 원칙이다.

항목정의
